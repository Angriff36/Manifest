<Step title="Write your first .manifest file">

Create `my-program.manifest` in your project root. This entity models a simple to-do item with two commands and an authorization policy.

```manifest
entity Todo {
  property id: string
  property title: string = ""
  property completed: boolean = false
  property createdAt: number = 0

  command create(newTitle: string) {
    guard newTitle != null and newTitle != ""
    guard length(newTitle) < 100

    mutate title = newTitle
    mutate completed = false
    mutate createdAt = now()
  }

  command complete() {
    guard self.completed == false

    mutate completed = true
    emit TodoCompleted { todoId: self.id }
  }

  policy AdminRead read: user.role == "admin"
  policy Authenticated execute: user.id != null "Authentication required"
}
```

Each `command` defines a named business operation. The `guard` lines gate execution â€” if any guard evaluates to `false`, the command stops immediately and reports which guard failed. `mutate` assigns a bare property name; expressions read the current instance through `self.*`. The `policy` lines are entity-level authorization rules â€” each has a name, a scope (`read`, `write`, `execute`, `all`, or `override`), an expression, and an optional denial message â€” and they are enforced before guards even run.

</Step>

<Step title="Compile the source to IR">

Use the `compileToIR` export to parse the source and produce an IR object. Always check diagnostics before proceeding â€” compilation errors are surfaced here, not thrown as exceptions.

```typescript
import { compileToIR } from '@angriff36/manifest/ir-compiler';
import { readFileSync } from 'fs';

const source = readFileSync('my-program.manifest', 'utf-8');

const { ir, diagnostics } = await compileToIR(source);

if (diagnostics.some((d) => d.severity === 'error')) {
  console.error('Compilation errors:');
  diagnostics.forEach((d) => console.error(`  ${d.message}`));
  process.exit(1);
}

console.log('Compiled IR:', JSON.stringify(ir, null, 2));
```

You can also compile inline source directly â€” useful for tests and prototyping:

```typescript
import { compileToIR } from '@angriff36/manifest/ir-compiler';

const source = `
  entity Todo {
    property id: string
    property title: string
  }
`;

const { ir, diagnostics } = await compileToIR(source);
```

<Note>
  The IR is plain JSON. You can serialize it to disk, cache it, or pass it to any downstream
  consumer â€” including the `RuntimeEngine`, projections, and conformance tooling.
</Note>

</Step>

<Step title="Execute a command with RuntimeEngine">

Import `RuntimeEngine` from the package root and construct it with your compiled IR and a runtime context. `user.*` bindings in guards and policies resolve from `context.user` (an object with at least an `id`; `role` and other keys are up to you), and `context.tenantId` scopes tenant-aware commands.

```typescript
import { RuntimeEngine } from '@angriff36/manifest';

const runtime = new RuntimeEngine(ir, {
  user: { id: 'user-123', role: 'admin' },
  tenantId: 'tenant-456',
});

// Run the "create" command on the Todo entity.
// Signature: runCommand(commandName, input, { entityName })
const createResult = await runtime.runCommand(
  'create',
  { newTitle: 'Learn Manifest' },
  { entityName: 'Todo' },
);

if (!createResult.success) {
  console.error('Failed:', createResult.guardFailure ?? createResult.error);
  process.exit(1);
}

console.log('Created:', createResult.instance);
```

When the command succeeds, `createResult.instance` contains the resulting entity state. When it fails â€” because a guard evaluated to `false` or a policy denied access â€” inspect `createResult.guardFailure` (and `policyDenial`, `constraintOutcomes`, etc.) to see exactly which check failed and what values were resolved. `CommandResult` has no `diagnostics` field.

</Step>

<Step title="Chain a second command and inspect events">

Use the instance returned by `create` to run `complete`. Check `completeResult.emittedEvents` to see the `TodoCompleted` event that the command emitted.

```typescript
const completeResult = await runtime.runCommand(
  'complete',
  {},
  { entityName: 'Todo', instanceId: createResult.instance.id },
);

if (completeResult.success) {
  console.log('Events:', completeResult.emittedEvents);
  // Output: Events: [{ name: 'TodoCompleted', channel: 'TodoCompleted',
  //   payload: { todoId: '...', ... }, subject: { entity: 'Todo', ... }, ... }]
}
```

You can also subscribe to events as they are emitted. `runtime.onEvent(listener)` receives every event (filter by `event.name` yourself); `runtime.subscribe(entityName, listener)` receives only events whose subject is that entity. Both return an unsubscribe function.

```typescript
const unsubscribe = runtime.onEvent((event) => {
  if (event.name !== 'TodoCompleted') return;
  console.log('Todo completed:', event.payload.todoId);

  // Trigger side effects
  sendEmail(event.payload.todoId);
  updateAnalytics(event.payload.todoId);
});
```

</Step>

</Steps>

## Running the CLI instead

If you prefer a command-line workflow, use the `manifest` CLI that ships with the package:

```bash
# Interactive project setup (creates manifest.config.yaml)
manifest init

# Config-driven whole project (preferred after init)
manifest compile --all
manifest generate --all
# or: manifest build --all / manifest watch --all

# Single-file compile / one-shot build still work
manifest compile my-program.manifest --output ir/
manifest build my-program.manifest --projection nextjs --code-output app/api/
```

<Tip>
  Run `manifest init` once to create a `manifest.config.yaml`. After that, prefer
  `manifest build --all` (or `compile --all` + `generate --all`) so every configured projection
  stays in sync â€” verified CLI flags @ v3.6.41.
</Tip>

## Troubleshooting

~~**"Entity not found in IR"** â€” Entity names are case-sensitive. Make sure the name you pass to `runCommand` exactly matches the `entity` declaration in your source file.~~

> **Correction (2026-07-15):** `runCommand`'s first argument is the **command** name. Pass the
> entity as `options.entityName` (case-sensitive match to the `entity` declaration).

**"Guard failed but no detail"** â€” Check that you aren't silently catching errors before inspecting `result.guardFailure`. The runtime populates it for every guard failure, including the guard index, the formatted expression, and resolved sub-expression values.

**Compilation errors after a small change** â€” Conformance tests treat compilation output as executable semantics. If something stops compiling, you've changed language meaning. Review your source against the language specification.

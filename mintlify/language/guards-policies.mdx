Manifest separates authorization into two distinct mechanisms: **policies**, which control who is allowed to execute a command at all, and **guards**, which verify that the current state satisfies the preconditions for the command to proceed. Both are evaluated before any action runs, and neither can be bypassed.

## Guards

A guard is a boolean expression declared inside a command body. Guards are evaluated in declaration order. Execution halts on the first expression that evaluates to `false` — no later guards run, and no actions execute.

```manifest
entity Task {
  property completed: boolean = false
  property priority: number = 1

  command complete() {
    guard not self.completed
    mutate completed = true
    emit TaskCompleted
  }

  command setPriority(level: number) {
    guard level >= 1 and level <= 5
    mutate priority = level
  }
}
```

### Guard diagnostics

When a guard fails, the runtime returns a `guardFailure` in the `CommandResult` with:

- `index` — 1-based position of the failing guard
- `expression` — the source expression as a string
- `resolved` — array of `{ expression, value }` pairs showing what each sub-expression evaluated to

This gives you enough information to explain the failure to the caller without any manual debugging.

```typescript
if (result.guardFailure) {
  console.error(`Guard ${result.guardFailure.index} failed: ${result.guardFailure.expression}`);
  for (const r of result.guardFailure.resolved) {
    console.error(`  ${r.expression} => ${r.value}`);
  }
}
```

### Available bindings in guards

Guard expressions evaluate in the command's evaluation context. You have access to:

| Binding             | Value                                                             |
| ------------------- | ----------------------------------------------------------------- |
| `self` / `this`     | The current entity instance (or `null` when no instance is bound) |
| `user`              | The current user object, or `null` when unauthenticated           |
| `context`           | The runtime context object                                        |
| `context.tenantId`  | Active tenant ID (if provided by caller)                          |
| `context.orgId`     | Active organization ID (if provided)                              |
| `context.actorId`   | Acting user ID (if provided)                                      |
| `context.requestId` | Caller-supplied request ID                                        |
| Command params      | Named parameters from the command signature                       |
| `now()`             | Current time as milliseconds since epoch                          |
| `uuid()`            | A new globally unique identifier                                  |

<Warning>
  If a guard references `user` and no user is provided to the runtime, the guard correctly fails.
  This is not a bug — it is the expected behavior. The runtime does not inject a permissive default
  `user` to make demos work.
</Warning>

### Operators in guard expressions

Guards support the full expression language:

| Category    | Operators                        |
| ----------- | -------------------------------- |
| Arithmetic  | `+`, `-`, `*`, `/`, `%`          |
| Comparison  | `==`, `!=`, `<`, `>`, `<=`, `>=` |
| Logical     | `and`, `or`, `not`, `!`          |
| Membership  | `in`, `contains`                 |
| Conditional | `? :` (ternary)                  |

`==` uses loose equality (`undefined == null` is `true`). Use `is not` or `!=` for strict inequality.

## Policies

A policy is a named boolean expression with a **scope** that controls when it is checked. Policies are declared at entity level (inside an entity block) or at module level (outside any entity block).

```manifest
policy adminOnly execute: user.role == "admin" "Only administrators can execute commands"
```

The expression after the scope keyword is evaluated before any command runs. If it returns `false`, the runtime returns a `policyDenial` result and no further execution occurs.

### Policy scopes

| Scope      | When enforced                                                |
| ---------- | ------------------------------------------------------------ |
| `execute`  | Before any command execution                                 |
| `read`     | At the runtime read gate — `getInstance` / `getAllInstances` |
| `write`    | On entity write operations (not enforced by default runtime) |
| `all`      | Both command execution and the read gate                     |
| `override` | Before authorizing a constraint override (vNext)             |

The runtime enforces `execute` and `all` scopes before command execution. `read` (and `all`) scopes are **also** enforced at a single central read gate in `getInstance` / `getAllInstances`: a denied read fails closed — `getInstance` returns `undefined` (no existence leak) and `getAllInstances` omits the denied rows. A read policy referencing `self.*` is evaluated per row; a context-only policy is evaluated once per call. The internal execution read path (guards, actions, computed properties) is un-gated, so read policies never change command results. `write` and `delete` scopes are still enforced only at command execution (via `command.policies`), not on the store — they are otherwise available for custom adapter or projection logic.

### Binding a policy to an entity

A policy applies to all commands on a specific entity when you reference it inside the entity block or use the entity name in a standalone policy:

```manifest
entity Document {
  property required title: string
  property isPublic: boolean = false

  command makePublic() {
    mutate isPublic = true
    emit DocumentPublished
  }
}

policy adminOnly execute: user.role == "admin" "Only admins can execute"
```

In this example, `adminOnly` is a module-level policy. Use `default policy` inside an entity block to apply a policy to every command on that entity.

### Per-command access restrictions

Policies are entity-level or module-level — there is no inline policy syntax inside a command body. To enforce stricter access on a single command, use a guard inside that command:

```manifest
entity InventoryItem {
  property stock: number = 0

  default policy KitchenRoles execute: user.role in ["kitchen_staff", "kitchen_lead", "manager", "admin"]

  command consume(quantity: number) {
    // Inherits KitchenRoles policy
    guard quantity > 0
    mutate stock = self.stock - quantity
  }

  command adjust(delta: number) {
    // Per-command stricter access: add a guard rather than an inline policy
    guard user.role in ["kitchen_lead", "manager", "admin"]
    mutate stock = self.stock + delta
  }
}
```

<Note>
  Guards produce a `guardFailure` result; policies produce a `policyDenial` result. Use guards for
  state-based preconditions and additional per-command role checks; use policies for entity-wide
  authorization.
</Note>

## Default policies (vNext)

Use `default policy` on an entity to apply a policy to every command that does not declare its own:

```manifest
entity Task {
  property required title: string
  property status: string = "pending"
  property assigneeId: string

  default policy RequireAuth execute: user.id != null "Authentication required"

  command complete() {
    guard self.assigneeId == user.id or user.role == "admin"
    mutate status = "completed"
    emit TaskCompleted
  }

  command reassign(newAssigneeId: string) {
    guard user.role == "admin" or user.role == "manager"
    mutate assigneeId = newAssigneeId
  }
}
```

Both `complete` and `reassign` inherit `RequireAuth`. Add a guard inside any command for additional per-command access checks.

### Inheritance rules

1. If the entity declares a `default policy`, every command on that entity inherits it.
2. If no default policy applies, no policy check runs (the scanner will warn).

The IR compiler expands default policies into each command's effective policy list at compile time. The runtime always evaluates an explicit, expanded list — it does not perform inheritance at runtime.

## Override policies (vNext)

The `override` scope is a special policy used to authorize constraint bypasses. When a constraint is marked `overrideable`, the runtime evaluates the referenced override policy before granting the bypass:

```manifest
entity FinancialRecord {
  command process() {
    constraint overrideable requiresApproval {
      expression: self.status == "approved" or self.amount <= 500
      severity: block
      message: "Must be approved before processing"
      overridePolicy: canApproveLargeAmounts
    }

    mutate status = "processed"
    emit RecordProcessed
  }
}

policy canApproveLargeAmounts override:
  user.role == "manager" or user.role == "admin"
  "Only managers and admins can approve large amounts"
```

If the override policy passes, the constraint outcome is marked `overridden: true`. Both override paths are audited: the auto-policy path emits the same synthetic `OverrideApplied` event the explicit-request path emits, with `authorizedBy` derived from the acting user in context (falling back to `policy:<name>` when no user is present) and the event `reason` recording the authorizing policy. Attempts to override a constraint that is not marked `overrideable` are always rejected.

## Default-deny enforcement

When the scanner detects a command with no policy coverage (neither explicit nor inherited from entity defaults), it emits a warning. In strict environments, this warning can be promoted to an error:

```bash
manifest scan --strict
```

The `--strict` flag causes the scanner to exit non-zero when any command lacks policy coverage. This enforces a default-deny posture: every command must explicitly declare who is authorized to run it, or inherit a default policy from its entity.

<Warning>
  A command with no policies is **not blocked** by the runtime — it executes normally. The
  default-deny enforcement is a static analysis check (via `manifest scan`), not a runtime gate. To
  enforce it in CI, add `manifest scan --strict` to your pipeline.
</Warning>

## Roles and permission inheritance

For fine-grained authorization based on role hierarchies (e.g., Admin → Manager → User with inherited permissions), see [Roles and permissions](/language/roles). Roles integrate with policies through the `user.role` binding and the `hasPermission()` / `roleAllows()` builtins.

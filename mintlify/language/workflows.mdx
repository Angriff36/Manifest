Manifest provides two workflow primitives for coordinating execution across time and services: **saga orchestration** for distributed transactions with compensation, and **scheduled commands** for time-based triggers.

## Saga orchestration

A saga declares a sequence of steps that execute across entities or services. Each step names a `command` and an optional `compensate` command for rollback. If any step fails, the saga runs compensate for all previously completed steps in reverse order.

### Declaring a saga

Use the `saga` keyword at the program or module level. Each `step` is a block naming its `command` and optional `compensate` as `Entity.command` references. The `on_failure` clause takes a single identifier — `compensate` (the default) or `abort` — and `emit` declares the saga's lifecycle events:

```manifest
event SagaStarted: "saga.started" { sagaName: string }
event SagaCompleted: "saga.completed" { sagaName: string }
event SagaFailed: "saga.failed" { sagaName: string }
event SagaStepCompleted: "saga.step.completed" { sagaName: string }

saga ProcessOrder {
  step reserveInventory {
    command: Inventory.reserve
    compensate: Inventory.release
  }

  step chargePayment {
    command: Payment.charge
    compensate: Payment.refund
  }

  step shipOrder {
    command: Shipping.dispatch
  }

  on_failure: "compensate"
  ~~emit OrderProcessingFailed~~
  emit SagaStarted
  emit SagaCompleted
  emit SagaFailed
  emit SagaStepCompleted
}
```

> **Correction (2026-07-15) @RYANSIGNED:** List the fixed lifecycle event names you want emitted (and declare matching `event`s). A single custom `emit OrderProcessingFailed` does **not** substitute for `SagaFailed` / `SagaCompleted`. Start with `runtime.runSaga('ProcessOrder', stepInputs)`.

### Saga lifecycle

<Steps>
  <Step title="Start">
    ~~The saga begins when the first step's command is invoked.~~ **Correction (2026-07-15)
    @RYANSIGNED:** The saga begins when the host calls `runSaga(name, stepInputs)`. Steps then run
    in declaration order via `runCommand`.
  </Step>
  <Step title="Execute steps">
    Each step's command runs in sequence. If the command succeeds, the saga advances to the next
    step.
  </Step>
  <Step title="On failure — compensate">
    If any step's command fails, the saga enters compensation mode. It runs each completed step's
    compensate command in reverse order (last completed first).
  </Step>
  <Step title="Complete or failed">
    After all steps complete (or all compensations run), the saga emits a completion or failure
    event.
  </Step>
</Steps>

### Best-effort compensation

Compensation is best-effort. If a compensate command itself fails, the saga logs the failure and continues compensating remaining steps. This ensures partial rollback rather than aborting the compensation process entirely.

### Saga lifecycle events

~~| Event | When emitted |
| --- | --- |
| `{SagaName}Started` | … |
| `{SagaName}StepCompensated` | … |~~

> **Correction (2026-07-15) @RYANSIGNED:** Lifecycle event names are **fixed** (`SagaStarted`, `SagaStepCompleted`, `SagaCompleted`, `SagaFailed`) — not `{SagaName}…`. The runtime emits a lifecycle event **only if** that name appears in the saga's `emit` list **and** a matching `event` is declared in the IR (`emitSagaLifecycle`). There is **no** `SagaStepCompensated` lifecycle emission. Invoke with `runtime.runSaga(name, stepInputs)`.

```manifest
event SagaStarted: "saga.started" { sagaName: string }
event SagaCompleted: "saga.completed" { sagaName: string }
event SagaFailed: "saga.failed" { sagaName: string }
event SagaStepCompleted: "saga.step.completed" { sagaName: string }

entity Order {
  property required id: string
  command charge() {
    mutate charged = true
  }
  command refund() {
    mutate charged = false
  }
  store in memory
}

saga ProcessOrder {
  step chargePayment {
    command: Order.charge
    compensate: Order.refund
  }
  on_failure: "compensate"
  emit SagaStarted
  emit SagaCompleted
  emit SagaFailed
  emit SagaStepCompleted
}
```

<Note>
  Declared saga lifecycle events carry `correlationId` / `causationId` like other emissions when
  present on the emitted record.
</Note>

## Scheduled commands

Scheduled commands trigger entity commands on a time-based schedule using cron expressions, fixed intervals, or periodic timers.

### Declaring a scheduled command

Use the `schedule` keyword inside an entity block:

```manifest
entity Report {
  property required id: string
  property lastRunAt: number = 0
  property status: string = "idle"

  schedule generateDaily
    cron "0 6 * * *"           // Every day at 6:00 AM
    run generate

  schedule healthCheck
    interval 300000            // Every 5 minutes (in ms)
    run checkHealth

  schedule cleanup
    every 24 hours             // Every 24 hours
    run purgeExpired

  command generate() {
    mutate status = "running"
    emit ReportGenerationStarted
  }

  command checkHealth() {
    mutate lastRunAt = now()
  }

  command purgeExpired() {
    mutate status = "cleaned"
  }
}
```

### Trigger types

| Type       | Syntax                            | Description                                                      |
| ---------- | --------------------------------- | ---------------------------------------------------------------- |
| `cron`     | `cron "0 6 * * *"`                | Standard 5-field cron expression (minute hour day month weekday) |
| `interval` | `interval <ms>`                   | Fixed interval in milliseconds                                   |
| `every`    | `every <n> hours/minutes/seconds` | Human-readable period                                            |

### Schedule resolution

Schedules are resolved at compile time and emitted to the IR as `IRSchedule` objects. The `RuntimeEngine` class has no built-in timer, but the package ships an optional worker (`startScheduleWorker` from `@angriff36/manifest/schedule-worker`) that drives schedules on a tick loop and sweeps approval expiry; projection targets also consume the metadata:

- **Next.js projection**: Generates a cron route per schedule (`/api/cron/[name]/route.ts`) plus a `vercel.json` registering them, so the platform invokes them on schedule.
- **Express/Hono projections**: Emit route handlers intended for external scheduler invocation.
- **Infrastructure projections** (Terraform): Generate CloudWatch Event rules or similar scheduled resources.

### Inline parameters

Scheduled commands can pass static parameters to the target command:

```manifest
entity Cache {
  schedule warmCache
    cron "0 */6 * * *"
    run refresh(keys: "all")

  command refresh(keys: string) {
    // ...
  }
}
```

<Warning>
  The `RuntimeEngine` class has no built-in timer. On Vercel, the generated cron routes and
  `vercel.json` fire schedules automatically; on other hosts, run `startScheduleWorker(runtime)`
  from `@angriff36/manifest/schedule-worker`, or invoke `runSchedule(name)` from your own scheduler.
  Cron triggers are matched in UTC to the minute.
</Warning>

## Combining sagas and schedules

~~Sagas and schedules can work together. A scheduled command can initiate a saga…~~

> **Correction (2026-07-15) @RYANSIGNED:** Schedules invoke an entity **command** (`run <command>`). They do **not** call `runSaga` automatically. To start a saga from a schedule, the scheduled command (or a reaction it triggers) must call into host code that invokes `runtime.runSaga(...)`, or model the workflow as ordinary commands/reactions. Emitting an event alone does not start a saga.

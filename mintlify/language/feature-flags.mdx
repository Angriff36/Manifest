The `flag(name)` built-in resolves a named feature flag through a runtime-provided provider, letting guards and computed properties gate behavior on flags declared in the program but evaluated against external flag state. When no provider is configured, `flag` returns `false`, so features are off by default.

## Syntax

`flag(name)` is used in guards and computed properties. The single argument is the flag name as a string literal, and `flag(...)` composes with other expressions:

```manifest
entity FeatureGatedEntity {
  property required id: string
  property status: string = "inactive"

  command activate() {
    guard flag("new-activation-flow")
    mutate status = "active"
  }

  command deactivate() {
    guard flag("allow-deactivation") and self.status == "active"
    mutate status = "inactive"
  }
}

entity FlagInspector {
  property required id: string
  property flagValue: string = ""

  computed isNewUIEnabled: boolean = flag("new-ui")
}

store FeatureGatedEntity in memory
store FlagInspector in memory
```

## Behavior

The `flag` built-in returns `false` if the argument is not a string. Otherwise, if a `flagProvider` runtime option is set, it delegates to that provider — `this.options.flagProvider(name)` — and returns whatever the provider returns. When no provider is configured, it returns `false`.

The provider is typed as `flagProvider?: (name: string) => unknown`. Its return value is passed through unchanged, so a flag can resolve to a boolean, string, number, or object depending on the provider — the engine does not coerce it. This is the documented integration point for wiring an external system such as LaunchDarkly:

```typescript
flagProvider: (name) => launchDarklyClient.variation(name, false)
```

Because resolution happens entirely through the runtime-supplied provider and the IR carries only the `flag("name")` call, the same compiled program yields different gating behavior in different runtime contexts without any IR change — consistent with Manifest's rule that variability enters through runtime context, not by editing the IR. In a guard, a falsey `flag(...)` result halts command execution at that guard in order, the same as any other guard expression.

Feature flags are a runtime expression concern, not a schema construct, so there is no dedicated projection mapping; the `flag("name")` call is carried in the IR like any other expression.

<Note>
The safe default is off: with no `flagProvider`, every `flag(...)` returns `false`, so guards depending on a flag deny by default. Provider return values are not validated or coerced by the engine, so a provider that returns a non-boolean passes that value straight into the surrounding expression.
</Note>

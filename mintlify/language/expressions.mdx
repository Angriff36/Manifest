Manifest's expression language ships a library of deterministic, side-effect-free built-in functions: string and numeric helpers, UTC date components, and aggregate operations over collections. The aggregates accept optional mapper or predicate lambdas, letting computed properties summarize related entities declaratively. All functions are side-effect-free and produce identical results given identical input and runtime context.

## Scalar and string built-ins

String and numeric helpers can be used anywhere expressions are evaluated ā€” `compute` bindings, guards, computed properties, and constraints:

```manifest
command testStringOps(input: string) {
  guard input != ""
  compute cleaned = trim(input)
  compute partCount = count(split(input, ","))
  compute hasPrefix = startsWith(input, "hello")
  compute hasSuffix = endsWith(input, "world")
  compute swapped = replace(input, "old", "new")
  compute upper = toUpperCase(input)
  compute lower = toLowerCase(input)
  compute charCount = length(input)
  compute slice = substring(input, 0, 5)
  compute foundAt = indexOf(input, "world")
}
```

Numeric helpers `abs`, `round`, `floor`, `ceil`, `min`, and `max` and the array helper `sum` appear in the same fixture, and the date components `year`/`month`/`day`/`hours`/`minutes`/`seconds` operate on a millisecond timestamp.

String helpers wrap the corresponding `String.prototype` methods (returning the input unchanged for non-strings), numeric helpers delegate to `Math`, and `min`/`max` are variadic over numeric arguments. The date components use UTC methods so results are timezone-independent ā€” see [Types](/language/types) for the full list.

## Aggregate built-ins with lambdas

The aggregates reduce a collection ā€” typically a `hasMany` relationship reached via `self.` ā€” to a single value, optionally transforming or filtering each element with a lambda:

```manifest
entity Order {
  property required customerName: string
  property status: string = "pending"

  hasMany lineItems: LineItem

  computed totalAmount: number = sum(self.lineItems, (item) => item.price * item.quantity)
  computed itemCount: number = count_of(self.lineItems)
  computed averagePrice: number = avg(self.lineItems, (item) => item.price)
  computed cheapestItem: number = min_of(self.lineItems, (item) => item.price)
  computed mostExpensiveItem: number = max_of(self.lineItems, (item) => item.price)
  computed highValueCount: number = count_of(self.lineItems, (item) => item.price * item.quantity > 50)
}

entity LineItem {
  property required price: number
  property quantity: number = 1

  belongsTo order: Order

  computed lineTotal: number = price * quantity
}

store Order in memory
store LineItem in memory
```

For `sum`, `avg`, `min_of`, and `max_of` the lambda maps each element to a number; for `count_of` the lambda is a predicate selecting which elements to count.

The behavior with a lambda is:

- `sum` adds each mapped number.
- `avg` averages the mapped numbers (`0` for an empty collection).
- `min_of` / `max_of` return the smallest / largest mapped number (`undefined` for an empty collection).
- `count_of` returns the total length without a predicate, or the count of elements for which the predicate is truthy.

Without a lambda, `sum` and `avg` operate on the raw numeric elements and `count_of` returns the collection length. Each aggregate guards against a non-array first argument: `sum` returns its argument unchanged, `avg` returns `0`, `count_of` returns `0`, `min_of`/`max_of` return `undefined`, and `filter`/`map` return an empty array (or the array unchanged when no callback is supplied).

Because aggregating over a `hasMany` relationship requires resolving related instances asynchronously, the lambda-bearing branches return promises that the evaluator awaits. The element mapper is invoked once per element and its result awaited, keeping evaluation deterministic for a fixed runtime context.

<Note>
Two distinct count helpers exist: `count(array)` (a plain length used above in `count(split(input, ","))`) and `count_of(collection, predicate?)` (the aggregate). Likewise `min`/`max` are variadic over scalar arguments, which is different from `min_of`/`max_of`, which reduce a collection. Using an aggregate on a non-array yields the safe fallbacks listed above rather than an error.
</Note>

## Standard library

The Manifest standard library (`stdlib`) extends the built-in expression functions with additional utilities for common operations. These functions are available automatically ā€” no import is required.

### Full-text search

The `search(text, query)` function performs full-text matching against a `searchable` property:

```manifest
entity Article {
  property required title: string
  property searchable content: string

  constraint relevant: search(self.content, "manifest dsl")
}
```

The `searchable` modifier on string properties signals to projections that a full-text index should be created (e.g., Prisma `@@fulltext`, Drizzle GIN index). The `search()` function is the runtime counterpart that evaluates the match.

### Feature flags in expressions

The `flag(name)` function resolves a feature flag value at expression evaluation time:

```manifest
constraint betaEnabled: flag("new-pricing") == true

guard flag("max-order-limit") == false or self.total < flag("max-order-limit")
```

`flag()` supports boolean, string, number, and object flag values. See [Feature flags](/language/feature-flags) for the full feature flag system.

### Extended built-ins

Additional built-in functions available in the standard library:

| Function | Signature | Description |
|----------|-----------|-------------|
| `between` | `between(value, low, high)` | Returns `true` if `low <= value <= high` |
| `matches` | `matches(text, pattern)` | Tests `text` against a regex `pattern` |
| `search` | `search(text, query)` | Full-text search match |
| `flag` | `flag(name)` | Resolve a feature flag value |

## Custom expression functions

You can register custom expression functions via the plugin API. Custom functions are available in guards, computed properties, constraints, and other expression contexts.

### Registering a custom function

```typescript
import { RuntimeEngine, registerBuiltin } from "@angriff36/manifest";

// Register a custom function before creating the runtime
registerBuiltin("discount", (price: number, tier: string) => {
  if (tier === "premium") return price * 0.8;
  if (tier === "standard") return price * 0.9;
  return price;
});
```

Once registered, the function is available in Manifest expressions:

```manifest
entity Order {
  property total: number = 100
  property tier: string = "standard"

  computed discountedTotal: number = discount(self.total, self.tier)
}
```

### Constraints on custom functions

<Warning>
Custom functions are not deterministic by default ā€” they can reference external state, make network calls, or produce different results for the same inputs. If you use custom functions in guards or constraints, you are responsible for ensuring they produce consistent results within a single `runCommand` invocation. Violating this breaks the Manifest determinism guarantee.
</Warning>

For this reason, custom functions used in expressions should be pure functions with no side effects. If external data is needed, inject it through the runtime context (`user`, `context.*`) rather than calling out from inside a custom function.

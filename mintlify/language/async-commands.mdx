Async commands defer action execution to a background worker queue. The `async` keyword prefix tells the runtime to validate the command synchronously (policies, constraints, guards) and then enqueue it for later execution rather than running actions immediately. The caller receives a job ID and a `pending` status without waiting for the work to complete.

## Declaring an async command

Add the `async` keyword before `command`:

```manifest
entity Order {
  property required id: string
  property status: string = "pending"
  property processedAt: number = 0

  async command processOrder(amount: number) {
    guard amount > 0
    mutate status = "processing"
    mutate processedAt = now()
    emit OrderProcessing
  }
}
```

The `async` modifier is available at program-level, module-level, and entity-level command declarations.

## How async commands execute

<Steps>
  <Step title="Synchronous validation">
    The runtime evaluates policies, constraints, and guards synchronously — exactly as it would for a non-async command. If any check fails, the command fails immediately with the normal `policyDenial`, `constraintOutcomes`, or `guardFailure` result. No job is enqueued.
  </Step>
  <Step title="Enqueue job">
    If validation passes, the runtime creates a `JobRecord` and enqueues it via the configured `JobQueue` adapter.
  </Step>
  <Step title="Return immediately">
    The caller receives a result with `{ jobId, status: 'pending', enqueuedAt }`. Actions have not yet executed.
  </Step>
  <Step title="Worker executes">
    When the job is drained (via `drainJobs()` or a production queue consumer), the runtime re-enters the command with `context.source === 'job'`, bypassing the async branch and executing all actions normally.
  </Step>
</Steps>

<Note>
  Fail-fast validation is the key design decision: policies, constraints, and guards are checked
  synchronously so that invalid commands are rejected immediately. Only valid commands are enqueued.
  This means callers get instant feedback on authorization and precondition failures without waiting
  for a background worker.
</Note>

## Return value

```typescript
// Incorrect flat shape (do not use):
// { success: true, jobId, status: "pending", enqueuedAt }
```

Async enqueue still returns a normal `CommandResult`. The job metadata is nested under `result`:

```typescript
{
  success: true,
  result: { jobId: "job-abc123", status: "pending", enqueuedAt: 1748131200000 },
  emittedEvents: []
}
```

Read `result.result.jobId` / `result.result.status`. The `jobId` correlates with the synthesized completion/failure events.

## Auto-synthesized events

The compiler automatically creates two events for each async command:

| Event                    | When emitted                        |
| ------------------------ | ----------------------------------- |
| `{CommandName}Completed` | After the job executes successfully |
| `{CommandName}Failed`    | If the job execution fails          |

Synthesized events use channel `jobs.<commandName>` (not a domain-style channel), and payloads include timestamps:

```manifest
// Compiler-synthesized (illustrative):
// event ProcessOrderCompleted: "jobs.processOrder" {
// jobId: string; result: any; completedAt: number
// }
// event ProcessOrderFailed: "jobs.processOrder" {
// jobId: string; error: string; failedAt: number
// }
```

Names are `{commandName}Completed` / `{commandName}Failed`, appended to the IR `events` array.

<Warning>
  If you declare an event with the same name as a synthesized event, the compiler produces an error
  diagnostic. Rename your event to avoid the collision.
</Warning>

## JobQueue adapter

Configure a job queue in `RuntimeOptions` to enable async command execution:

```typescript
import { RuntimeEngine, MemoryJobQueue } from '@angriff36/manifest';

const jobQueue = new MemoryJobQueue();

Constructor is `(ir, context, options)`. Put `user` in context and `jobQueue` in options:

```typescript
const runtime = new RuntimeEngine(
  ir,
  { user: { id: 'system', role: 'worker' } },
  { jobQueue },
);
```

The `JobQueue` interface defines three methods:

`enqueue` returns `Promise<void>` (the runtime allocates `jobId` before enqueue). Optional second arg is an opaque transaction handle:

```typescript
interface JobQueue {
  enqueue(job: JobRecord, tx?: unknown): Promise<void>;
  drainPending(): Promise<JobRecord[]>;
  updateStatus(jobId: string, status: string, error?: string): Promise<void>;
}
```

A `JobRecord` carries:

```typescript
interface JobRecord {
  jobId: string;
  commandName: string;
  entityName?: string;
  instanceId?: string;
  input: Record<string, unknown>;
  correlationId?: string;
  causationId?: string;
  enqueuedAt: number;
  status: 'pending' | 'running' | 'completed' | 'failed';
}
```

<Warning>
  `MemoryJobQueue` is for testing and development only. It stores jobs in memory and they are lost
  on process restart. Production deployments should use a durable queue — the shipped
  `PostgresJobQueue` (below), or your own `JobQueue` backed by Redis, SQS, Google Cloud Tasks, or
  similar. If no `jobQueue` is configured and an async command is invoked, the runtime returns a
  `MISSING_JOB_QUEUE` error result.
</Warning>

## Production: durable queue + drain worker

`PostgresJobQueue` is a durable `JobQueue` adapter, and `runJobWorker` is the
shipped drain loop — you no longer hand-write either. Apply the companion
schema (`src/manifest/jobs/stores/postgres.sql`) once, then:

```typescript
import { RuntimeEngine } from '@angriff36/manifest';
import { PostgresJobQueue } from '@angriff36/manifest/jobs/postgres';
import { runJobWorker } from '@angriff36/manifest/jobs/worker';
import { Pool } from 'pg';

const pool = new Pool({ connectionString: process.env.DATABASE_URL });
const runtime = new RuntimeEngine(ir, context, {
  jobQueue: new PostgresJobQueue({ pool }),
});

// Long-lived worker: each tick drains pending jobs, runs their actions, and
// records completed/failed status. Stop it on shutdown.
const worker = runJobWorker(runtime);
await worker.stop();
```

`PostgresJobQueue.drainPending` claims pending rows with `FOR UPDATE SKIP
LOCKED` and flips them to `running` in one statement, so multiple worker
replicas drain disjoint jobs. For a cron/serverless deployment that wants a
single pass instead of a loop, call `drainJobsOnce(runtime)`.

## Deterministic testing with drainJobs()

The `drainJobs()` method processes pending jobs in FIFO order, making async commands fully testable without real message queues:

`runCommand(commandName, input, options?)` — not
entity-first. Command params belong in `input`; `entityName` / `instanceId` in `options`.

```typescript
import { RuntimeEngine, MemoryJobQueue } from '@angriff36/manifest';
import { ir } from './compiled.ir.json';

const jobQueue = new MemoryJobQueue();

const runtime = new RuntimeEngine(
  ir,
  { user: { id: 'system', role: 'worker' } },
  { jobQueue },
);

// Seed an order instance
runtime.createInstance('Order', { id: 'order-1', status: 'pending' });

// Invoke the async command
const result = await runtime.runCommand(
  'processOrder',
  { amount: 50 },
  { entityName: 'Order', instanceId: 'order-1' },
);

console.log(result.success); // true

console.log(result.result.status); // "pending"
console.log(result.result.jobId); // runtime-allocated id

// The instance has NOT changed yet
const before = runtime.getInstance('Order', 'order-1');
console.log(before.status); // "pending"

// Drain the job queue — executes the enqueued command
await runtime.drainJobs();

// Now the instance reflects the async command's mutations
const after = runtime.getInstance('Order', 'order-1');
console.log(after.status); // "processing"
console.log(after.processedAt); // <timestamp>
```

`drainJobs()` is the primary testing surface for async commands. It processes all pending jobs in order, emitting synthesized completion/failure events as appropriate.

## Re-entry guard

When a job is drained, the runtime re-enters the command with `context.source === 'job'`. The async branch detects this and executes actions normally instead of enqueuing again. This prevents infinite enqueue loops:

1. First call (`context.source` is not `'job'`) -- validate and enqueue.
2. Second call during drain (`context.source === 'job'`) -- skip the async branch, execute actions.

You can also use `context.source` in your own guard expressions if you need to enforce different behavior for job-initiated versus user-initiated calls.

## Complete example

```manifest
entity Order {
  property required id: string
  property amount: number = 0
  property status: string = "pending"
  property processedAt: number = 0

  command submit(amount: number) {
    guard amount > 0
    mutate amount = amount
    mutate status = "submitted"
    emit OrderSubmitted
  }

  async command processOrder() {
    guard self.status == "submitted"
    mutate status = "processing"
    mutate processedAt = now()
    emit OrderProcessing
  }
}
```

Same `runCommand(commandName, input, options?)` shape.

```typescript
import { RuntimeEngine, MemoryJobQueue } from '@angriff36/manifest';
import { ir } from './compiled.ir.json';

const jobQueue = new MemoryJobQueue();

const runtime = new RuntimeEngine(
  ir,
  { user: { id: 'system', role: 'worker' } },
  { jobQueue },
);

// Submit the order synchronously
runtime.createInstance('Order', { id: 'order-42', status: 'pending' });
await runtime.runCommand(
  'submit',
  { amount: 250 },
  { entityName: 'Order', instanceId: 'order-42' },
);

// Process the order asynchronously
const result = await runtime.runCommand(
  'processOrder',
  {},
  { entityName: 'Order', instanceId: 'order-42' },
);

console.log(result.result.status); // "pending"
console.log(result.result.jobId); // defined

// Execute the background job
await runtime.drainJobs();

const order = runtime.getInstance('Order', 'order-42');
console.log(order.status); // "processing"
```

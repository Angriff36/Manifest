A command is a named operation bound to an entity. When you call `runCommand`, the runtime executes the command in a strict, deterministic order:
~~rate-limit → policies → constraints → guards → actions → emits → return~~

> **Correction (2026-07-15):** parameters/context → rate-limit → policies → constraints → guards →
> approval gate (when declared) → actions → declared `emits` → return. Nothing in that order can be
> rearranged or skipped.

## Declaring a command

Commands live inside an entity block. A command takes an optional parameter list and a body of guards and actions.

```manifest
entity Counter {
  property value: number = 0

  command increment() {
    mutate value = value + 1
    emit CounterIncremented
  }

  command setValue(newValue: number) {
    guard newValue >= 0
    mutate value = newValue
    emit CounterSet
  }
}
```

Parameters are typed with the same types as properties: `string`, `number`, `boolean`, `timestamp`. Parameters are available by name in all guard and action expressions.

## Execution order

The runtime executes every command in exactly this order:

<Steps>
  <Step title="Process parameters">
    Apply each declared parameter's `defaultValue` to any omitted argument, then fail closed with a
    `parameterFailure` (`MISSING_REQUIRED_PARAMETER`) when a `required` parameter with no default is
    absent. An explicit `undefined` is treated as absent; `null` counts as supplied.
  </Step>
  <Step title="Build evaluation context">
    Bind `self` and `this` to the current entity instance, inject command parameters (with defaults
    already applied), and attach the runtime context (`user`, `context.*`).
  </Step>
  <Step title="Check rate limit">
    If the command declares a `rateLimit`, check whether the caller has exceeded the configured
    window. If the limit is exceeded, execution stops immediately with a `rateLimitDenial` result.
  </Step>
  <Step title="Evaluate policies">
    ~~Check every applicable policy (entity-level defaults first, then command-level). If any policy
    denies, execution stops immediately with a `policyDenial` result.~~

    **Correction (2026-07-15):** Evaluate the command's effective policy list only. Entity
    `default policy` values are expanded into `IRCommand.policies` at compile time; when a command
    already has an explicit IR `policies` array, defaults are **not** merged. If any policy denies,
    execution stops with a `policyDenial` result. (Source has no command-body policy clause —
    per-command stricter checks use guards.)
  </Step>
  <Step title="Evaluate command constraints">
    Evaluate any `block` constraints declared inside the command. If any fail without an authorized
    override, execution stops with `constraintOutcomes` in the result.
  </Step>
  <Step title="Evaluate guards">
    Evaluate each guard expression in declaration order. Execution halts on the first falsey result
    and returns a `guardFailure` with the failing guard's index and expression.
  </Step>
  <Step title="Check approval gate (when declared)">
    **Added (2026-07-15):** If the entity declares an approval for this command and applicable
    stages are not fully granted, execution stops with `approvalRequired` instead of running
    actions. See [Approvals](/language/approvals).
  </Step>
  <Step title="Execute actions">
    Run each action (`mutate`, `emit`, `compute`, `publish`, `persist`, `effect`) in declaration
    order.
  </Step>
  <Step title="Emit declared events">
    Emit the events listed in the `emits` declaration (if any) in order, attaching payload and
    metadata.
  </Step>
  <Step title="Return CommandResult">
    Return the `CommandResult` with `success: true`, the last action result, and all emitted events.
  </Step>
</Steps>

## Actions

Actions are the statements inside a command body that produce effects.

### `mutate`

Assigns a value to an entity property. The expression is evaluated against the current context and stored on the instance.

```manifest
command complete() {
  guard not self.completed
  mutate completed = true
  mutate completedAt = now()
}
```

### `emit`

Emits an event and optionally attaches an inline payload. Also produces an action result with the evaluated value.

```manifest
command increment() {
  mutate value = value + 1
  emit CounterIncremented
}
```

### `compute`

Calculates a value **without mutating** entity state. `compute <name> = <expr>` binds `<name>` as a command-scoped local into the evaluation scope, available to later actions and to event-payload expressions in the same command. The binding is never persisted and never appears in `getInstance`/`getAllInstances`. Bare `compute <expr>` just returns the value. To persist a field, use `mutate`.

The compiler emits a `COMPUTE_USED_AS_MUTATE` warning when a `compute` target matches a declared property — the common sign of a mis-used `mutate`.

```manifest
command addItem(amount: number) {
  compute newQuantity = quantity + amount   // ephemeral local, not persisted
  mutate quantity = newQuantity              // persists
}
```

### `publish`

External delivery of the **named** event `target` (e.g. `publish OrderShipped`). Requires `RuntimeOptions.outboxStore`: when it is absent the command **fails closed** with a `MISSING_OUTBOX_STORE` error and persists nothing. When configured, the named event is delivered in-process (result + listeners + reactions) AND durably enqueued to the outbox, joining the command transaction under a `TransactionProvider`. The compiler errors `EMIT_ACTION_UNKNOWN_EVENT` if `target` is missing or undeclared. In deterministic mode, throws `ManifestEffectBoundaryError`.

### `persist`

Explicitly flushes the command's pending working-copy changes to the store at its point in the action sequence, then clears the pending change set. It does not open or close a transaction. Under a `TransactionProvider` the flush joins the command transaction (atomic-on-failure); without one it is an immediate, non-reversible write. Multiple `persist` actions are permitted. In deterministic mode, throws `ManifestEffectBoundaryError`.

### `effect`

Invokes the host `RuntimeOptions.effectHandler` with the evaluated value and command context (`{ name?, value, commandName, entityName?, instanceId?, context }`); the handler's result becomes the action result. An optional `effect <name> = <expr>` form names the effect. When no handler is configured the command **fails closed** with a `MISSING_EFFECT_HANDLER` error. In deterministic mode, throws `ManifestEffectBoundaryError`.

<Warning>
`publish` and `effect` are fail-closed adapter actions: without `outboxStore` / `effectHandler` respectively the command returns `{ success: false }` (it does not silently no-op). `emit` is in-process and always available. When `deterministicMode` is `true`, calling `publish`, `effect`, or `persist` throws `ManifestEffectBoundaryError` — this is by design to enforce the effect boundary contract.
</Warning>

## The `emits` keyword

You can declare events at the command level using the `emits` keyword outside the action body. This serves as a declaration that the command emits a specific event after all actions succeed.

```manifest
entity Order {
  property status: string = "pending"

  command complete() {
    guard self.status == "in_progress"
    mutate status = "completed"
    emit OrderCompleted
  }
}

event OrderCompleted: "order.completed" {
  orderId: string
  completedAt: number
}
```

See [Events](/language/events) for how to declare event schemas.

## Command constraints (vNext)

Commands can declare their own `constraint` blocks that are evaluated after policies but before guards. The same severity levels (`ok`, `warn`, `block`) apply:

```manifest
entity Order {
  property status: string = "pending"
  property amount: number = 0

  command updateStatus(newStatus: string) {
    constraint notCancelled: self.status != "cancelled" "Cannot update a cancelled order"
    constraint amountVerified:warn self.amount >= 100 "Orders under $100 require manual approval"

    mutate status = newStatus
    emit OrderStatusChanged
  }
}
```

A failing `block` constraint stops execution and surfaces `constraintOutcomes` in the `CommandResult`. A `warn` constraint produces an outcome entry but does not stop execution.

## CommandResult shape

Every call to `runCommand` returns a `CommandResult`:

```typescript
interface CommandResult {
  success: boolean;
  error?: string;

  // Present when a policy denied execution
  policyDenial?: {
    policyName: string;
    expression: string;
  };

  // Present when a guard halted execution
  guardFailure?: {
    index: number; // 1-based
    expression: string;
    resolved: { expression: string; value: unknown }[];
  };

  // Present when constraints were evaluated
  constraintOutcomes?: ConstraintOutcome[];

  // Present when optimistic concurrency detected a conflict
  concurrencyConflict?: ConcurrencyConflict;

  // All events emitted during this run
  emittedEvents: EmittedEvent[];

  // ~~lastActionResult?: unknown;~~
  result?: unknown; // last action value
}
```

> **Correction (2026-07-15):** The field is `result?: unknown`. There is no `lastActionResult` on
> `CommandResult` in `runtime-engine.ts` @ v3.6.4.

  { newStatus: 'processing' },
  { entityName: 'Order', instanceId: 'order-abc' },
);

  if (result.policyDenial) {
    console.error('Denied by policy:', result.policyDenial.policyName);
  } else if (result.guardFailure) {
    console.error(`Guard ${result.guardFailure.index} failed:`, result.guardFailure.formatted);
  } else if (result.constraintOutcomes) {
    const blocked = result.constraintOutcomes.filter((o) => o.severity === 'block' && !o.passed);
    console.error(
      'Blocked by constraints:',
      blocked.map((o) => o.code),
    );
  }
  process.exit(1);
}

console.log('Events:', result.emittedEvents);
```

## Retry policies

Commands can declare retry policies for transient failures with configurable backoff strategies. When a command fails (guard failure excluded — guards fail fast), the runtime retries according to the declared policy.

```manifest fragment
entity Notification {
  command sendEmail(recipient: string, body: string) {
    retry {
      maxAttempts: 3
      backoff: exponential
      delay: 1000
      jitter: true
    }
    // command body
  }
}
```

### Retry configuration

| Field         | Description                                                                   |
| ------------- | ----------------------------------------------------------------------------- |
| `maxAttempts` | Maximum number of retry attempts                                              |
| `backoff`     | Strategy: `fixed`, `exponential`, or `linear`                                 |
| `delay`       | Base delay in milliseconds (compiles to IR `delayMs`)                         |
| `jitter`      | Add random jitter to prevent thundering herd (default: `false`)               |
| `retryOn`     | One or more error codes that trigger retry (repeat the field per code)        |

~~| `initialDelay` | First retry delay in milliseconds |~~
~~| `maxDelay` | Cap on retry delay |~~

> **Correction (2026-07-15):** Source field is `delay:` (see fixture `72-command-retry-policy.manifest`).
> IR stores `delayMs`. There is no `initialDelay` / `maxDelay` field on the retry block in IR v1.

### Backoff strategies

| Strategy      | Behavior                                                      |
| ------------- | ------------------------------------------------------------- |
| `fixed`       | Constant delay between retries (e.g., 1000ms, 1000ms, 1000ms) |
| `exponential` | Doubles each time (e.g., 1000ms, 2000ms, 4000ms)              |
| `linear`      | Increases by `delay` each time (e.g., 1000ms, 2000ms, 3000ms) |

### Retry events

~~The runtime emits events for each retry attempt and when retries are exhausted:~~

~~| Event | When emitted |
| `{Command}RetryAttempted` | Each retry attempt, with attempt number |
| `{Command}RetryExhausted` | All retries failed, with total attempts |~~

> **Correction (2026-07-15):** No `{Command}RetryAttempted` / `{Command}RetryExhausted` events exist
> in the reference runtime. Retry status is returned on `CommandResult.retry`
> (`{ attempts, exhausted, lastErrorCode?, delaysMs }`).

<Note>
  Retry policies only apply when the failed attempt's derived error code appears in `retryOn`.
  Guard failures, policy denials, and blocking constraint outcomes are never retried — they
  represent intentional business rule enforcement, not transient errors.
</Note>

## Rate limiting

Rate limiting constrains how frequently a command can be invoked per user, tenant, or globally.
When the limit is exceeded, the command returns a `rateLimitDenial` on `CommandResult`.

```manifest fragment
entity ApiKey {
  property required id: string

  command makeRequest() {
    rateLimit {
      windowMs: 60000
      maxRequests: 100
      scope: user
    }
    // command body
  }
}
```

### Rate limit configuration

| Field         | Description                                                |
| ------------- | ---------------------------------------------------------- |
| `windowMs`    | Time window in milliseconds                                |
| `maxRequests` | Maximum allowed requests within the window                 |
| `scope`       | `user`, `tenant`, or `global` (IR enum; not `user.id`)     |

~~| `window` | Time window in milliseconds |~~
~~| `scope` | `user.id`, `tenant.id`, or `global` |~~
~~| `strategy` | `fixed` or `sliding` window (default: `fixed`) |~~

> **Correction (2026-07-15):** IR `IRRateLimit` requires `maxRequests`, `windowMs`, and
> `scope: "user" | "tenant" | "global"`. Optional `burstAllowance` exists. There is no `strategy`
> field on the rate-limit block.

When the rate limit is exceeded, the `CommandResult` includes:

~~```typescript
{
  success: false,
  rateLimitExceeded: {
    scope: "user.id",
    limit: 100,
    remaining: 0,
    retryAfterMs: 45000
  }
}
```~~

```typescript
{
  success: false,
  rateLimitDenial: {
    scope: 'user', // 'user' | 'tenant' | 'global'
    scopeKey: 'user:user-123',
    limit: 100,
    windowMs: 60000,
    retryAfterMs: 45000,
  },
}
```

> **Correction (2026-07-15):** Field name is `rateLimitDenial`, not `rateLimitExceeded`
> (`runtime-engine.ts` `CommandResult`).

<Warning>
  Rate limiting is enforced by the runtime engine before policies/guards run. It is not a
  network-level rate limiter — it operates within the Manifest runtime context. For HTTP-level rate
  limiting, configure your API gateway or middleware. The reference limiter is in-memory only
  (no durable adapter).
</Warning>

<Note>
  Call shape: `runCommand(commandName, input, { entityName?, instanceId?, ... })`. `instanceId` and
  `entityName` belong in **options**, not in the command parameter object. Command parameters are
  only the named values declared in the command signature.
</Note>

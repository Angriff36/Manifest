Reactions connect events to commands declaratively. When an event is emitted, the runtime evaluates matching reactions in declaration order and dispatches the target command with resolved parameters. This creates reactive workflows without manual event wiring.

## Declaring a reaction

Use the `on` keyword followed by an event name, `run`, and a target command reference:

```manifest
event OrderSubmitted: "order.submitted" {
  orderId: string
  amount: number
}

entity Order {
  property required id: string

  command submit(orderId: string, amount: number) {
    emit OrderSubmitted { orderId: orderId, amount: amount }
  }
}

entity OrderProcessor {
  property required id: string
  property status: string = "idle"

  command processOrder(orderId: string, amount: number) {
    guard amount > 0
    mutate status = "processing"
  }
}

on OrderSubmitted run OrderProcessor.processOrder
  resolve "processor-1"
  params {
    orderId: payload.orderId
    amount: payload.amount
  }
```

The `resolve` expression evaluates to the ID of the target entity instance. The `params` block maps values from the event payload into the command's parameters. `resolve` is always required on a single-target reaction; `params` is optional and may be omitted when the target command takes no arguments.

## Resolve expressions

The `resolve` clause provides the ID of the entity instance that receives the dispatched command. It evaluates to a single value — the instance's identifier:

```manifest fragment
on TaskAssigned run Notification.sendAssigned
  resolve payload.assigneeId
  params {
    taskId: payload.taskId
  }
```

The runtime always dispatches `runCommand` with the resolved `instanceId` (`docs/spec/semantics.md` § Reactions). There is **no** silent skip when the instance is missing — the target command still runs; `self`/`this` are unbound when lookup fails, so mutates that need an instance do not persist. Seed target instances (or use a `create` target command) before relying on a reaction.

Use a literal string when the target instance is fixed at design time (a singleton handler, a system processor, etc.):

```manifest fragment
on OrderSubmitted run OrderProcessor.processOrder
  resolve "processor-1"
  params {
    orderId: payload.orderId
    amount: payload.amount
  }
```

Use `payload._subject.id` to target the same entity instance that emitted the event, or `payload.<field>` when the target ID is carried explicitly in the event payload.

## Parameter mappings

The `params` block maps event payload fields to command parameters. Each entry is an expression evaluated in the event's enriched context:

```manifest fragment
params {
  orderId: payload.orderId // Direct field reference
  total: payload.amount * 1.1 // Expression with arithmetic
  currency: "USD" // Literal value
}
```

Inside a `params` block, `payload.*` references the event payload fields. The expression language supports the full set of operators: arithmetic, comparison, logical, conditional, and function calls.

## Follow-on emit payloads

A reaction only sees the fields the triggering event actually carries. By default an `emit` carries the command's input plus the last action result — so a follow-on reaction reading `payload.<field>` for anything else resolves `undefined`. To pass real values down a reaction chain, declare them explicitly with an `emit Event { field: expr }` block. The fields are computed **at emit time**, in the emitting command's context:

```manifest
entity ScheduleShift {
  property required id: string
  property scheduleId: string = ""

  command assign(scheduleId: string) {
    mutate scheduleId = scheduleId
    emit ScheduleShiftCreated { scheduleId: self.scheduleId }
  }
}

event ScheduleShiftCreated: "scheduleShift.created" {}

// The follow-on reaction now reads a real value, not undefined
on ScheduleShiftCreated run Schedule.syncShiftCount
  resolve self.scheduleId
```

Each payload field is `name: <expr>`, where the expression is evaluated in the emitting command's context (`self.*`, command inputs). The declared fields are merged into the emitted event's `payload`, where a follow-on reaction's `resolve`, `params`, `fanOut where`, and `count` predicates can read them as `self.<field>` / `payload.<field>`. This closes the gap consumers otherwise papered over with hand-written after-emit middleware.

## Fan-out reactions (1:N cascades)

A single-target reaction dispatches on ONE resolved instance. For a **1:N cascade** — run a command on _every_ related row, e.g. "when a gig is cancelled, unassign every staff member assigned to it" — use the `fanOut` form. It replaces `resolve` with a collection match:

```manifest
entity Gig {
  property required id: string
  property status: string = "active"
  hasMany staff: GigStaff

  command cancel() {
    mutate status = "cancelled"
    emit GigCancelled
  }
}

entity GigStaff {
  property required id: string
  property gigId: string = ""
  property status: string = "assigned"
  belongsTo gig: Gig

  command unassign(reason: string) {
    mutate status = "unassigned"
  }
}

store Gig in memory
store GigStaff in memory

event GigCancelled: "gig.cancelled" {}

on GigCancelled fanOut GigStaff where gigId = self.id
  run unassign
  params {
    reason: "gig cancelled"
  }
```

This dispatches `GigStaff.unassign` on **every** `GigStaff` row where `row.gigId == self.id` (the source instance's id, read from the event payload). Each match runs the target command through the normal pipeline — its own guards, policies, actions, and emits fire — exactly like a hand-written after-emit handler, but declared.

- `where <field> = <sourceExpr>` selects the collection; `<field>` is a property on the matched entity (an indexed or foreign-key field is preferred — the Convex projection reads it via `withIndex`, falling back to a filter otherwise).
- `<sourceExpr>` is evaluated against the event payload (`self.id` / `payload.*`), so it sees the source entity's id and any declared `emit Event { field }` values.
- `params` maps the same way as a single-target reaction and is shared across all matches. Inside fan-out, `self` refers to the matched row from the source entity; `target` refers to the target entity instance.
- The reaction depth limit and `correlationId`/`causationId` propagation apply per dispatched command.

Fan-out is the declarative replacement for the common "query children by FK, loop, dispatch" middleware pattern: cancel every line item, create a shipment per line, release every reservation, deactivate every child account, close every tip pool in a period, etc.

## Foreach-create (cross-entity fan-out)

Same-entity `fanOut … run unassign` updates each matched row in place. **Foreach-create** is the cross-entity form: match one collection, then `run Target.create` (or another target-entity command) **without** binding the matched row as `instanceId` / `docId`. Each match allocates a new target row; use matched-row fields in `params` to seed it (`self` is the fan-out source row).

Shipped in Manifest **3.6.36+** (conformance fixture `112-fanout-foreach-create`).

```manifest fragment
on OrderPlaced fanOut LineItem where orderId = payload.orderId
  run Shipment.create
  params {
    lineItemId: self.id
    quantity: self.quantity
  }
```

This is how you declare “for each matching source row, create a child” — not a separate `foreach` keyword. The keywords are still `fanOut` + `run Target.create`.

## Natural-key upsert (`match … else create`)

**Shipped in Manifest 3.6.38+** — natural-key lookups for single-target and fan-out reactions.

Instead of resolving by `instanceId`, match the target entity by one or more field values (natural key). If a match is found, run the command on that instance; if zero rows match and `else create` is set, run the command without `instanceId` (allocate a new row / createVia path).

### Single-target: lookup or allocate

```manifest fragment
event ShoppingCartSubmitted: "cart.submitted" {
  cartId: string
  userEmail: string
}

on ShoppingCartSubmitted run Order.create
  match orderId = payload.cartId
  else create
  params {
    orderId: payload.cartId
    userEmail: payload.userEmail
  }
```

- If an `Order` with `orderId == cartId` exists → run `Order.create` on that row (update path, passing `instanceId`).
- If no match → run `Order.create` without `instanceId` (allocate / create a new row).
- If multiple matches → pick the lexicographically smallest `id` (deterministic tiebreak).
- Soft-deleted rows (`deletedAt != null`) are excluded from the match.

### Fan-out: per-row lookup or allocate

Fan-out can also use `match … else create`. When a source row matches, the runtime looks up the target entity by the match predicates:

```manifest fragment
event OrderPlaced: "order.placed" {
  orderId: string
  items: array
}

on OrderPlaced fanOut LineItem where orderId = payload.orderId
  match sku = self.sku
  else create
  run InventoryAdjustment.allocate
  params {
    sku: self.sku
    orderId: payload.orderId
  }
```

For each `LineItem` where `orderId == payload.orderId`:
- If an `InventoryAdjustment` with `sku == self.sku` exists → run `allocate` on that row.
- If no match → run `allocate` without `instanceId` (allocate new row).

### Syntax

```
on <Event> run <Entity>.<command>
  match <field> = <expression>, ...
  [else create]
  params { ... }
```

- `match` is mutually exclusive with `resolve`. Use `match` when you have natural-key predicates; use `resolve` when you have a direct `instanceId`.
- Predicates are **pure equality**, ANDed together. All must hold for a row to match.
- `else create` is optional. Without it, zero matches = no-op. With it, zero matches = run command without `instanceId`.
- For fan-out, `match` applies to `targetEntity`, not the fan-out source collection. Source rows are selected by the `fanOut … where …` clause; `match` then looks up each selected row's target.

## Aggregate count expressions

A reaction often needs to **recompute a stored count on a parent** after a child event — "how many children reference this parent?" — the pattern behind hand-written "count children where the foreign key equals the parent, then patch the parent" after-emit middleware. The `count(...)` aggregate makes that declarative. It is an expression, so it can appear anywhere a reaction param value does:

```manifest
entity Schedule {
  property required id: string
  property shiftCount: number = 0

  command syncShiftCount(shiftCount: number) {
    mutate shiftCount = shiftCount
  }
}

entity ScheduleShift {
  property required id: string
  property scheduleId: string = ""
  belongsTo schedule: Schedule

  command assign(scheduleId: string) {
    mutate scheduleId = scheduleId
    emit ScheduleShiftCreated { scheduleId: self.scheduleId }
  }
}

event ScheduleShiftCreated: "scheduleShift.created" {}

on ScheduleShiftCreated run Schedule.syncShiftCount
  resolve self.scheduleId
  params {
    shiftCount: count(ScheduleShift where scheduleId == self.scheduleId)
  }
```

When a `ScheduleShift` is assigned, the reaction resolves its parent `Schedule` (`self.scheduleId` is the event payload value), counts `ScheduleShift` rows whose `scheduleId` equals that value, and stores the result via `syncShiftCount`.

### Syntax

```
count(<Entity> where <field> == <value>, <field2> == <value2>, ...)
```

- `<Entity>` is the counted collection. Predicate values (`<value>`) are expressions evaluated in the reaction's context — the event payload, just like other params (`self.*` / `payload.*`).
- Predicates are **pure equality**, ANDed together. A second predicate adds a filter — e.g. counting only in-progress tasks:

```manifest fragment
params {
  currentTaskCount: count(PrepTask where stationId == self.stationId, status == "in_progress")
}
```

- `count` is a **contextual** operator, not a reserved word: it is recognized only in the `count(<Entity> where ...)` shape, so a property can still be named `count`.
- At least one predicate (the foreign-key match) is required. There is **no** group-by, join, multi-hop traversal, or arbitrary SQL — `count` only. Scope is deliberately narrow.

### Projection notes

The runtime evaluates `count` by scanning the entity's collection and counting rows matching every predicate (deterministic — count is order-independent). The Convex projection reads via the counted entity's `by_<field>` index on the foreign-key predicate when one exists (the `belongsTo`/`ref` FK or an `indexed` field), applying any remaining predicates — plus the entity's tenant and soft-delete filters — as post-read `.filter`s, then `.length`. With no indexed predicate it falls back to a table scan and emits an `info` diagnostic.

### What this does NOT replace

`count(...)` covers the **per-parent** recompute (one parent's count, updated by the child event that points at it). It does not cover:

- **Tenant-wide reconcile** (recompute _every_ parent on each event) or **dual-target updates** (a reassignment that moves a child from parent A to parent B needs both updated). These are multi-step / conditional fan-out — model them as sagas, actions, or keep the hand middleware. They are intentional scope boundaries, not gaps in `count`.

## Reaction scopes

Reactions can be declared at three levels:

| Scope   | Location                      | Behavior                              |
| ------- | ----------------------------- | ------------------------------------- |
| Program | Top-level in `.manifest` file | Global — fires for any matching event |
| Module  | Inside a `module` block       | Scoped to the module                  |
| Entity  | Inside an `entity` block (`on Event run …` / `fanOut`) | Collected into IR with an `entity` field |

Entity-scoped reactions must use the `on Event run …` / `fanOut` form. Bare `behavior` / `on Event { … }` blocks inside an entity are a hard compile error (conformance fixture 110; `docs/spec/semantics.md` § Entities).

```manifest
event OrderSubmitted: "order.submitted" {
  orderId: string
}

event OrderFailed: "order.failed" {}
event OrderReprocessed: "order.reprocessed" {}

entity OrderProcessor {
  property required id: string
  property status: string = "idle"

  command processOrder(orderId: string) {
    mutate status = "processing"
  }
}

entity Order {
  property required id: string
  property status: string = "pending"

  command submit(orderId: string) {
    mutate status = "submitted"
    emit OrderSubmitted { orderId: orderId }
  }

  command fail() {
    mutate status = "failed"
    emit OrderFailed
  }

  command reprocess() {
    mutate status = "pending"
    emit OrderReprocessed
  }

  // Entity-level reaction -- fires within the entity's context
  on OrderFailed run Order.reprocess
    resolve payload._subject.id
}

// Program-level reaction -- fires globally
on OrderSubmitted run OrderProcessor.processOrder
  resolve "processor-1"
  params {
    orderId: payload.orderId
  }
```

Reactions at all three scopes are collected into the IR's `reactions` array. Module-scoped reactions carry a `module` field; entity-scoped reactions carry an `entity` field. The runtime evaluates all matching reactions regardless of scope.

## Cascading reactions and depth limit

When a reaction's target command emits its own events, those events can trigger further reactions. This creates cascading reaction chains:

```
OrderSubmitted
  -> OrderProcessor.processOrder
    -> OrderProcessing
      -> Warehouse.prepareShipment
```

The runtime enforces a maximum reaction depth (`MAX_REACTION_DEPTH = 10`). If a chain exceeds this limit, the runtime throws a `ManifestReactionDepthError` and stops processing further reactions. This prevents infinite loops from circular reaction chains.

<Warning>
  Design your reaction chains to terminate naturally. If event A triggers command B which emits
  event C which triggers command D which emits event A, the runtime will halt after 10 iterations
  with a `ManifestReactionDepthError`. This is a safety net, not a design pattern — circular
  reaction chains indicate a modeling problem.
</Warning>

## correlationId and causationId propagation

Reactions propagate workflow metadata through the chain:

- **`correlationId`** stays the same across the entire workflow. It is set on the initial event and carried through every subsequent reaction.
-

`causationId` on each reaction-triggered command is the triggering event's **name** (`emitted.name`), not an emit/instance ID (`docs/spec/semantics.md` § Reactions).

This enables cross-reaction tracing for debugging distributed workflows. See [Events](/language/events) for how `correlationId` and `causationId` are attached to emitted events.

## Deterministic ordering

Reactions execute in declaration order. If multiple reactions match the same event, they are dispatched in the order they appear in the source file. This guarantees deterministic behavior: given identical IR, identical runtime context, and identical input, the reaction chain always produces the same result.

The runtime evaluates matching reactions synchronously within the same command execution turn — reactions do not queue asynchronously. This means the full reaction chain completes before the original `CommandResult` is returned to the caller.

## Complete example

```manifest
event OrderSubmitted: "order.submitted" {
  orderId: string
  amount: number
  customerId: string
}

event InvoiceCreated: "invoice.created" {
  invoiceId: string
  orderId: string
  total: number
}

event ShipmentPrepared: "shipment.prepared" {}

entity Order {
  property required id: string
  property status: string = "pending"

  command submit(orderId: string, amount: number, customerId: string) {
    mutate status = "submitted"
    emit OrderSubmitted { orderId: orderId, amount: amount, customerId: customerId }
  }
}

entity Invoice {
  property required id: string
  property orderId: string = ""
  property total: number = 0
  property status: string = "draft"
  belongsTo order: Order

  command createFromOrder(orderId: string, total: number) {
    guard total > 0
    mutate orderId = orderId
    mutate total = total
    mutate status = "issued"
    emit InvoiceCreated
  }
}

entity Shipping {
  property required id: string
  property orderId: string = ""
  property status: string = "pending"
  belongsTo order: Order

  command prepareShipment(orderId: string) {
    mutate orderId = orderId
    mutate status = "preparing"
    emit ShipmentPrepared
  }
}

// Reaction chain: OrderSubmitted -> create invoice
on OrderSubmitted run Invoice.createFromOrder
  resolve "invoice-auto"
  params {
    orderId: payload.orderId
    total: payload.amount
  }

// Reaction chain: InvoiceCreated -> prepare shipment
on InvoiceCreated run Shipping.prepareShipment
  resolve "shipping-auto"
  params {
    orderId: payload.orderId
  }
```

`runCommand(commandName, input, options?)` — not
entity-first. `amount` is command input; `entityName` / `instanceId` go in options.
Do **not** wrap fenced code in `` — Mintlify MDX treats the fence as broken and
parses `{…}` as JSX (`Could not parse expression with acorn`), which blocks deploy.

```typescript
import { RuntimeEngine } from '@angriff36/manifest';
import { ir } from './compiled.ir.json';

const runtime = new RuntimeEngine(ir, {
  user: { id: 'system', role: 'worker' },
});

// Seed instances
runtime.createInstance('Invoice', { id: 'invoice-auto' });
runtime.createInstance('Shipping', { id: 'shipping-auto' });

// Emitting OrderSubmitted triggers the reaction chain
const result = await runtime.runCommand(
  'submit',
  { orderId: 'order-001', amount: 250, customerId: 'customer-123' },
  { entityName: 'Order', instanceId: 'order-001' },
);

// The reaction chain fires automatically:
// OrderSubmitted -> Invoice.createFromOrder -> InvoiceCreated -> Shipping.prepareShipment
console.log(result.emittedEvents.length); // Events from all reactions
```

See [Events](/language/events) for declaring event schemas and [Commands](/language/commands) for the command execution pipeline. For async reactions that defer execution to a background queue, see [Async commands](/language/async-commands).

An entity is the primary building block in Manifest. It combines a typed data schema with behavioral constraints, commands, and computed properties into a single declaration. The compiler turns entities into IR that the runtime can instantiate, query, and mutate.

## Basic declaration

Use the `entity` keyword followed by a name and a block. Property declarations go inside the block.

```manifest
entity Product {
  property required name: string
  property price: number = 0
  property inStock: boolean = true
  property description: string = ""
}
```

### External entities

Prefix the declaration with `external` to mark an entity as owned by another system or file. The compiler records `external: true` on the IR, and persistence projections (Prisma, Drizzle, Kysely, Convex, prisma-store) skip it — they emit no table or model for it, only honoring relationships that point at it. Use this to reference data managed outside the current program.

```manifest
external entity ImportedUser {
  property required id: string
  property email: string
}
```

`external` is a contextual modifier rather than a reserved word, so a property may still be named `external` (`property external: string`).

## Properties

Every property has a type.

`timestamp` **is** a valid property type — it is a normative alias for `datetime` (`docs/spec/semantics.md:281,286-289`). Runtimes MUST treat `timestamp` identically to `datetime`. Common scalar types include `string`, `number`, `boolean`, plus date/time primitives `datetime` / `timestamp` / `date` / `time` / `duration`, `decimal` / `money`, collections, enums, and value-object names. When a property is omitted at create time, the runtime applies the property's `defaultValue` if present, otherwise the **type default** (for example `""`, `0`, `false`) — not `undefined`. An explicit `required` modifier is not satisfied by a zero-filled type default alone (`E_REQUIRED`). See [Types](/language/types).

### Modifiers

Place modifiers between the `property` keyword and the name:

| Modifier     | Meaning                                                                                                                                                                                                                                                                                                                                                                 |
| ------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `required`   | The caller must supply a value. The runtime rejects a create (`E_REQUIRED`) when the property has no supplied value, `defaultValue`, `autoNow`, auto-managed field, or creating-command action; a zero-filled type default does not satisfy it.                                                                                                                         |
| `optional`   | Explicitly marks the property as not required. Equivalent to the default behavior.                                                                                                                                                                                                                                                                                      |
| `readonly`   | Immutable after creation. The runtime rejects an update that changes the value (`E_READONLY`); writes during the creating command and same-value writes are allowed.                                                                                                                                                                                                    |
| `unique`     | Values must be unique within the entity collection. The runtime rejects a create/update whose non-null value another instance already holds (`E_UNIQUE`, scanned within the active tenant scope); adapters may additionally emit a database unique constraint.                                                                                                          |
| `encrypted`  | Declares the property for field-level encryption. The runtime encrypts at store-write boundaries and decrypts on read. Requires an `EncryptionProvider` in runtime options.                                                                                                                                                                                             |
| `masked`     | Applies dynamic data masking to the property value on read. Write `masked` alone for the default `redact` strategy, or `masked(strategy, ...numbers)` to pick a strategy with optional numeric parameters. Strategies are exactly: `redact`, `partial`, `email`, `phone`, `last4`. Add a trailing `unmask when <expr>` clause to bypass masking for authorized readers. |
| `searchable` | Declares the property for full-text search indexing (string-only). Adds a `search(text, query)` built-in with AND-semantics whole-word matching. Projections emit platform-specific full-text indexes (e.g., Prisma `@@fulltext`, Drizzle GIN tsvector).                                                                                                                |
| `private`    | Prevents inclusion in public API projections. The property is omitted from generated client types and API response shapes, and the reference runtime strips every `private` property (with or without `masked`) from `getInstance`/`getAllInstances`. Execution paths (guards, constraints, computed properties) use the raw read path and still see the real value.    |
| `indexed`    | Signals to adapters that the property should be indexed for faster lookups. Projections emit platform-specific index declarations.                                                                                                                                                                                                                                      |

```manifest
entity User {
  property required id: string
  property required email: string
  property optional displayName: string = ""
  property readonly createdAt: number = 0
}
```

### Default values

Any property can carry a literal default. When you create an instance and omit that property, the runtime applies the default. Explicitly supplying a value — including an empty string `""` — skips the default entirely.

```manifest
entity Task {
  property status: string = "pending"
  property priority: number = 1
  property completed: boolean = false
}
```

<Note>
  Defaults apply only when the property is **omitted** from the creation payload. An empty string
  `""` counts as a provided value and overrides the default.
</Note>

## Relationships

Relationships declare connections between entities. The runtime resolves them synchronously from the current store context.

| Keyword     | Cardinality      | Returns                                                          |
| ----------- | ---------------- | ---------------------------------------------------------------- |
| `hasMany`   | One-to-many      | Array of related instances (never `null`; empty array when none) |
| `hasOne`    | One-to-one       | Single related instance or `null`                                |
| `belongsTo` | Many-to-one      | Single related instance or `null`                                |
| `ref`       | Simple reference | Single related instance or `null`                                |

```manifest
entity Author {
  property required name: string
  hasMany books: Book
  computed bookCount: number = self.books.length
}

entity Book {
  property required title: string
  property year: number = 2024
  belongsTo author: Author
}
```

For `belongsTo` and `ref`, the foreign key property on the source instance holds the target ID. For `hasMany` and `hasOne`, the runtime queries the target entity for instances whose foreign key matches the current instance's ID.

<Tip>
  Accessing a relationship on a non-existent instance returns `null` for singular relationships and
  `[]` for `hasMany`. Circular relationships are handled gracefully.
</Tip>

## Constraints

Constraints are boolean expressions that validate entity state. You can attach them at the entity level (evaluated whenever an instance is created or mutated) or at the command level (evaluated before that command runs — see [Commands](/language/commands)).

```manifest
entity Product {
  property required name: string
  property price: number = 0

  constraint pricePositive: self.price > 0 "Price must be positive"
  constraint nameNotEmpty: self.name != "" "Name cannot be empty"
}
```

### Constraint severity

Each constraint has a severity that controls what happens when it evaluates to `false`:

| Severity | Behavior                                                                                                                                                                              |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `ok`     | Informational only. Does not halt execution. The runtime forces `passed = true` — an `ok` constraint never appears as a failure. The expression is still evaluated for observability. |
| `warn`   | Produces a `ConstraintOutcome` with the real result but does **not** halt execution.                                                                                                  |
| `block`  | Halts execution when the expression evaluates to `false` (default).                                                                                                                   |

```manifest
entity Task {
  property status: string = "pending"
  property priority: number = 1
  property dueDate: number = 0

  constraint okInfo {
    expression: priority > 0
    severity: ok
    message: "Priority is positive"
  }

  constraint warnHighPriority {
    expression: priority <= 5
    severity: warn
    message: "Priority is higher than recommended"
  }

  constraint blockStatus {
    expression: status in ["pending", "in_progress", "completed", "cancelled"]
    severity: block
    message: "Status must be one of the allowed values"
  }
}
```

The long-form block syntax also supports `messageTemplate` and `details` for structured diagnostics:

```manifest
constraint amountLimit:block self.amount > 10000 {
  messageTemplate: "Amount {currentAmount} exceeds limit of {maxAmount}"
  details: {
    maxAmount: 10000
    currentAmount: self.amount
    excessAmount: self.amount - 10000
  }
}
```

### Overrideable constraints

Mark a `block` constraint `overrideable` to allow authorized bypass at runtime. Pair it with an `overridePolicy` to control who can authorize the override.

```manifest
entity FinancialRecord {
  property required amount: number = 0
  property required status: string = "pending"

  command process() {
    constraint overrideable requiresApproval {
      expression: self.status == "approved" or self.amount <= 500
      severity: block
      message: "Must be approved before processing"
      overridePolicy: canApproveLargeAmounts
    }

    mutate status = "processed"
    emit RecordProcessed
  }
}

policy canApproveLargeAmounts override:
  user.role == "manager" or user.role == "admin"
  "Only managers and admins can approve large amounts"
```

When a caller supplies an `OverrideRequest` matching the constraint code, the runtime checks the policy, marks the outcome `overridden: true`, and emits a synthetic `OverrideApplied` event.

## Composite keys and foreign key structure

### Composite primary keys

Declare `key [field1, field2]` inside an entity to define a composite primary key. The listed fields together form the entity's identity — no single `id` field is needed.

```manifest
entity Order {
  key [tenantId, id]
  property required tenantId: string
  property required id: string
  property total: number = 0
}
```

When the Prisma projection sees a `key`, it emits `@@id([tenantId, id])` instead of a single `@id` field. The reference runtime also honors `entity.key`: it derives a deterministic canonical identity string from the key tuple (each component percent-encodes `%` and `|`, joined with `|`) and persists each instance under that string, so a composite-key entity is addressable by it through `getInstance`/`updateInstance`/`deleteInstance` and `runCommand`'s `instanceId`. Instances differing in any key component are distinct even when they share another column's value.

### Alternate keys

Declare `unique [field1, field2]` to define a non-PK uniqueness constraint (an alternate key). You can declare multiple.

```manifest
entity AdminChatParticipant {
  key [tenantId, id]
  unique [tenantId, threadId, userId]
  property required tenantId: string
  property required id: string
  property required threadId: string
  property required userId: string
}
```

The Prisma projection emits `@@unique([...])` for each group, and Drizzle emits the equivalent constraint. Convex, Kysely, OpenAPI, and Zod do not emit uniqueness constraints for alternate keys, and the reference runtime does not enforce alternate-key uniqueness.

### Composite foreign keys

For `belongsTo` and `ref` relationships, declare the FK columns explicitly when your relationship spans multiple columns:

```manifest
// Single-column FK using the shorthand (backward-compatible)
belongsTo author: Author with authorId

// Composite FK — explicit fields and references
belongsTo thread: AdminChatThread fields [tenantId, threadId] references [tenantId, id] onDelete cascade

// Cross-order composite FK
belongsTo proposal: Proposal fields [proposalId, tenantId] references [id, tenantId] onDelete cascade
```

`fields` names properties on the current entity; `references` names the corresponding columns on the target. If `references` is omitted, `[id]` is assumed.

**Runtime note**: The Manifest runtime resolves composite foreign keys. A `belongsTo`/`ref` with `fields.length > 1` (and the inverse `hasMany`/`hasOne`) selects the target row where every mapped `fields`/`references` column is equal, so it picks the exact row even when several targets share a first-column value. Local FK columns are paired with target columns via `references`; when `references` is absent or mismatched, the target entity's declared `key` columns are paired positionally, else the field names are assumed to match. If any local FK column is unset the relationship resolves to `null`/`[]`.

### Referential actions

Add `onDelete` and/or `onUpdate` to any `belongsTo` or `ref` to declare referential integrity behavior. Valid values: `cascade`, `restrict`, `setNull`, `setDefault`, `noAction`.

```manifest
belongsTo account: Account with tenantId onDelete restrict
belongsTo item: LineItem fields [tenantId, itemId] references [tenantId, id] onDelete cascade onUpdate restrict
```

When no action is declared, the projection emits nothing and the target platform uses its own default.

Referential actions **are** enforced in the reference runtime via `ReferentialActionApplier` (`src/manifest/runtime-referential-actions.ts`, wired at `runtime-engine.ts:1245-1270`). The runtime applies cascade/restrict/setNull/setDefault semantics on `deleteInstance()` based on the IR relationship configuration. Projection adapters also emit referential-integrity constraints (for example Prisma `onDelete`/`onUpdate`).

## Concurrency controls (vNext)

Entities can declare optimistic concurrency controls using `versionProperty` and `versionAtProperty`. The runtime compares the caller-supplied version against the stored value on every mutation.

```manifest
entity Document {
  property required id: string = ""
  property title: string = ""
  property content: string = ""

  versionProperty version: number
  versionAtProperty versionAt: number

  command update(newTitle: string, currentVersion: number) {
    guard self.version == currentVersion
    mutate title = newTitle
    emit DocumentUpdated
  }
}
```

If the caller's version does not match, the runtime returns a `ConcurrencyConflict` and does not apply any mutations:

```typescript
{
  entityType: "Document",
  entityId: "doc-1",
  expectedVersion: 2,
  actualVersion: 3,
  conflictCode: "VERSION_MISMATCH"
}
```

## State transitions (vNext)

Declare allowed state changes with the `transition` keyword. The runtime validates every mutation of the property against the rules before applying it.

```manifest
entity Document {
  property required id: string
  property status: string = "draft"

  transition status from "draft" to ["review", "archived"]
  transition status from "review" to ["published", "draft"]
  transition status from "published" to ["archived"]

  command submit() {
    mutate status = "review"
  }

  command approve() {
    mutate status = "published"
  }

  command archive() {
    mutate status = "archived"
  }
}
```

<Note>
  Transition validation runs **before** entity constraint validation. If the current value has no
  matching `from` rule, the transition is unconstrained from that state. Properties not mentioned in
  any transition rule are always unconstrained.
</Note>

## Default policies (vNext)

Attach a `default policy` to an entity to apply it to every command that does not declare its own policy list:

```manifest
entity Task {
  property required title: string
  property status: string = "pending"

  default policy RequireAuth execute: user.id != null "Authentication required"

  command complete() {
    guard self.assigneeId == user.id or user.role == "admin"
    mutate status = "completed"
    emit TaskCompleted
  }

  command reassign(newAssigneeId: string) {
    guard user.role == "admin" or user.role == "manager"
    mutate assigneeId = newAssigneeId
  }
}
```

Both `complete` and `reassign` inherit `RequireAuth` because neither declares an explicit policy. See [Guards and Policies](/language/guards-policies) for the full inheritance rules.

## Cross-entity constraints

Constraints can reference properties on related entities through relationship traversal. The runtime resolves the relationship chain and evaluates the constraint against the related instance's properties.

```manifest
entity Customer {
  property required id: string
  property status: string = "active"
}

entity Order {
  property required id: string
  property required customerId: string
  property amount: number = 0

  belongsTo customer: Customer with customerId

  constraint customerActive: self.customer.status == "active"
    "Customer must be active to place orders"
}
```

The runtime enriches resolved relationship instances with `_entity` metadata, enabling arbitrary-depth chaining (e.g., `self.order.customer.address.city`). When a relationship resolves to `null` (missing or nonexistent target), any member access on it evaluates to `null`, which typically causes the constraint to fail.

<Note>
  Cross-entity constraints work because the runtime resolves relationships synchronously from the
  current store context. No parser or IR compiler changes are needed -- the existing infrastructure
  handles nested member access syntax and compilation.
</Note>

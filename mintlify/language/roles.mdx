Roles define named permission sets with single-parent inheritance. A role that extends a parent inherits all its permissions transitively. Use roles with the [policy](/language/guards-policies) system to control command authorization based on the user's assigned role.

## Declaring a role

Use the `role` keyword at the program or module level with a block body of `allow` and `deny` declarations:

```manifest
role User {
  allow read
}

role Manager extends User {
  allow write
  allow execute
}

role Admin extends Manager {
  allow delete
  allow override
  deny impersonate
}
```

Each `allow` or `deny` line names an action (or optionally a target) that the role permits or forbids. The `extends` keyword establishes a single-parent inheritance chain.

## Permission inheritance

Roles inherit permissions transitively through the `extends` chain. The compiler walks the chain from root to leaf, unioning all `allow` permissions along the way:

| Role                      | Inherited from | Effective permissions                  |
| ------------------------- | -------------- | -------------------------------------- |
| `User`                    | (none)         | read                                   |
| `Manager` (extends User)  | User           | read, write, execute                   |
| `Admin` (extends Manager) | Manager, User  | read, write, execute, delete, override |

### Compile-time validation

The compiler validates the role graph and produces error diagnostics for:

- **Duplicate role names**: Each role name must be unique within the program or module.
- **Unknown parent**: A role that extends a name not declared as a role produces a compile error.
- **Circular inheritance**: If the `extends` chain forms a cycle (A extends B extends A), the compiler detects it via DFS coloring and produces an error diagnostic listing the cycle path.

## Deny-is-absolute semantics

If any role in the inheritance chain denies an action, the denial takes precedence over all `allow` rules regardless of where they appear:

```manifest
role User {
  allow read
}

role Support extends User {
  allow write
  allow execute
  deny delete       // Support CANNOT delete, even if a parent allows it
}
```

Deny rules are applied after the full inheritance union is computed. This means:

1. Walk the inheritance chain and union all `allow` permissions.
2. Walk the inheritance chain and collect all `deny` permissions.
3. ~~The result is the `effectivePermissions` map.~~ **Correction (2026-07-15) @RYANSIGNED:** Filter allows against denies → `effectivePermissions: IRRolePermission[]`.

A `deny` at any level is absolute. There is no mechanism to override a deny with a subsequent allow. This is by design — it prevents privilege escalation through role chain manipulation.

<Warning>
  Deny is not a suggestion; it is a hard rule. If `User` denies `impersonate`, then `Admin extends
  User` cannot re-allow `impersonate`. The deny from `User` will always win. Structure your role
  hierarchy so that denials only appear at the appropriate level.
</Warning>

## Unknown roles default to deny

If a user's role does not match any declared role, the runtime returns `false` for all permission checks. There are no permissive defaults — an unrecognized role has no permissions. This is consistent with Manifest's default-deny posture. See [Guards and policies](/language/guards-policies) for the broader authorization model.

<Warning>
  Role matching is **exact and case-sensitive**. `'Admin'` and `'admin'` are treated as different
  roles. A caller passing the wrong casing silently gets `false` with no error — the runtime treats
  it as an unknown role, not a near-miss. Ensure that the role strings your identity system produces
  exactly match the names declared in the `.manifest` file.
</Warning>

## Using roles in policies

Reference roles in policy expressions via the `user.role` binding and the built-in permission functions. Authorization is declared at the entity level with named `policy <name> execute:` rules (or a `default policy`); command bodies do not accept a `policy` clause:

```manifest
entity Document {
  default policy RequireAuth execute: user.id != null "Authentication required"

  ~~policy CanDelete execute: hasPermission(user, "delete")~~
  policy CanApprove execute: roleAllows(user.role, "approve") or user.role == "admin"
  policy CanDelete execute: hasPermission("delete")

  command approve() {
    mutate status = "approved"
    emit DocumentApproved
  }

  command removeDocument() {
    mutate status = "deleted"
    emit DocumentDeleted
  }
}
```

## Runtime builtins

Two built-in functions are registered on the runtime engine when roles are present in the IR:

~~| Function | Signature | Returns |
| --- | --- | --- |
| `hasPermission` | `hasPermission(user, permission)` | … |
| `roleAllows` | `roleAllows(roleName, action)` | … |~~

> **Correction (2026-07-15) @RYANSIGNED:** Per `docs/spec/builtins.md` and conformance fixture `71-role-hierarchy.manifest`:

| Function | Signature | Returns |
| --- | --- | --- |
| `hasPermission` | `hasPermission(action, target?)` | `true` if `context.user.role`'s effective permissions grant `action` (optional `target`) |
| `roleAllows` | `roleAllows(roleName, action, target?)` | same check for an explicit role name |

`effectivePermissions` is an `IRRolePermission[]` (not a `Record<string, boolean>`). Lookups scan that array. Passing a user object as the first argument of `hasPermission` always returns `false` (action must be a string).

### `hasPermission(action, target?)`

Uses the bound runtime `user.role`. Returns `false` if there is no role string or the role is undeclared.

```manifest
guard hasPermission("delete")
guard hasPermission("execute", "CreateReport")
```

### `roleAllows(roleName, action, target?)`

Checks whether a named role allows a specific action, without requiring a user object.

```manifest
policy CanExecute execute: roleAllows(user.role, "execute") or user.role == "admin"
```

## Context-sensitive `role` keyword

The `role` keyword is context-sensitive, not a globally reserved keyword. It is emitted as an identifier rather than a reserved token, so existing programs that use `role` as a property name continue to parse correctly:

```manifest
entity UserProfile {
  property role: string = "user"           // property named "role"

  command setRole(newRole: string) {
    mutate role = newRole                   // mutate "role" property
  }
}
```

The parser only treats `role` as a declaration keyword at the top level of a program or module (when followed by an identifier and a block). Inside entity blocks, `role` is an ordinary identifier.

<Note>
  The `role` keyword is only treated as a role declaration at the top level of a program or module.
  Inside entity blocks, `role` is an ordinary identifier. This prevents the keyword from breaking
  existing code that uses `role` as a property name.
</Note>

## IR representation

Roles are stored in the IR as an optional `roles` array on the root `IR` object (and on `IRModule` for module-scoped roles). Each `IRRole` contains:

~~```typescript
interface IRRole {
  name: string;
  extends?: string;
  permissions: IRRolePermission[];
  effectivePermissions: Record<string, boolean>;
}
```~~

> **Correction (2026-07-15) @RYANSIGNED:** IR shape (`src/manifest/ir.ts`):

```typescript
interface IRRolePermission {
  action: string;
  target?: string;
}

interface IRRole {
  name: string;
  module?: string;
  parent?: string; // from `extends` in source
  allow: IRRolePermission[];
  deny: IRRolePermission[];
  effectivePermissions: IRRolePermission[]; // post inheritance + deny
}
```

The runtime reads `effectivePermissions` directly and never re-computes it.

## Complete example

```manifest
role User {
  allow read
}

role Manager extends User {
  allow write
  allow execute
}

role Admin extends Manager {
  allow delete
  allow override
  deny impersonate
}

entity Document {
  property required id: string
  property title: string = ""
  property status: string = "draft"

  default policy RequireAuth execute: user.id != null "Authentication required"

  ~~policy CanRead execute: hasPermission(user, "read")
  policy CanWrite execute: hasPermission(user, "write")
  policy CanDelete execute: hasPermission(user, "delete")~~
  policy CanRead execute: hasPermission("read")
  policy CanWrite execute: hasPermission("write")
  policy CanApprove execute: roleAllows(user.role, "execute")
  policy CanDelete execute: hasPermission("delete")

  command view() {
    emit DocumentViewed
  }

  command edit(newTitle: string) {
    mutate title = newTitle
    emit DocumentEdited
  }

  command approve() {
    guard self.status == "draft"
    mutate status = "approved"
    emit DocumentApproved
  }

  command removeDocument() {
    mutate status = "deleted"
    emit DocumentDeleted
  }
}
```

IR version control persists IR snapshots with semantic version tags, generates changelogs between versions, and verifies integrity via SHA-256 hashing. The pure versioning logic lives in `src/manifest/ir-version-store.ts`; filesystem I/O and the user-facing commands live in the `manifest versions` CLI command group.

## Usage

Versions are managed through the `manifest versions` subcommands against a store directory (default `.manifest-versions`):

```bash
manifest versions save src/app.manifest --auto-tag --label "initial"
manifest versions list
manifest versions show latest
manifest versions diff 1 2 --breaking --sql
manifest versions tag 2 1.1.0
manifest versions rollback 1 -o restored.ir.json
manifest versions verify --all
manifest versions changelog 1 2
```

See the [CLI command reference](/cli/commands) for the full flag set.

## How it works

The store module is pure and deterministic — no I/O — operating on IR and an index structure.

The index (`IRVersionIndex`) holds `storeVersion: 1`, the current highest version number, and an ordered list of `IRVersionMeta` (oldest first). `createVersionIndex`, `addVersionToIndex`, `removeTagFromIndex`, and `tagVersionInIndex` are immutable transforms; tagging a version removes the tag from any other version that previously held it.

`createVersionMeta` extracts metadata from an IR and its provenance: version number, optional semver tag, `irHash`, `contentHash`, `savedAt` (ISO timestamp), `compilerVersion`, `schemaVersion`, and an optional label.

Integrity verification (`verifyIRIntegrity`) recomputes the IR hash with `computeIRHash` and compares it against the stored hash, returning `{ valid, storedIrHash, computedIrHash }`.

Semantic versioning helpers parse and format `major.minor.patch` strings. `autoIncrementSemver` derives the next tag from diff and breaking-change analysis: a breaking change bumps major (resetting minor and patch), any other change bumps minor, no change bumps patch, and the absence of a previous tag yields `0.1.0`.

`resolveVersionRef` resolves a reference to a version number, accepting `latest` (or undefined → current version), a numeric string, or a semver tag.

`generateChangelog` builds a `ChangelogEntry` between two IRs by running the existing `diffIR`, `classifyBreakingChanges`, and `generateMigration` engines, capturing the from/to version numbers and tags alongside the diff, breaking, and migration reports.

## Reference

Public types: `SemanticVersion`, `IRVersionMeta`, `IRVersionIndex`, `SaveVersionOptions` (`tag?`, `autoTag?`, `label?`), `VerifyResult`, `ChangelogEntry`.

Functions: `createVersionIndex`, `addVersionToIndex`, `removeTagFromIndex`, `tagVersionInIndex`, `createVersionMeta`, `verifyIRIntegrity`, `parseSemverTag`, `formatSemver`, `autoIncrementSemver`, `resolveVersionRef`, `generateChangelog`.

CLI subcommands (under `manifest versions`, all accepting `--store <path>`): `list`, `show <version>`, `save [source] [--tag|--auto-tag|--label]`, `diff <from> <to> [--breaking|--sql]`, `changelog [from] [to]`, `tag <version> <tag>`, `rollback <version> [-o]`, `verify [version] [--all]`.

## Notes and limitations

<Note>
  The store module performs no filesystem access; persistence is handled by the CLI command layer.
</Note>

Semver tags must match `major.minor.patch` exactly — prerelease and build-metadata suffixes are not parsed. `autoIncrementSemver` resolves a major bump only from breaking changes and otherwise treats any change as a minor bump, so it does not distinguish deprecations from compatible additions in the resulting version number. Version references resolve against the in-index entries; an unknown reference returns `undefined`.

## IR compression and binary serialization

For large IR payloads, Manifest supports compressing IR JSON to a binary format that reduces storage size and transfer time. The compression is lossless — the decompressed IR is byte-identical to the original.

```bash
# Compress an IR file
manifest versions compress ir/app.ir.json --output app.ir.bin

# Decompress back to JSON
manifest versions decompress app.ir.bin --output ir/app.ir.json
```

### Programmatic API

```typescript
import { compressIR, decompressIR } from '@angriff36/manifest/compression';

// Compress
const compressed = compressIR(irJson);
// Buffer — typically 60-80% smaller than the JSON string

// Decompress
const original = decompressIR(compressed);
// Identical to the original IR JSON
```

<Note>
  IR compression uses standard deflate/inflate. It is deterministic — the same IR always produces
  the same compressed output. This makes it safe for content-addressed storage and caching.
</Note>

## IR entity relationship graph visualizer

The graph visualizer generates visual representations of the entity relationship structure from compiled IR. It produces Mermaid diagrams or DOT (Graphviz) format.

```bash
# Generate an ER diagram as Mermaid
manifest diagram ir/app.ir.json --type er --output diagrams/

# Generate as DOT format
manifest diagram ir/app.ir.json --type er --format dot --output diagrams/
```

The visualizer produces:

| Diagram type | Content                                                        |
| ------------ | -------------------------------------------------------------- |
| `er`         | Entity boxes with properties, relationships as labeled edges   |
| `state`      | State transition diagrams for entities with state fields       |
| `sequence`   | Command execution sequences showing guard → action → emit flow |

The graph is derived entirely from the IR — no source file access is needed.

## Automated changelog generation from IR diffs

The `generateChangelog` function (exposed via the CLI as `manifest changelog`) builds a structured changelog entry between two IR versions by running the diff engine, breaking-change classifier, and migration generator:

```bash
# Generate a Markdown changelog between two version tags
manifest versions changelog 1.0.0 2.0.0 --output CHANGELOG.md

# Generate between two IR files
manifest changelog --old-ir v1.ir.json --new-ir v2.ir.json --output CHANGELOG.md
```

The changelog includes:

1. **Version header** — from/to version numbers and tags
2. **Breaking changes** — classified by element (entity, property, command) with impact
3. **Compatible additions** — new entities, new optional properties, new commands
4. **Deprecations** — changed computed expressions, loosened constraints
5. **Migration steps** — SQL and Prisma migration actions

<Note>
  The changelog is generated from IR structural comparison, not from Git history. This means it
  captures semantic changes that a text diff would miss (e.g., renaming a property that produces
  identical JSON structure but different semantics).
</Note>

The federation layer enables multiple Manifest runtime instances -- typically running as separate microservices -- to discover each other's entity schemas, invoke commands across service boundaries, and enforce policy-based authorization end-to-end. Federation is transport-agnostic: the default implementation uses HTTP with `fetch`, but the `FederationTransport` interface supports gRPC, message queues, or any custom protocol.

## Service descriptors

Each runtime instance publishes a `ServiceDescriptor` that declares which entities and commands it exposes to the federation mesh. Descriptors are built from compiled IR by the `buildDescriptor` function.

```typescript
import { buildDescriptor } from '@angriff36/manifest/federation';
import ir from './compiled.ir.json';

// buildDescriptor(serviceId, ir, options)
const descriptor = buildDescriptor('inventory-service', ir, {
  endpoint: 'https://inventory.internal',
  // exposeEntities: ['InventoryItem'],  // omit to expose all entities with commands
});
```

### ServiceDescriptor schema

| Field           | Type                 | Description                                          |
| --------------- | -------------------- | ---------------------------------------------------- |
| `serviceId`     | `string`             | Unique identifier for this service in the mesh       |
| `displayName`   | `string?`            | Human-readable name                                  |
| `endpoint`      | `string`             | Base URL for reaching this service                   |
| `schemaVersion` | `string`             | IR schema version this service speaks                |
| `entities`      | `ExposedEntity[]`    | Entities this service exposes, with their commands   |
| `health`        | `ServiceHealth?`     | Health status, populated by registry health checks   |
| `auth`          | `ServiceAuthConfig?` | Service-to-service auth scheme for the policy bridge |

The `buildDescriptor` function is pure: identical IR input always produces the same descriptor output, supporting deterministic builds and reproducible deployments.

## Service registry

The `FederationRegistry` is a central discovery layer. Services register their descriptors with the registry, which then provides command lookup, health tracking, and reachable-service filtering.

```typescript
import { FederationRegistry } from '@angriff36/manifest/federation';

const registry = new FederationRegistry();

// Register each service
registry.register(inventoryDescriptor);
registry.register(orderDescriptor);

// Look up which service handles a given command
const target = registry.findCommand('InventoryItem', 'reserve');
// => { service: ServiceDescriptor, command: { name: 'reserve', idempotent, requiredPolicies } }
```

### Health checks

When configured with `healthCheckIntervalMs`, the registry periodically probes each service's health endpoint. Services that fail their health check are marked unreachable and excluded from command lookup until they recover. Health checks use a separate timeout to avoid blocking federation calls.

## Federation client

The `FederationClient` invokes commands on remote services. It resolves the target service via the registry, constructs a `FederationRequest`, and dispatches it through the configured transport.

```typescript
import { FederationClient } from '@angriff36/manifest/federation';

// new FederationClient(registry, options?, transport?)
// The transport defaults to HttpFederationTransport (globalThis.fetch).
const client = new FederationClient(registry, {
  defaultTimeoutMs: 5000, // per-call timeout (default 30000)
  maxRetries: 2, // retry transient failures on idempotent commands (default 0)
  retryDelayMs: 1000, // delay between retries
});

const result = await client.invoke({
  serviceId: 'inventory-service',
  entity: 'InventoryItem',
  command: 'reserve',
  input: { instanceId: 'item-sku-123', quantity: 5 },
  bridge: { actorId: 'user-1', tenantId: 'tenant-1' },
});
```

### Idempotency heuristics

The client uses command-level idempotency heuristics to decide whether a failed request is safe to retry:

| Command contains only       | Idempotent?                 |
| --------------------------- | --------------------------- |
| `mutate`, `persist`         | Yes -- safe to retry        |
| `emit`, `effect`, `publish` | No -- may have side effects |

Commands with `emit`, `effect`, or `publish` actions are conservatively marked non-idempotent. Only commands whose actions are limited to `mutate` and `persist` are retried on transient failure.

### Transport interface

The default transport uses `globalThis.fetch`. Replace it for testing or alternative protocols by implementing the `FederationTransport` interface:

```typescript
interface FederationTransport {
  invoke(
    descriptor: ServiceDescriptor,
    request: FederationRequest,
    options: { timeoutMs: number; authToken?: string },
  ): Promise<FederationResponse>;
}
```

## Policy bridge

The policy bridge propagates caller identity across service boundaries so that policy expressions on the receiving service can evaluate correctly. Identity is encoded as HTTP headers with the `X-Manifest-*` prefix:

| Header              | Source                         | Reconstructed as        |
| ------------------- | ------------------------------ | ----------------------- |
| `X-Manifest-Actor`  | `context.actorId` or `user.id` | `context.actorId`       |
| `X-Manifest-Tenant` | `context.tenantId`             | `context.tenantId`      |
| `X-Manifest-Org`    | `context.orgId`                | `context.orgId`         |
| `X-Manifest-Roles`  | `user.role`                    | `user.role`             |
| `X-Request-Id`      | `context.requestId`            | `context.requestId`     |
| `X-Correlation-Id`  | Generated per invocation       | `context.correlationId` |

The sending side injects these headers into the outgoing request. The receiving side extracts them and reconstructs a `RuntimeContext` so that policy expressions like `user.role == "admin"` evaluate correctly on the downstream service.

```typescript
import {
  buildBridgeFromContext,
  buildBridgeHeaders,
  parseBridgeHeaders,
  contextFromBridgeHeaders,
} from '@angriff36/manifest/federation';

// Sending side: RuntimeContext -> bridge -> X-Manifest-* HTTP headers
const bridge = buildBridgeFromContext(runtimeContext);
const headers = buildBridgeHeaders(bridge);

// Receiving side (in your API route handler): HTTP headers -> bridge -> RuntimeContext
const reconstructed = contextFromBridgeHeaders(parseBridgeHeaders(request.headers));
```

<Warning>
  The policy bridge carries identity but not authorization decisions. Each service evaluates its own
  policies against the reconstructed context. A policy that denies access on service B will deny the
  federated call even if service A authorized it.
</Warning>

## HTTP adapter generation

The federation module includes a code generator that produces typed HTTP client classes for calling remote services. Each exposed command becomes a typed method on the generated client.

```typescript
import { generateHttpAdapter } from '@angriff36/manifest/federation';

const adapter = generateHttpAdapter(inventoryDescriptor);
const client = adapter.create('https://inventory.example.com', {
  authToken: process.env.INVENTORY_API_KEY,
  timeoutMs: 5000,
});

// client now has typed methods for each exposed command
await client.checkAvailability({ instanceId: 'sku-123', quantity: 10 });
```

The generated adapter includes:

- A `create(baseUrl, options?)` method that returns a typed client
- One async method per exposed command on the client
- Typed request and response objects derived from the IR
- Automatic `X-Manifest-*` header injection
- Configurable base URL, auth token, and timeout

## Example: Order service calling Inventory service

A common pattern is an order service that checks inventory availability before confirming an order:

```typescript
import {
  FederationRegistry,
  FederationClient,
  buildDescriptor,
} from '@angriff36/manifest/federation';
import { orderIr } from './ir/order.ir.json';
import { inventoryDescriptor } from './inventory-descriptor';

// 1. Set up the registry
const registry = new FederationRegistry();
registry.register(inventoryDescriptor);

// 2. Create the federation client
const federation = new FederationClient(registry, {
  defaultTimeoutMs: 3000,
  maxRetries: 3,
  retryDelayMs: 1000,
});

// 3. In the order service's "place" command middleware or action,
//    call the inventory service to reserve stock
async function placeOrder(orderId: string, sku: string, qty: number) {
  // Check inventory availability
  const check = await federation.invoke({
    serviceId: 'inventory-service',
    entity: 'InventoryItem',
    command: 'checkAvailability',
    input: { instanceId: sku, quantity: qty },
    bridge: { actorId: 'order-service' },
  });

  if (!check.success) {
    return { success: false, error: 'INSUFFICIENT_INVENTORY' };
  }

  // Reserve stock
  const reserve = await federation.invoke({
    serviceId: 'inventory-service',
    entity: 'InventoryItem',
    command: 'reserve',
    input: { instanceId: sku, quantity: qty },
    bridge: { actorId: 'order-service' },
  });

  if (!reserve.success) {
    return { success: false, error: 'RESERVATION_FAILED' };
  }

  // Proceed with order creation on the local runtime
  return localRuntime.runCommand(
    'create',
    { sku, quantity: qty },
    { entityName: 'Order', instanceId: orderId },
  );
}
```

## Importing federation

The federation module is available as a subpath export:

```typescript
// All federation types and functions
import {
  FederationRegistry,
  FederationClient,
  HttpFederationTransport,
  buildDescriptor,
  generateHttpAdapter,
  buildBridgeFromContext,
  contextFromBridgeHeaders,
} from '@angriff36/manifest/federation';
```

<Note>
  The federation module is transport-agnostic by design. The default HTTP transport works out of the
  box with any Manifest service exposed via the generated Next.js or Express routes. For
  message-queue or gRPC-based deployments, implement the `FederationTransport` interface.
</Note>

Manifest ships with three interactive developer tools for inspecting, debugging, and profiling runtime execution. All three are designed for development and testing — they are not intended for production use.

## Runtime REPL

The interactive REPL (`manifest repl`) provides a command-line interface for loading IR, creating instances, running commands, and inspecting state — all without writing test code.

### Starting the REPL

```bash
# Start with a compiled IR
manifest repl ir/app.ir.json

# Start with a source file (compiled on the fly)
manifest repl src/app.manifest
```

### Available commands

~~`load`, `create`, `run Entity.command`, `reset`~~

> **Correction (2026-07-15) @RYANSIGNED:** Verified against `packages/cli/src/commands/repl.ts`.
> The REPL loads source at startup (`manifest repl <path>`); there is no `load` / `create` /
> `reset`. `run` takes **two** positional args (`run <entity> <command> [json]`), not
> `Entity.command` dotted form. `eval` is stubbed (“not yet implemented”).

| Command | Description |
| ------- | ----------- |
| `help` | Show available commands |
| `list` | List entities (or instances when scoped) |
| `inspect` / `show` / `info` | Inspect IR / entity / runtime info |
| `get <Entity> <id>` | Retrieve an instance |
| `run <entity> <command> [json]` | Execute a command via `runCommand` |
| `cmds` | List commands |
| `eval <expression>` | Stub — expression eval not implemented yet |
| `set` | Set REPL context (e.g. tenant) |
| `policies` | Show policies |
| `events` | Show `runtime.getEventLog()` |
| `clear` | Clear console / event log helpers |
| `reload` | Recompile and reload the startup manifest |
| `json` | Toggle JSON output mode |
| `exit` | Exit the REPL |

### Usage example

```
$ manifest repl src/app.manifest
manifest> run Order addItem {"amount": 29.99}
✓ Command Order.addItem succeeded

manifest> get Order order-1
{ id: "order-1", status: "pending", total: 29.99 }

manifest> events
Event Log (1 events):
  OrderUpdated …
```

<Note>
  The REPL uses an in-memory store by default. State is lost when you exit. For persistent
  debugging, consider the snapshot testing infrastructure instead.
</Note>

## Time-travel debugger

The time-travel debugger records every state mutation during command execution and lets you step forward and backward through the execution timeline. It wraps `RuntimeEngine` and captures a snapshot before and after each action.

### Enabling time-travel

~~```typescript
import { TimeTravelDebugger } from '@angriff36/manifest/debug';
const debugger_ = new TimeTravelDebugger(runtime);
const result = await debugger_.runCommand('Order', 'complete', {
  instanceId: 'order-1',
});
```~~

> **Correction (2026-07-15) @RYANSIGNED:** Shipped debug API is `CommandTraceRecorder` from
> `@angriff36/manifest/debug`. Its `runCommand` matches
> `RuntimeEngine.runCommand(commandName, input, options?)` — not entity-first.
> There is no exported `TimeTravelDebugger` class.

```typescript
import { RuntimeEngine } from '@angriff36/manifest';
import { CommandTraceRecorder } from '@angriff36/manifest/debug';

const runtime = new RuntimeEngine(ir, {}, { /* runtime options */ });
const tracer = new CommandTraceRecorder(runtime);

const result = await tracer.runCommand('complete', {}, {
  entityName: 'Order',
  instanceId: 'order-1',
});

const steps = tracer.getSteps();
// [{ index: 0, label: 'before', snapshot: { ... } }, { index: 1, label: 'after', ... }]
```

### Stepping through execution

~~```typescript
const stateAtStep2 = debugger_.getStateAt(2);
debugger_.stepForward();
debugger_.stepBack();
const diff = debugger_.diff(0, 3);
```~~

> **Correction (2026-07-15) @RYANSIGNED:** `CommandTraceRecorder` exposes `getSteps()` and
> `restoreToStep(stepIndex)` (via `RuntimeEngine.restore`). There is no `getStateAt` /
> `stepForward` / `stepBack` / `diff` API.

```typescript
const steps = tracer.getSteps();
await tracer.restoreToStep(0); // restore pre-command snapshot
```

### Timeline entries

~~Each timeline entry contains `step`, `action`, `state`, `timestamp`.~~

> **Correction (2026-07-15) @RYANSIGNED:** Each `CommandTraceStep` has `index`, `label`, and
> `snapshot` (`SerializedRuntimeSnapshot`).

<Warning>
  Trace snapshots capture store state. For large datasets or high-frequency command execution,
  this can consume significant memory. Use it in development and testing only.
</Warning>

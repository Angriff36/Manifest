Snapshot tests compile a representative Manifest program to IR, run every built-in projection against it, and snapshot the generated artifacts. Any change to generated code surfaces as a snapshot diff in review. Verified against `src/manifest/projections/snapshot.test.ts` and `src/manifest/projections/__snapshots__/snapshot.test.ts.snap`.

## Usage

The tests run as part of the normal vitest suite. To update snapshots after an intentional change to a generator:

```bash
npx vitest -u src/manifest/projections/snapshot.test.ts
```

## How it works

The test file builds a fixed IR fixture in `snapshotIR()` — a small but representative program with two entities (`Task`, `User`), a computed property, relationships (`belongsTo`/`hasMany`), commands with guards and actions, events with channels, a policy, constraints, and a `durable` store for both entities so the ORM projections (Prisma, Drizzle) emit output. The IR provenance is hardcoded (`contentHash: 'snapshot-fixture-hash'`, fixed `compiledAt`) so the input is deterministic.

The suite enumerates projections via `listBuiltinProjections()` and asserts there are exactly **29** built-in projections.

`snapshot.test.ts` uses
 **29**) is a deliberate tripwire: it fails when the built-in projection set changes, forcing a conscious snapshot update. Only ISO-millisecond timestamps are stabilized; any other non-deterministic output in a generator would make snapshots flaky and must be handled in the generator itself, since the runtime determinism assertion would also catch it.

## Command and guard coverage reporter

The coverage reporter correlates compiled IR with conformance and unit test evidence to produce a coverage report. It answers the question: "Which commands, guards, policies, and constraints have test coverage?"

### Usage

```bash
# Report coverage for a compiled IR
manifest coverage --ir ir/app.ir.json

# Fail CI if coverage drops below a threshold
manifest coverage --ir ir/app.ir.json --min-coverage 80 --strict

# JSON output for CI integration
manifest coverage --ir ir/app.ir.json --format json
```

### Coverage categories

| Category    | What it checks                                            |
| ----------- | --------------------------------------------------------- |
| Commands    | Each declared command has at least one test case          |
| Guards      | Each guard expression is exercised (true and false paths) |
| Policies    | Each policy rule has a test that evaluates it             |
| Constraints | Each constraint expression is tested for pass and fail    |

### Programmatic API

`@angriff36/manifest/coverage` exports
`coverageCommand` (and coverage result types), **not** a `CoverageReporter` class.

```typescript
import { coverageCommand } from '@angriff36/manifest/coverage';

const report = await coverageCommand({
  ir: 'ir/app.ir.json',
  root: '.',
  format: 'json',
  minCoverage: 80,
  strict: true,
});
console.log(report.overall.percentage);
```

## Property-based testing

In-repo property tests use **fast-check** in
`src/manifest/runtime-guard-properties.test.ts` (guard evaluation properties). There is no
dedicated `property-based` vitest grep title and no documented `FC_SEED` env contract on this
page. Run the file via the normal suite (`pnpm test`) or target that test file explicitly.

Invariants covered there include guard failure indexing / short-circuit behavior under random
inputs — not a full “random IR program” generator as previously implied.

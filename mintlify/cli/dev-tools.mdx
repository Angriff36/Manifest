Manifest ships a set of developer tools that keep source files consistent, rebuild generated code on every save, and provide rich editing support in your IDE.

## `manifest fmt`

`manifest fmt` normalizes `.manifest` source files using deterministic whitespace rules. It converts CRLF to LF, tabs to two spaces, strips trailing whitespace and trailing blank lines, and ensures a final newline.

```bash
# Format files in place
manifest fmt src/

# Check formatting without writing (exit 1 if any file needs changes)
manifest fmt src/ --check

# Format with a custom glob
manifest fmt --glob "modules/**/*.manifest"
```

### Options

<ParamField path="--check" type="boolean" default="false">
  CI mode. Report files that would change without writing them. Exits non-zero when any file needs
  formatting.
</ParamField>

<ParamField path="--write" type="boolean">
  Write formatted output in place. This is the default when `--check` is not set.
</ParamField>

<ParamField path="-g, --glob" type="string">
  Custom glob pattern when `source` is a directory.
</ParamField>

### What the formatter does

The formatter applies the following transformations deterministically:

- CRLF line endings are normalized to LF
- Tab characters are replaced with two spaces
- Trailing whitespace on each line is removed
- Trailing blank lines at the end of the file are removed
- The file ends with exactly one newline character
- Empty files produce a single newline

The formatter is intentionally minimal. It normalizes whitespace only. It does not reorder properties, normalize brace style, or restructure code.

### Idempotency

Running `manifest fmt` on an already-formatted file produces no diff. This makes it safe to run in a pre-commit hook or CI pipeline without introducing spurious changes.

### CI integration

Use `--check` in CI to enforce formatting standards without modifying files:

```bash
# In your CI pipeline
manifest fmt src/ --check
```

The command exits non-zero when any file would change, giving you an actionable signal.

## `manifest watch`

`manifest watch` monitors `.manifest` files for changes and triggers an incremental compile-and-generate cycle on every save. It uses Node.js `fs.watch` with recursive mode for efficient file system monitoring (with `watchFile` polling fallback for older platforms).

```bash
# Watch a directory and rebuild on change
manifest watch src/

# Watch with JSON change events for downstream tooling
manifest watch src/ --events

# Watch with a custom debounce
manifest watch src/ --debounce 500
```

### Options

`watch` accepts all the same flags as [`manifest build`](/cli/commands) (`-p`, `-s`, `--ir-output`, `--code-output`, `-g`, and the import-path flags) plus:

<ParamField path="--debounce" type="number" default="300">
  Debounce delay in milliseconds. Rapid file changes within this window are coalesced into a single
  rebuild.
</ParamField>

<ParamField path="--events" type="boolean" default="false">
  Emit structured JSON change events to stdout. Useful for piping into other build tools.
</ParamField>

<ParamField path="--clear" type="boolean" default="false">
  Clear the terminal on each rebuild.
</ParamField>

### Structured events

When `--events` is set, `manifest watch` prints one JSON object per line to stdout:

```json
{"type":"ready","timestamp":1717312000000,"files":12,"irOutput":"ir/","codeOutput":"generated/"}
{"type":"change","timestamp":1717312010000,"file":"src/orders.manifest"}
{"type":"build:start","timestamp":1717312010100}
{"type":"build:success","timestamp":1717312010800,"filesCompiled":1,"filesGenerated":3}
{"type":"build:error","timestamp":1717312010800,"error":"Parse error at line 42: unexpected token"}
```

These events are machine-readable, making it straightforward to pipe the output into other tools or build orchestrators.

### Graceful shutdown

The watcher traps `SIGINT` and `SIGTERM` and shuts down cleanly. Build errors do not terminate the watcher; the process continues monitoring and will rebuild on the next change.

## VS Code extension

The Manifest VS Code extension provides syntax highlighting, autocomplete, hover documentation, go-to-definition, and real-time diagnostics. It is available as a bundled extension with an embedded LSP server -- no separate install needed.

### Installation

Build and install the extension from source:

```bash
# Build the root library first
pnpm run build:lib

# Build and package the extension
cd packages/vscode-extension
pnpm run build
pnpm run package
```

This produces a `.vsix` file you can install via the VS Code Extensions view ("Install from VSIX...") or via the command line:

```bash
code --install-extension manifest-lang-*.vsix
```

For development, use the Extension Development Host:

```bash
code --extensionDevelopmentPath=packages/vscode-extension
```

### Features

**Syntax highlighting.** A TextMate grammar covers all Manifest keywords: entity declarations, property modifiers, control flow constructs, type names, operators, constants, relationship keywords, access control terms, store targets, and severity levels.

**Code snippets.** Fifteen snippets for common patterns: `entity`, `property`, `command`, `policy`, `constraint`, `computed`, `hasMany`, `belongsTo`, `store`, `event`, `module`, `enum`, `guard`, `mutate`, and `emit`.

**Language configuration.** Comment toggling (`//` and `/* */`), bracket matching, auto-closing pairs, and indentation rules.

**Diagnostics.** Real-time error and warning squiggles from the Manifest compiler, debounced at 200ms to avoid noisy feedback during typing. Positions are converted from the Manifest compiler's 1-based coordinates to the LSP protocol's 0-based coordinates.

**Document symbols.** The Outline view shows entities (with nested properties, commands, relationships, policies, and constraints), modules, enums, stores, and events.

**Hover.** Formatted markdown information for entities (property and command counts), properties (type and modifiers), commands (parameters and guard count), computed properties, policies, constraints, enums, and keywords.

**Autocomplete.** All keywords, type names, entity and enum names from the parsed AST, and context variables (`self`, `user`, `context`).

**Go-to-definition.** Intra-file resolution for entity, enum, and module references.

## Language Server Protocol

The Manifest LSP server provides IDE-agnostic language support. It lives in `packages/lsp-server/` and reuses the existing Manifest compiler pipeline (Lexer, Parser, IR Compiler) to produce the artifacts that each LSP feature consumes.

### Capabilities

| Feature              | Description                                                                       |
| -------------------- | --------------------------------------------------------------------------------- |
| **Diagnostics**      | Compilation errors and warnings with 200ms publish debounce                       |
| **Completion**       | Context-aware keyword, type, and symbol completion using brace-depth tracking     |
| **Hover**            | Markdown-formatted documentation for entities, properties, commands, and keywords |
| **Go-to-definition** | Jump to entity, enum, and module declarations                                     |
| **Document symbols** | Hierarchical outline of all declarations in a file                                |

### Architecture

The LSP server consists of these modules:

- **compiler-bridge.ts** -- Orchestrates the full compilation pipeline and returns tokens, AST, and IR for feature consumption
- **position-utils.ts** -- Handles the Manifest lexer's end-position convention (token position is one column past the last character) with conversion utilities between 1-based and 0-based coordinates
- **document-store.ts** -- Manages open document state and invalidation
- **Features** -- Each feature (diagnostics, completion, hover, definition, document-symbols) is a set of pure functions that take compiled artifacts and return LSP protocol objects

The server communicates over stdio and is launched by the editor client. Each feature is stateless: no shared mutable state exists between requests.

### Connecting a custom editor

The LSP server binary is `packages/lsp-server/bin/manifest-lsp.js`. Configure your editor to launch it as a language server over stdio:

```json
{
  "languageserver": {
    "manifest": {
      "command": "node",
      "args": ["path/to/packages/lsp-server/bin/manifest-lsp.js", "--stdio"],
      "filetypes": ["manifest"],
      "rootPatterns": ["manifest.config.yaml", "manifest.config.yml"]
    }
  }
}
```

<Tip>
  The LSP server bundles the Manifest compiler, so no separate compile step is needed. It reads
  `.manifest` files directly from disk and compiles them on every document change.
</Tip>

## More developer commands

Concise reference for the remaining inspection, debugging, and authoring commands. Run any with `--help` for the full flag list.

### `manifest check [source]`

Compile `.manifest` source and validate the generated IR in one step — a fast "does this compile and conform?" gate. Flags mirror `compile` plus `validate`: `-o, --output`, `-g, --glob`, `-d, --diagnostics`, `--pretty`, `--schema <path>`, `--strict` (fail on warnings).

```bash
manifest check modules/recipe.manifest --strict
```

### `manifest preflight`

Validate environment variables against the `env` mapping in `manifest.config.yaml`, or generate a `.env.example`. Consumes the structured `env` block documented in [Configuration](/cli/configuration#param-env).

<ParamField path="-f, --format" type="string" default="text">
  Output format: `text` or `json`.
</ParamField>
<ParamField path="--generate-example" type="boolean">
  Generate `.env.example` instead of checking the environment.
</ParamField>
<ParamField path="-o, --output" type="string" default=".env.example">
  Output path for the generated `.env.example`.
</ParamField>

```bash
manifest preflight
manifest preflight --generate-example
```

### `manifest install-hooks`

Install a Git pre-commit hook that runs `manifest fmt` / `manifest validate` (driven by the [`hooks`](/cli/configuration#param-hooks) config block).

<ParamField path="-f, --force" type="boolean">
  Overwrite an existing hook configuration.
</ParamField>
<ParamField path="--provider" type="string" default="husky">
  Hook provider: `husky` or `simple-git-hooks`.
</ParamField>

```bash
manifest install-hooks --provider husky
```

### `manifest generate-tests [source]` (alias `gen-tests`)

LLM-driven conformance fixture generation: analyzes existing fixtures and emits new `.manifest` sources plus expected IR covering edge cases. Generated fixtures are compiled and validated before being written.

Key flags: `--feature <description>`, `--category <edge-cases|boundary|adversarial|coverage>` (default `edge-cases`), `--count <n>` (default `3`), `--output <path>`, `--api-key <key>` (or `ANTHROPIC_API_KEY`), `--model <model>`, `--temperature <n>`, `--max-retries <n>`, `--dry-run`, `--verbose`.

```bash
manifest generate-tests src/ --category boundary --count 5 --dry-run
```

### `manifest mock <source>`

Start a local HTTP mock server from compiled IR (or a `.manifest` file), backed by the runtime engine with in-memory stores — lets frontend teams develop against a realistic API before the backend ships.

<ParamField path="-p, --port" type="string" default="4000">
  Port number.
</ParamField>
<ParamField path="--host" type="string" default="127.0.0.1">
  Bind host.
</ParamField>
<ParamField path="--cors" type="boolean" default="false">
  Enable CORS headers.
</ParamField>
<ParamField path="--scenario" type="string" default="default">
  Hint mode: `default`, `guard-fail`, or `constraint-fail`.
</ParamField>

```bash
manifest mock ir/app.ir.json --port 4000 --cors
```

### `manifest repl [source]`

Interactive runtime REPL for inspecting IR and exercising guards, policies, and entity state against the runtime.

<ParamField path="--json" type="boolean">
  Machine-readable JSON response mode.
</ParamField>
<ParamField path="--user" type="string">
  User id for runtime context.
</ParamField>
<ParamField path="--tenant" type="string">
  Tenant id for runtime context.
</ParamField>
<ParamField path="--context" type="string">
  Additional runtime context as JSON.
</ParamField>

```bash
manifest repl src/app.manifest --user u_123 --tenant t_1
```

### `manifest harness <manifest>`

Run a fixture-generator-style harness script against compiled IR and report failed steps/assertions.

<ParamField path="-s, --script" type="string" required>
  Path to the harness script JSON.
</ParamField>
<ParamField path="-f, --format" type="string" default="text">
  Output format: `text` or `json`.
</ParamField>

```bash
manifest harness src/app.manifest --script harness.json
```

### `manifest load-test [source]`

Generate k6 or Artillery load-test scripts from IR entities and commands, with realistic data, ramp-up profiles, and SLO thresholds.

Key flags: `-o, --output <path>` (default `load-tests`), `-f, --format <k6|artillery>` (default `k6`), `--base-url <url>`, `--ramp-up <stages>`, `--slo <thresholds>`, `--command <name...>`, `--entity <name...>`, `--profile`, `--timeout <ms>` (default `30000`), `--json`.

```bash
manifest load-test src/app.manifest --format k6 --base-url http://localhost:3000
```

### `manifest routes`

Compile all `.manifest` files and emit the canonical route manifest (the agent-accessible equivalent of the DevTools Route Surface tab).

<ParamField path="-s, --src" type="string">
  Source glob for `.manifest` files.
</ParamField>
<ParamField path="-f, --format" type="string" default="json">
  Output format: `json` or `summary`.
</ParamField>
<ParamField path="-b, --base-path" type="string" default="/api">
  Base path prefix for routes.
</ParamField>

```bash
manifest routes --format summary
```

### `manifest lint-routes`

Scan client directories for hardcoded route strings and fail when violations are found — the enforcement layer for canonical routes.

<ParamField path="-f, --format" type="string" default="text">
  Output format: `text` or `json`.
</ParamField>
<ParamField path="-c, --config" type="string">
  Config file path.
</ParamField>

```bash
manifest lint-routes
```

### `manifest runtime-check <entity> <command>`

Correlate route surface, source manifests, and precompiled IR for a single command to spot drift.

<ParamField path="--route" type="string">
  Optional canonical route path to validate (exact match).
</ParamField>
<ParamField path="--json" type="boolean">
  JSON output.
</ParamField>
<ParamField path="--src" type="string">
  Source manifest glob pattern.
</ParamField>
<ParamField path="--ir-root" type="path...">
  Compiled IR root directory/directories.
</ParamField>

```bash
manifest runtime-check Recipe publish --route /api/recipe/commands/publish
```

### `manifest inspect entity <entityName>`

Inspect a single entity across source manifests and precompiled IR.

<ParamField path="--json" type="boolean">
  JSON output.
</ParamField>
<ParamField path="--src" type="string">
  Source manifest glob pattern.
</ParamField>
<ParamField path="--ir-root" type="path...">
  Compiled IR root directory/directories.
</ParamField>

```bash
manifest inspect entity Recipe
```

Summarize duplicate merge reports (`*.merge-report.json`) produced by merge compilation.

<ParamField path="--entity" type="string">

List plugins declared in `manifest.config` (see the [`plugins`](/cli/configuration#param-plugins) config block). Pass `--json` for machine-readable output.

## `manifest versions`

IR snapshot versioning with semantic tags, changelog, and integrity verification. Snapshots live under `.manifest-versions/` (override with `--store <path>` on every subcommand).

| Subcommand                       | Purpose                                          | Key flags                                                |
| -------------------------------- | ------------------------------------------------ | -------------------------------------------------------- |
| `versions list`                  | List saved IR versions                           | `--store`, `--json`                                      |
| `versions show <version>`        | Show version metadata (number, tag, or `latest`) | `--store`, `--json`                                      |
| `versions save [source]`         | Compile and save a new IR snapshot               | `--tag <tag>`, `--auto-tag`, `--label <text>`, `--store` |
| `versions diff <from> <to>`      | Compare two saved versions                       | `--breaking`, `--sql`, `--json`, `--store`               |
| `versions changelog [from] [to]` | Generate a changelog between two versions        | `--store`, `--json`                                      |
| `versions tag <version> <tag>`   | Apply a semantic version tag to a saved version  | `--store`                                                |
| `versions rollback <version>`    | Output a previous IR snapshot                    | `-o, --output <path>`, `--store`                         |
| `versions verify [version]`      | Verify IR integrity via SHA-256 comparison       | `--all`, `--json`, `--store`                             |

```bash
# Save a tagged snapshot, then verify integrity
manifest versions save src/app.manifest --tag 1.0.0
manifest versions verify latest

# Diff two versions with breaking-change analysis
manifest versions diff 1.0.0 latest --breaking
```

## Typical workflow

A common development workflow combines all four tools:

1. **Format** your source before committing:

   ```bash
   manifest fmt src/ --check
   ```

2. **Watch** for changes during development:

   ```bash
   manifest watch src/ --events
   ```

3. **Edit** in VS Code with real-time diagnostics, completion, and hover.

4. **Commit** with confidence that formatting and generated code are in sync.

```bash
# Full workflow in a Makefile or package.json script
fmt: manifest fmt src/ --write
dev: manifest watch src/ --events
ci: manifest fmt src/ --check && manifest build
```

Manifest provides three testing tools for validating program behavior beyond unit and conformance tests: an interactive constraint test harness, an auto-generated mock server for integration testing, and a load testing fixture generator.

## Interactive constraint test harness

The constraint test harness provides an interactive interface for exploring constraint boundaries — you supply entity property values and immediately see which constraints pass, warn, or block. It is useful during development to understand how constraint expressions evaluate for different inputs without writing test code.

### Usage

```bash
# Launch the harness for a specific entity
manifest test constraints --entity Order --ir ir/app.ir.json

# Launch with an interactive REPL-style interface
manifest test constraints --entity Order --source src/app.manifest --interactive
```

### Interactive session

```
Constraint harness for entity: Order

Properties:
  status (string): pending
  total (number): 0
  customerStatus (string): active

Constraints:
  [ok]   validStatus: self.status == "pending" or self.status == "confirmed" or self.status == "shipped"
  [ok]   activeCustomer: self.customerStatus == "active"
  [warn] highValue: self.total > 10000 — "High-value order may require additional approval"
  [ok]   positiveTotal: self.total >= 0

> set total 15000
  [ok]   validStatus: ...
  [ok]   activeCustomer: ...
  [warn] highValue: self.total > 10000 — "High-value order may require additional approval"  ← WARN
  [ok]   positiveTotal: ...

> set customerStatus inactive
  [ok]   validStatus: ...
  [block] activeCustomer: self.customerStatus == "active"  ← BLOCK
  [warn] highValue: self.total > 10000 — ...
  [ok]   positiveTotal: ...
```

### Programmatic API

```typescript
import { ConstraintTestHarness } from '@angriff36/manifest/testing';

const harness = new ConstraintTestHarness(ir, 'Order');

// Set property values
harness.setProperty('total', 15000);
harness.setProperty('customerStatus', 'active');

// Evaluate all constraints
const results = harness.evaluate();
// [
//   { code: 'validStatus', severity: 'ok', expression: '...', passed: true },
//   { code: 'activeCustomer', severity: 'ok', expression: '...', passed: true },
//   { code: 'highValue', severity: 'warn', expression: '...', passed: false, message: '...' },
//   { code: 'positiveTotal', severity: 'ok', expression: '...', passed: true },
// ]

// Test override authorization
const overrideResult = harness.testOverride('highValue', {
  authorizedBy: 'user-999',
  reason: 'Emergency processing',
});
```

## Auto-generated mock server

The mock server generates a lightweight HTTP server that mimics the Manifest runtime API without requiring a real runtime or database. It is useful for:

- **Frontend development**: Test UI against realistic API responses without a backend
- **Integration testing**: Verify consumer code handles all response shapes correctly
- **Contract testing**: Ensure downstream services match the Manifest API contract

### Usage

```bash
# Generate and start a mock server
manifest mock ir/app.ir.json --port 3456

# Generate mock handlers as code (for embedding in test suites)
manifest mock ir/app.ir.json --output mocks/ --format ts
```

### Mock server behavior

| Endpoint                       | Behavior                                                  |
| ------------------------------ | --------------------------------------------------------- |
| `GET /api/{entity}`            | Returns a paginated list of mock instances                |
| `GET /api/{entity}/{id}`       | Returns a single mock instance                            |
| `POST /api/{entity}/{command}` | Validates input, returns a `CommandResult` with mock data |
| `GET /api/events`              | Returns the mock event log                                |

Mock instances are generated from the entity's property types with realistic defaults. Command execution follows the declared guard and policy rules — a guard denial returns the appropriate error response.

### Options

| Option     | Default | Description                                                  |
| ---------- | ------- | ------------------------------------------------------------ |
| `--port`   | `3456`  | Port to listen on                                            |
| `--output` | —       | Output directory (generates code instead of starting server) |
| `--format` | `ts`    | Output format: `ts` (Hono handlers) or `json` (OpenAPI mock) |
| `--seed`   | `42`    | Seed for deterministic mock data generation                  |

<Note>
  The mock server does not execute the real runtime engine. It validates input structure and
  guard/policy conditions, then returns pre-built mock responses. For full runtime behavior, use the
  embedded runtime in your test suite.
</Note>

## Load testing fixture generator

The load testing fixture generator creates realistic test data sets from IR entity definitions. It produces JSON files containing hundreds or thousands of entity instances with property values that conform to the declared types, constraints, and relationships.

### Usage

```bash
# Generate 1000 instances per entity
manifest generate-fixtures ir/app.ir.json --count 1000 --output fixtures/

# Generate with relationship linking (preserves hasMany/belongsTo)
manifest generate-fixtures ir/app.ir.json --count 500 --link-relationships --output fixtures/

# Generate for a specific entity only
manifest generate-fixtures ir/app.ir.json --entity Order --count 2000 --output fixtures/
```

### Generated fixture format

```json
{
  "entities": {
    "Order": [
      { "id": "order-001", "status": "pending", "total": 45.99, "customerName": "Customer 1" },
      { "id": "order-002", "status": "confirmed", "total": 129.5, "customerName": "Customer 2" }
    ],
    "LineItem": [
      { "id": "li-001", "orderId": "order-001", "price": 25.99, "quantity": 1 },
      { "id": "li-002", "orderId": "order-001", "price": 10.0, "quantity": 2 }
    ]
  },
  "metadata": {
    "generatedAt": "2026-06-02T00:00:00.000Z",
    "seed": 42,
    "entityCount": { "Order": 2, "LineItem": 2 }
  }
}
```

### Options

| Option                 | Default     | Description                                   |
| ---------------------- | ----------- | --------------------------------------------- |
| `--count`              | `100`       | Number of instances per entity                |
| `--entity`             | all         | Generate for a specific entity only           |
| `--seed`               | `42`        | Random seed for deterministic generation      |
| `--link-relationships` | `false`     | Generate relationship-consistent foreign keys |
| `--output`             | `fixtures/` | Output directory                              |
| `--format`             | `json`      | Output format: `json` or `csv`                |

<Note>
  When `--link-relationships` is enabled, the generator ensures that `belongsTo` and `ref`
  properties reference valid instances of the target entity, and `hasMany` relationships have
  consistent back-references. This produces more realistic data for testing joins and cross-entity
  queries.
</Note>

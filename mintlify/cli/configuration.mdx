`manifest.config.yaml` is the project-level config file that the CLI loads from your project root before executing any command. You create it by running `manifest init` or by writing it by hand. Every field is optional — the CLI falls back to built-in defaults for any field you omit.

## Config file locations

The CLI searches for a config file in the following order, using the first one it finds:

1. `manifest.config.yaml`
2. `manifest.config.yml`
3. `.manifestrc.yaml`
4. `.manifestrc.yml`

All four files live at your project root (the directory where you run `manifest` commands).

Runtime bindings that contain JavaScript values—store classes, store instances,
factories, and `resolveUser`—belong in `manifest.config.ts` or
`manifest.config.js`. A TypeScript config may also contain a `build` block; its
build values override YAML.

## Durable storage for generated runtime companions

`DATABASE_URL` only supplies connection details. It does not tell Manifest
which entity uses which adapter or table. Put those bindings in
`manifest.config.ts`:

```ts
import { defineConfig } from '@angriff36/manifest/config';
import { PostgresStore } from '@angriff36/manifest/stores';

const connectionString = process.env.DATABASE_URL;
if (!connectionString) throw new Error('DATABASE_URL is required');

export default defineConfig({
  stores: {
    Customer: { implementation: new PostgresStore({ connectionString, tableName: 'customers' }) },
    Invoice: { implementation: new PostgresStore({ connectionString, tableName: 'invoices' }) },
  },
});
```

Then point the Next.js runtime companion at that file:

```yaml
projections:
  nextjs:
    output: generated
    options:
      # Relative to the emitted lib/manifest-runtime.ts file.
      runtimeConfigImport: '../manifest.config'
```

Only `memory` and browser `localStorage` work without this binding. Generated
runtimes containing `postgres`, `supabase`, `durable`, or custom stores fail
clearly when `runtimeConfigImport` is absent. See [Stores](/language/stores#complete-generated-companion-postgresql-setup) for the full source-to-runtime example.

## Option priority

When resolving any value, the CLI checks three sources in this order:

1. **CLI flag** (highest priority) — value passed directly on the command line
2. **Config file** — value from `manifest.config.yaml`
3. **Built-in default** (lowest priority) — fallback hardcoded in the CLI

You can always override a config value for a single run without modifying the file.

## Top-level fields

<ParamField path="$schema" type="string">
  Optional pointer to the config JSON Schema for editor IntelliSense. Manifest does **not** publish a resolvable schema URL, and `manifest config validate` loads the schema bundled with the package — it never fetches a URL. For autocomplete without a dead link, map the bundled schema in `.vscode/settings.json`:

```jsonc
{
  "yaml.schemas": {
    "./node_modules/@angriff36/manifest/docs/spec/config/manifest.config.schema.json": [
      "manifest.config.yaml",
      ".manifestrc.yaml",
    ],
  },
}
```

If you do set `$schema`, point it at a real local path — never a public URL Manifest doesn't host.

</ParamField>

<ParamField path="src" type="string" default="**/*.manifest">
  Glob pattern that selects your `.manifest` source files. Used by `manifest compile` and `manifest build` when no `source` argument is passed on the command line.

```yaml
src: modules/**/*.manifest
```

</ParamField>

<ParamField path="output" type="string" default="ir/">
  Directory where compiled IR JSON files are written. Used by `manifest compile` as the default `--output` value, and by `manifest build` as the default `--ir-output` value.

```yaml
output: ir/
```

</ParamField>

<ParamField path="naming" type="string | object">
  Global identifier-casing convention inherited by projections that map IR names to physical database names (currently the Prisma projection). Opt-in; omit to emit IR names verbatim. A per-projection `projections.<name>.options.naming` overrides this default.

The string shorthand `snake_case` expands to `{ table: snake_case, column: snake_case, pluralizeTables: true }` — e.g. `createdAt` → `@map("created_at")`, `Widget` → `@@map("widgets")`. The object form tunes each axis (`table`: `snake_case` | `camelCase` | `PascalCase` | `preserve`; `column`: `snake_case` | `camelCase` | `preserve`; `pluralizeTables`: boolean, default `true`).

```yaml
naming: snake_case
# or:
naming:
  table: snake_case
  column: snake_case
  pluralizeTables: true
```

Only `@map`/`@@map` attributes are added — Prisma model names and field identifiers stay as the IR name, so relations and indexes are unaffected. Explicit `tableMappings`/`columnMappings` always win and are the escape hatch for irregular plurals.

</ParamField>

<ParamField path="prismaSchema" type="string">
  Path to your Prisma schema file, used by property alignment checks. When omitted, the CLI searches `prisma/schema.prisma`, `schema.prisma`, and `db/schema.prisma` in that order.

```yaml
prismaSchema: prisma/schema.prisma
```

</ParamField>

<ParamField path="hooks" type="object">
  Hook settings: Git pre-commit settings consumed by `manifest install-hooks`, plus optional build lifecycle scripts.

<ParamField path="hooks.skipInCi" type="boolean" default="true">
  Skip running the generated pre-commit hook in CI environments.
</ParamField>
<ParamField path="hooks.provider" type="string" default="husky">
  Git hook manager the pre-commit hook is installed into. Accepted values: `husky`,
  `simple-git-hooks`.
</ParamField>
<ParamField path="hooks.runFmt" type="boolean" default="true">
  Run `manifest fmt` from the generated pre-commit hook.
</ParamField>
<ParamField path="hooks.runValidate" type="boolean" default="true">
  Run `manifest validate` from the generated pre-commit hook.
</ParamField>

<ParamField path="hooks.lifecycle" type="object">
  Build lifecycle scripts (not git hooks). These are shell scripts or Node scripts run at specific points in the build process. Paths are relative to the project root.
</ParamField>

<ParamField path="hooks.lifecycle.beforeCompile" type="array">
  Scripts (paths) run before `manifest compile`. Executes in the order listed. Fails the build if any script exits non-zero.
</ParamField>

<ParamField path="hooks.lifecycle.afterGenerate" type="array">
  Scripts (paths) run after a successful `manifest generate` or `manifest generate --all`. Executes in the order listed. Fails the build if any script exits non-zero.
</ParamField>

```yaml
hooks:
  # Git pre-commit hooks
  provider: husky
  skipInCi: true
  runFmt: true
  runValidate: true
  # Build lifecycle scripts
  lifecycle:
    beforeCompile:
      - scripts/pre-compile.sh
    afterGenerate:
      - scripts/post-generate.js
```

</ParamField>

<ParamField path="plugins" type="array">
  Third-party plugin declarations. At CLI startup any declared plugins are loaded and the CLI commands they contribute are registered, so they appear in `manifest --help` and are dispatchable; `manifest plugins list` reports their real load status and diagnostics. Loading is conditional — a project with no `plugins` declared loads nothing. A plugin that fails to load surfaces a diagnostic and is skipped; it never bricks the CLI. For runtime use (custom store adapters, expression builtins, audit sinks), compose the loaded registries into your runtime options with `pluginRegistriesToRuntimeOptions` from `@angriff36/manifest/plugin-loader` — see [Plugin API](/extensibility/plugin-api). Each entry requires a `module`; `enabled`, `options`, `order`, and `capabilities` are optional.
</ParamField>

<ParamField path="plugins[].module" type="string" required>
  npm package name or relative file path to the plugin module.
</ParamField>
<ParamField path="plugins[].options" type="object">
  Plugin-specific options forwarded to the plugin's `onLoad` hook and to its store-adapter and
  audit-sink factories.
</ParamField>
<ParamField path="plugins[].enabled" type="boolean" default="true">
  Whether the plugin is active.
</ParamField>
<ParamField path="plugins[].order" type="integer">
  Load order for this plugin. Lower numbers load first. Ties break by `module` string (localeCompare). Missing `order` sorts after any explicit order (stable relative to other unordered entries by module). Use this when plugin A depends on plugin B being loaded first.
</ParamField>
<ParamField path="plugins[].capabilities" type="array">
  Well-known capability tags the plugin advertises. Accepted values: `storeAdapter`, `auditSink`, `builtin`, `cliCommand`, `projection`. Unknown tags are kept (host extension) but surface an info diagnostic. Used for plugin discovery and dependency checking.
</ParamField>

```yaml
plugins:
  - module: '@my-org/manifest-plugin-audit'
    enabled: true
    order: 10
    capabilities:
      - auditSink
    options:
      level: strict
  - module: './local-plugins/naming-lint.js'
    order: 20
    capabilities:
      - cliCommand
```

</ParamField>

<ParamField path="env" type="object">
  Structured environment-variable mapping for **preflight validation** — grouped definitions of the variables your stores, auth, adapters, and custom settings require. Consumed by `manifest preflight` to check the environment and to generate `.env.example`.

~~This is distinct from the `env(VAR)` expression syntax…~~
> **Correction (2026-07-22):** There is **no** `env(VAR)` config-load inliner. This `env` block
> only _declares_ variable metadata for `manifest preflight` / `.env.example`. Inject secrets with
> `process.env` in `manifest.config.ts` (see [Durable storage](#durable-storage-for-generated-runtime-companions)).

The block has four optional groups — `stores`, `auth`, `adapters`, `custom` — each a map of an arbitrary key to an `EnvVarDefinition`. Each definition requires `name` and may set `description`, `required` (default `true`), `default`, and `example`.

```yaml
env:
  stores:
    primaryDb:
      name: DATABASE_URL
      description: Postgres connection string
      required: true
      example: postgres://user:pass@localhost:5432/app
  auth:
    clerk:
      name: CLERK_SECRET_KEY
      required: true
  adapters:
    outbox:
      name: OUTBOX_TABLE
      required: false
      default: manifest_outbox
  custom:
    region:
      name: AWS_REGION
      default: us-east-1
```

</ParamField>

## `mergeIntegrity`

<ParamField path="mergeIntegrity" type="object">
  Controls multi-file compilation behavior when duplicate names or cross-module references appear. Only applies when compiling multiple `.manifest` files (e.g. `manifest compile --all` or multi-module repos).
</ParamField>

<ParamField path="mergeIntegrity.onDuplicateEntity" type="string" default="error">
  Policy when multiple files declare an entity with the same name.

  - `error` (default): emit a diagnostic and fail compilation.
  - `lastWins`: silently use the last declaration (file load order wins).

  ```yaml
  mergeIntegrity:
    onDuplicateEntity: lastWins  # Allow duplicate entity names; last file wins
  ```
</ParamField>

<ParamField path="mergeIntegrity.onDuplicateCommand" type="string" default="error">
  Policy when multiple files declare a command with the same name (on the same entity or across entities). Accepted values: `error`, `lastWins`. Defaults to `error`.
</ParamField>

<ParamField path="mergeIntegrity.allowCrossModuleRefs" type="boolean" default="true">
  Whether relationships (`hasMany`, `hasOne`, `belongsTo`, `ref`) may target entities defined in other modules (other `.manifest` files). When `false`, cross-module relationships emit a diagnostic and fail compilation. Defaults to `true`.

  ```yaml
  mergeIntegrity:
    allowCrossModuleRefs: false  # Block relationships across modules
  ```
</ParamField>

<ParamField path="mergeIntegrity.forbidCycles" type="boolean" default="true">
  Whether circular relationship chains are rejected. When `true`, the compiler emits a diagnostic if a cycle is detected (e.g. `A hasMany B`, `B hasMany A`). This is a **required invariant** — setting it to `false` is rejected at config resolution time. Defaults to `true`.
</ParamField>

<ParamField path="mergeIntegrity" type="object">
  **Why this exists.** In multi-file projects, duplicate names and cross-file references can introduce silent bugs or ambiguous semantics. `mergeIntegrity` makes these conditions explicit and configurable. Use `lastWins` during rapid prototyping with split files; use `error` in production to enforce uniqueness. Use `allowCrossModuleRefs: false` to enforce module boundaries.
</ParamField>

## `validation`

<ParamField path="validation" type="object">
  Preflight validation rules that emit diagnostics when certain structural conditions are detected. These are **compile-time checks only** — they do not affect runtime behavior. Rules are disabled by default; enable the ones you want.
</ParamField>

<ParamField path="validation.rules" type="array">
  List of validation rule names to enable. Accepted values: `missing-policy`, `unused-entity`, `orphan-relationship`.

  <ParamField path="validation.rules" type="array">
  **`missing-policy`** — Emit a diagnostic when an entity has commands but no policies target it **and** `defaultPolicies` is empty. Helps catch missing authorization for command-bearing entities.

  **`unused-entity`** — Emit a diagnostic for "bare declaration" entities: not external, have no store, no entity-scoped commands, and are not referenced by any other entity's relationships. Helps identify dead code.

  **`orphan-relationship`** — Emit a diagnostic for one-sided relationships where an entity has `belongsTo` or `ref` pointing to a parent, but the parent lacks a corresponding `hasMany` or `hasOne` inverse. Helps catch typos and missing inverse declarations.
  </ParamField>

  ```yaml
  validation:
    rules:
      - missing-policy    # Flag entities with commands but no policies
      - unused-entity     # Flag entities that are never referenced
      - orphan-relationship  # Flag one-sided relationships
  ```
</ParamField>

<ParamField path="validation.failOn" type="array">
  Severity levels that should fail compilation. Accepted values: `ok`, `warn`, `block`. When omitted, all severities pass (diagnostics are emitted but compilation continues). Set to `['block']` or `['warn', 'block']` to halt on specific severities.
</ParamField>

## `provenance`

<ParamField path="provenance" type="object">
  Controls IR provenance stamps and lockfile behavior. IR always carries required provenance fields (schema version, compiler version, content hash). This config determines whether timestamps are deterministic and whether a lockfile is written.
</ParamField>

<ParamField path="provenance.stamp" type="boolean" default="true">
  When `false`, skip writing the provenance lockfile. IR is still stamped with required fields (this flag only controls the lockfile write). Defaults to `true`.
</ParamField>

<ParamField path="provenance.fields" type="array" default="['sourceHash', 'generatorVersion', 'irSchemaVersion', 'gitSha']">
  Requested stamp fields. Required IR fields are always stamped regardless of this list. `gitSha` is accepted but not yet written (deferred). Defaults to all four fields.
</ParamField>

<ParamField path="provenance.deterministic" type="boolean" default="false">
  When `true`, use a fixed `compiledAt` timestamp (`1970-01-01T00:00:00.000Z`) instead of wall-clock time. Required for reproducible builds. Defaults to `false`.
</ParamField>

<ParamField path="provenance.lockfile" type="string">
  Path for the provenance lockfile (relative to the project root). When omitted, the default is `.manifest/versions.json`.
</ParamField>

<ParamField path="provenance.failIfStale" type="boolean" default="false">
  When `true` and a lockfile exists: if source content (`contentHash`) matches the lock but the IR hash (`irHash`) differs under deterministic mode, compilation fails. This catches hand-edited IR or non-deterministic compiler behavior. Defaults to `false`.
</ParamField>

```yaml
provenance:
  deterministic: true  # Use fixed timestamp for reproducible builds
  lockfile: .manifest/versions.json
  failIfStale: true    # Fail if IR drifts despite unchanged sources
```

## `runtime`

<ParamField path="runtime" type="object">
  Central runtime build knobs that affect generated companion factories and execution mode. These settings are consumed by projections (Next.js, Express, Hono, Remix, SvelteKit) and by the generated runtime helpers.
</ParamField>

<ParamField path="runtime.executionMode" type="string" default="inline">
  Dispatcher execution mode for the Next.js projection. Accepted values: `inline`, `externalExecutor`.

  - `inline` (default): Generated handlers call `createManifestRuntime(...)` then `runtime.runCommand(...)` per request.
  - `externalExecutor`: Handlers import a configured executor and delegate — no `createManifestRuntime` or `runtime.runCommand` appears in emitted code.

  See [`projections.nextjs.options.dispatcher`](#projectionsnextjsoptionsdispatcher) for the executor migration guide.
</ParamField>

<ParamField path="runtime.determinism" type="object">
  Determinism settings for the generated runtime factory.

  <ParamField path="runtime.determinism.deterministicMode" type="boolean" default="false">
    When `true`, the factory injects `RuntimeOptions.deterministicMode` (fixed clock, seeded `generateId`). Useful for reproducible tests.
  </ParamField>

  <ParamField path="runtime.determinism.forbidWallClock" type="boolean" default="false">
    When `true`, the factory requires `RuntimeOptions.now` to be provided (no `Date.now` fallback). Enforces explicit time control.
  </ParamField>

  <ParamField path="runtime.determinism.seed" type="number">
    Fixed clock milliseconds when `forbidWallClock` is `true`. Also seeds the `generateId` sequence. Ignored unless `forbidWallClock` is set.
  </ParamField>
</ParamField>

<ParamField path="runtime.stores" type="string">
  Import path for store bindings (`ManifestRuntimeConfig` module). Fans into projection `runtimeConfigImport` when that projection-specific key is unset. Used by Next.js, Express, Hono, Remix, and SvelteKit projections.

  ```yaml
  runtime:
    stores: './manifest.config'  # Points to the TS config with store bindings
  ```
</ParamField>

<ParamField path="runtime.defaultContext" type="object">
  Default runtime context merged into `createManifestRuntime` calls (caller-provided context wins on conflict). Useful for global constants like `appVersion` or `region`.
</ParamField>

<ParamField path="runtime.concurrency" type="object">
  Parallel execution settings.

  <ParamField path="runtime.concurrency.maxParallelCommands" type="integer">
    Maps to `RuntimeOptions.maxParallelCommands` on the generated factory. Positive integer only; invalid values are ignored (unlimited parallelism).
  </ParamField>
</ParamField>

## `projections`

The `projections` block configures code generation per target. Each key matches a registered projection name. The keys validated by the config schema are `nextjs`, `routes`, `prisma`, and `prisma-store`; the `nextjs` projection is documented in full below, with `prisma`/`prisma-store` summarized at the end of this section. (Many more projections are registered — zod, drizzle, convex, openapi, kysely, graphql, and others — and are driven entirely by CLI flags; only the four above accept structured config blocks today.)

### `projections.nextjs.output`

<ParamField path="projections.nextjs.output" type="string" default="generated/">
  Directory where the Next.js projection writes generated route files. Used by `manifest generate` and `manifest build` as the default `--output` / `--code-output` value.

```yaml
projections:
  nextjs:
    output: app/api/
```

</ParamField>

### `projections.nextjs.options`

These options control the import paths and filtering behavior written into generated route files.

<ParamField path="projections.nextjs.options.authProvider" type="string" default="none">
  Auth provider name. Used when `authImportPath` is not set. Accepted values: `clerk`, `nextauth`, `custom`, `none`. Defaults to `none` so a basic entity generates runnable code with no auth wrapper — opt into a provider explicitly.

```yaml
options:
  authProvider: clerk
```

</ParamField>

<ParamField path="projections.nextjs.options.authImportPath" type="string">
  Import path for your auth helper. When set, this takes precedence over `authProvider`. Can be a local alias (`@/lib/auth`) or a workspace package (`@repo/auth/server`).

```yaml
options:
  authImportPath: '@repo/auth/server'
```

</ParamField>

<ParamField path="projections.nextjs.options.databaseImportPath" type="string" default="@/lib/database">
  Import path for your database client, injected into generated route files.

```yaml
options:
  databaseImportPath: '@repo/database'
```

</ParamField>

<ParamField path="projections.nextjs.options.runtimeImportPath" type="string" default="@/lib/manifest-runtime">
  Import path for the Manifest runtime helper, injected into generated route files.

```yaml
options:
  runtimeImportPath: '@repo/manifest/runtime'
```

</ParamField>

<ParamField path="projections.nextjs.options.responseImportPath" type="string" default="@/lib/manifest-response">
  Import path for the Manifest response helper, injected into generated route files.

```yaml
options:
  responseImportPath: '@repo/manifest/response'
```

</ParamField>

<ParamField path="projections.nextjs.options.includeTenantFilter" type="boolean" default="false">
  When `true`, generated read routes include a `WHERE tenantId = <tenant>` filter and POST handlers resolve a `tenantId` before executing. Defaults to `false` (single-tenant). Set to `true` for multi-tenant applications.

```yaml
options:
  includeTenantFilter: true
```

</ParamField>

<ParamField path="projections.nextjs.options.includeSoftDeleteFilter" type="boolean" default="false">
  When `true`, generated read routes include a `WHERE deletedAt IS NULL` filter. Defaults to `false`. Set to `true` if your data model uses soft deletes.

```yaml
options:
  includeSoftDeleteFilter: true
```

</ParamField>

<ParamField path="projections.nextjs.options.tenantIdProperty" type="string" default="tenantId">
  Property name used for tenant scoping in generated filters.

```yaml
options:
  tenantIdProperty: tenantId
```

</ParamField>

<ParamField path="projections.nextjs.options.deletedAtProperty" type="string" default="deletedAt">
  Property name used for soft-delete filtering in generated queries.

```yaml
options:
  deletedAtProperty: deletedAt
```

</ParamField>

<ParamField path="projections.nextjs.options.appDir" type="string" default="app/api">
  App Router base directory. Every generated route's `pathHint` is relative to this. The default `app/api` suits a standard (non-monorepo) Next.js project. For a monorepo, set this to e.g. `apps/api/app/api`.

```yaml
options:
  appDir: apps/api/app/api
```

</ParamField>

<ParamField path="projections.nextjs.options.strictMode" type="boolean" default="true">
  Whether generated TypeScript is written to be strict-mode friendly. Leave on unless you have a
  specific compatibility reason to disable it.
</ParamField>

<ParamField path="projections.nextjs.options.includeComments" type="boolean" default="true">
  Whether to emit explanatory comments above generated handlers. Set `false` to slim generated
  files.
</ParamField>

<ParamField path="projections.nextjs.options.indentSize" type="integer" default="2">
  Spaces of indentation in generated code. Must be between 1 and 8.
</ParamField>

### `projections.nextjs.options.tenantProvider`

Replaces the default `userTenantMapping.findUnique` pattern with a project-supplied lookup helper. When set, generated handlers import `functionName` from `importPath` and call it with the auth field named by `lookupKey`.

<ParamField path="projections.nextjs.options.tenantProvider.importPath" type="string" required>
  Module path that exports the tenant lookup helper.
</ParamField>

<ParamField path="projections.nextjs.options.tenantProvider.functionName" type="string" required>
  Named export to call (e.g. `getTenantIdForOrg`).
</ParamField>

<ParamField path="projections.nextjs.options.tenantProvider.lookupKey" type="string" required>
  Which auth-context field is passed as the lookup argument. Accepted values: `orgId`, `userId`.
</ParamField>

```yaml
projections:
  nextjs:
    options:
      tenantProvider:
        importPath: '@my-app/data'
        functionName: 'getTenantIdForOrg'
        lookupKey: 'orgId'
```

### `projections.nextjs.options.dispatcher`

The dispatcher is the canonical Manifest write surface at `POST /api/manifest/[entity]/commands/[command]`. Two execution modes are supported, and either can be selected from config alone — no projection-source edits required.

<ParamField path="projections.nextjs.options.dispatcher.enabled" type="boolean" default="true">
  When `false`, the `nextjs.dispatcher` surface emits no artifact and the CLI surfaces an
  info-diagnostic.
</ParamField>

<ParamField path="projections.nextjs.options.dispatcher.executionMode" type="string" default="inline">
  Where command execution happens.

- `inline` (default, back-compat): handler calls `createManifestRuntime(...)` then `runtime.runCommand(...)` per request.
- `externalExecutor`: handler imports the configured executor and delegates — **no `createManifestRuntime` or `runtime.runCommand` appears in the emitted code.**
  </ParamField>

<ParamField
  path="projections.nextjs.options.dispatcher.executorImportPath"
  type="string"
  default="@/lib/manifest-executor"
>
  Module path for the external executor. Only used when `executionMode` is `externalExecutor`.
</ParamField>

<ParamField
  path="projections.nextjs.options.dispatcher.executorImportName"
  type="string"
  default="executeManifestCommand"
>
  Named export to call on the external executor module.
</ParamField>

<ParamField
  path="projections.nextjs.options.dispatcher.deriveInstanceId"
  type="boolean"
  default="true"
>
  When `true` (default), the dispatcher extracts an `instanceId` from `body.instanceId` or `body.id`
  and forwards it to `runCommand` (inline mode) or the configured executor (`externalExecutor`
  mode). Non-create commands (release, archive, update, …) need this to address the target instance;
  create commands ignore it harmlessly. Override to `false` only with strong reason.
</ParamField>

<ParamField
  path="projections.nextjs.options.dispatcher.path"
  type="string"
  default="/manifest/[entity]/commands/[command]/route.ts"
>
  Dispatcher route path relative to `appDir`. Override only for non-canonical prefixes (e.g.
  `/api/v1/manifest/...`).
</ParamField>

<ParamField path="projections.nextjs.options.unauthorizedStatus" type="integer" default="401">
  HTTP status returned when the auth helper rejects the request OR when it throws (invalid/expired
  token). Must be in the 400–499 range. Auth failures MUST NEVER surface as 500. Override only if
  you standardise on 403 to avoid user-existence leak.
</ParamField>

### `projections.nextjs.options.concreteCommandRoutes`

Policy for the deprecated per-command "concrete" routes (the `nextjs.command` surface). These are **opt-in** in current releases — the canonical dispatcher is the only write surface by default.

<ParamField
  path="projections.nextjs.options.concreteCommandRoutes.enabled"
  type="boolean"
  default="false"
>
  **Default false (opt-in).** When `false`, `nextjs.command` artifacts are suppressed entirely and
  `manifest generate --surface all` does not emit them. Opt in only if you still need per-command
  routes for legacy callers.
</ParamField>

<ParamField
  path="projections.nextjs.options.concreteCommandRoutes.legacyAliasesOnly"
  type="boolean"
  default="true"
>
  When `true` (default) and `enabled` is `true`, generated concrete routes carry the `DEPRECATED
  ALIAS` banner pointing callers at the dispatcher. Set `false` only if you intentionally treat
  per-command routes as a first-class surface.
</ParamField>

### `projections.nextjs.options.readRoutes`

Policy for direct database read routes (`nextjs.route` for list, `nextjs.detail` for single-entity GET). Direct reads bypass the runtime engine for read performance and assume a Prisma-compatible client at `databaseImportPath`.

<ParamField path="projections.nextjs.options.readRoutes.enabled" type="boolean" default="true">
  When `false`, both list and detail read routes are suppressed (info diagnostic only).
</ParamField>

<ParamField
  path="projections.nextjs.options.readRoutes.directDbReads"
  type="boolean"
  default="true"
>
  When `false`, read route handlers are emitted but contain no inline Prisma call — useful for
  projects that route reads through a separate query layer.
</ParamField>

<Note>
**Prisma 7 note.** When `includeTenantFilter` and/or `includeSoftDeleteFilter` are true, the detail route emits `findFirst` (not `findUnique`). The previous template emitted `findUnique({ where: { id, tenantId, deletedAt: null } })`, which fails type-check on Prisma 7+. This is fixed in current releases.
</Note>

### `projections.prisma`

Configuration for the Prisma schema projection (emits `schema.prisma`). All option keys are optional.

<ParamField path="projections.prisma.output" type="string" default="schema.prisma">
  Path hint for the emitted `schema.prisma` artifact.
</ParamField>

<ParamField path="projections.prisma.options" type="object">
  Per-entity overrides for the emitted schema. Keys include `provider` (emits a datasource block plus a `prisma.config.ts` companion; one of `postgresql`, `mysql`, `sqlite`, `sqlserver`, `mongodb`, `cockroachdb`), `urlEnvVar` (default `DATABASE_URL`), `tableMappings`, `columnMappings`, `precision`, `indexes`, `typeMappings`, `foreignKeys`, `dbAttributes`, `fieldAttributes`, and a projection-local `naming` (see the top-level [`naming`](#param-naming) field — explicit `tableMappings`/`columnMappings` always win).

```yaml
projections:
  prisma:
    output: prisma/schema.prisma
    options:
      provider: postgresql
      urlEnvVar: DATABASE_URL
      naming: snake_case
      tableMappings:
        OrderLine: order_lines
```

</ParamField>

### `projections.prisma-store`

Configuration for the Prisma store metadata/registry projection (`GenericPrismaStore`). It emits a model-metadata module and a store-registry module, and extends every `projections.prisma.options` key for consistent naming with `schema.prisma` emission.

<ParamField
  path="projections.prisma-store.options.metadataOutput"
  type="string"
  default="prisma-model-metadata.generated.ts"
>
  Path hint for the emitted `prisma-model-metadata.generated.ts`.
</ParamField>

<ParamField
  path="projections.prisma-store.options.registryOutput"
  type="string"
  default="prisma-store-registry.generated.ts"
>
  Path hint for the emitted `prisma-store-registry.generated.ts`.
</ParamField>

<ParamField path="projections.prisma-store.options.accessorNames" type="object">
  Per-entity Prisma client delegate override (e.g. `{ "OrderLine": "orderLines" }`).
</ParamField>

<ParamField path="projections.prisma-store.options.storeImportPath" type="string">
  Module path for the `GenericPrismaStore` import in generated output.

<ParamField path="projections.prisma-store.options.metadataImportPath" type="string">

## externalExecutor migration

Use `executionMode: externalExecutor` when your app owns Manifest runtime construction (shared executor, custom store wiring, extra context) and wants the dispatcher to be a thin transport adapter.

**Step 1.** Author an executor in your app. Place it where `executorImportPath` points; the call shape below is the contract.

```ts
// apps/api/lib/manifest-executor.ts
import { createManifestRuntime } from './manifest-runtime';

export interface ManifestExecutorCall {
  entityName: string;
  commandName: string;
  input: unknown;
  instanceId?: string;
  context: {
    tenantId?: string;
    orgId?: string;
    actorId?: string;
    requestId?: string;
    source?: string;
    user?: Record<string, unknown>;
  };
}

export async function executeManifestCommand(call: ManifestExecutorCall) {
  const runtime = await createManifestRuntime(call.context);
  return runtime.runCommand(call.commandName, call.input, {
    entityName: call.entityName,
    ...(call.instanceId ? { instanceId: call.instanceId } : {}),
  });
}
```

**Step 2.** Flip config to point at your executor:

```yaml
projections:
  nextjs:
    options:
      dispatcher:
        executionMode: externalExecutor
        executorImportPath: '@my-app/manifest-executor'
        executorImportName: 'executeManifestCommand'
```

**Step 3.** Regenerate. The dispatcher route now imports your executor and delegates to it — no `createManifestRuntime` or `runtime.runCommand` in the emitted code.

**Step 4.** Confirm with the CLI:

```bash
manifest config inspect --json | jq '.projections.nextjs.options.dispatcher'
```

## CI / drift workflow

Downstream repos should commit `manifest.config.yaml` and pin generated output under VCS. The recommended four-step CI gate catches both Manifest default changes and local config drift:

```bash
# 1. Fail on invalid config.
manifest config validate

# 2. Snapshot the effective config (commit the snapshot under VCS).
manifest config inspect --json > .manifest/effective-config.snapshot.json
git diff --exit-code .manifest/effective-config.snapshot.json

# 3. Regenerate code from IR.
manifest build

# 4. Fail on drift between checked-in and just-generated code.
git diff --exit-code
```

This flow guarantees that the effective config has not silently changed and that emitted code matches what the current Manifest + config combination produces.

## Workspace monorepo imports

In a pnpm/yarn workspace (e.g. Turborepo or a custom monorepo), your shared packages are available under a workspace prefix like `@repo/`. Set the import path options to point at those packages instead of local `@/lib/` aliases:

```yaml
projections:
  nextjs:
    output: apps/api/app/api/
    options:
      authImportPath: '@repo/auth/server'
      databaseImportPath: '@repo/database'
      runtimeImportPath: '@repo/manifest/runtime'
      responseImportPath: '@repo/manifest/response'
```

Generated routes will then import from those workspace packages:

```typescript
import { auth } from '@repo/auth/server';
import { database } from '@repo/database';
import { createRuntime } from '@repo/manifest/runtime';
import { manifestSuccessResponse, manifestErrorResponse } from '@repo/manifest/response';
```

`stores: { postgres: { url: env(...) } }`, and `MANIFEST_ENV` overlay files
(`manifest.config.staging.yaml`)~~

> **Correction (2026-07-22):** Those three mechanisms are **not implemented**. YAML load does not
> expand `env(…)` strings, there is no connection-string `stores:` block in build YAML, and the CLI
> does not merge `manifest.config.<env>.yaml` from `MANIFEST_ENV`.

**Use instead:**

1. **Preflight declarations** — the `env:` block above (metadata only) with `manifest preflight`

3. **Multi-environment** — supply different env vars / deploy secrets per environment, or keep
   separate config files and point the process at the right one yourself (no built-in overlay
   selector)

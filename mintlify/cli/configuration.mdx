`manifest.config.yaml` is the project-level config file that the CLI loads from your project root before executing any command. You create it by running `manifest init` or by writing it by hand. Every field is optional â€” the CLI falls back to built-in defaults for any field you omit.

## Config file locations

The CLI searches for a config file in the following order, using the first one it finds:

1. `manifest.config.yaml`
2. `manifest.config.yml`
3. `.manifestrc.yaml`
4. `.manifestrc.yml`

All four files live at your project root (the directory where you run `manifest` commands).

## Option priority

When resolving any value, the CLI checks three sources in this order:

1. **CLI flag** (highest priority) â€” value passed directly on the command line
2. **Config file** â€” value from `manifest.config.yaml`
3. **Built-in default** (lowest priority) â€” fallback hardcoded in the CLI

You can always override a config value for a single run without modifying the file.

## Top-level fields

<ParamField path="$schema" type="string">
  Optional pointer to the config JSON Schema for editor IntelliSense. Manifest does **not** publish a resolvable schema URL, and `manifest config validate` loads the schema bundled with the package â€” it never fetches a URL. For autocomplete without a dead link, map the bundled schema in `.vscode/settings.json`:

  ```jsonc
  {
    "yaml.schemas": {
      "./node_modules/@angriff36/manifest/docs/spec/config/manifest.config.schema.json": ["manifest.config.yaml", ".manifestrc.yaml"]
    }
  }
  ```

  If you do set `$schema`, point it at a real local path â€” never a public URL Manifest doesn't host.
</ParamField>

<ParamField path="src" type="string" default="**/*.manifest">
  Glob pattern that selects your `.manifest` source files. Used by `manifest compile` and `manifest build` when no `source` argument is passed on the command line.

  ```yaml
  src: modules/**/*.manifest
  ```
</ParamField>

<ParamField path="output" type="string" default="ir/">
  Directory where compiled IR JSON files are written. Used by `manifest compile` as the default `--output` value, and by `manifest build` as the default `--ir-output` value.

  ```yaml
  output: ir/
  ```
</ParamField>

<ParamField path="naming" type="string | object">
  Global identifier-casing convention inherited by projections that map IR names to physical database names (currently the Prisma projection). Opt-in; omit to emit IR names verbatim. A per-projection `projections.<name>.options.naming` overrides this default.

  The string shorthand `snake_case` expands to `{ table: snake_case, column: snake_case, pluralizeTables: true }` â€” e.g. `createdAt` â†’ `@map("created_at")`, `Widget` â†’ `@@map("widgets")`. The object form tunes each axis (`table`: `snake_case` | `camelCase` | `PascalCase` | `preserve`; `column`: `snake_case` | `camelCase` | `preserve`; `pluralizeTables`: boolean, default `true`).

  ```yaml
  naming: snake_case
  # or:
  naming:
    table: snake_case
    column: snake_case
    pluralizeTables: true
  ```

  Only `@map`/`@@map` attributes are added â€” Prisma model names and field identifiers stay as the IR name, so relations and indexes are unaffected. Explicit `tableMappings`/`columnMappings` always win and are the escape hatch for irregular plurals.
</ParamField>

<ParamField path="prismaSchema" type="string">
  Path to your Prisma schema file, used by property alignment checks. When omitted, the CLI searches `prisma/schema.prisma`, `schema.prisma`, and `db/schema.prisma` in that order.

  ```yaml
  prismaSchema: prisma/schema.prisma
  ```
</ParamField>

## `projections`

The `projections` block configures code generation for each target framework. Currently the only built-in projection is `nextjs`.

### `projections.nextjs.output`

<ParamField path="projections.nextjs.output" type="string" default="generated/">
  Directory where the Next.js projection writes generated route files. Used by `manifest generate` and `manifest build` as the default `--output` / `--code-output` value.

  ```yaml
  projections:
    nextjs:
      output: app/api/
  ```
</ParamField>

### `projections.nextjs.options`

These options control the import paths and filtering behavior written into generated route files.

<ParamField path="projections.nextjs.options.authProvider" type="string" default="clerk">
  Auth provider name. Used when `authImportPath` is not set. Accepted values: `clerk`, `nextauth`, `custom`, `none`.

  ```yaml
  options:
    authProvider: clerk
  ```
</ParamField>

<ParamField path="projections.nextjs.options.authImportPath" type="string">
  Import path for your auth helper. When set, this takes precedence over `authProvider`. Can be a local alias (`@/lib/auth`) or a workspace package (`@repo/auth/server`).

  ```yaml
  options:
    authImportPath: "@repo/auth/server"
  ```
</ParamField>

<ParamField path="projections.nextjs.options.databaseImportPath" type="string" default="@/lib/database">
  Import path for your database client, injected into generated route files.

  ```yaml
  options:
    databaseImportPath: "@repo/database"
  ```
</ParamField>

<ParamField path="projections.nextjs.options.runtimeImportPath" type="string" default="@/lib/manifest-runtime">
  Import path for the Manifest runtime helper, injected into generated route files.

  ```yaml
  options:
    runtimeImportPath: "@repo/manifest/runtime"
  ```
</ParamField>

<ParamField path="projections.nextjs.options.responseImportPath" type="string" default="@/lib/manifest-response">
  Import path for the Manifest response helper, injected into generated route files.

  ```yaml
  options:
    responseImportPath: "@repo/manifest/response"
  ```
</ParamField>

<ParamField path="projections.nextjs.options.includeTenantFilter" type="boolean" default="true">
  When `true`, generated read routes include a `WHERE tenantId = <tenant>` filter. Set to `false` for single-tenant applications.

  ```yaml
  options:
    includeTenantFilter: true
  ```
</ParamField>

<ParamField path="projections.nextjs.options.includeSoftDeleteFilter" type="boolean" default="true">
  When `true`, generated read routes include a `WHERE deletedAt IS NULL` filter. Set to `false` if your data model does not use soft deletes.

  ```yaml
  options:
    includeSoftDeleteFilter: true
  ```
</ParamField>

<ParamField path="projections.nextjs.options.tenantIdProperty" type="string" default="tenantId">
  Property name used for tenant scoping in generated filters.

  ```yaml
  options:
    tenantIdProperty: tenantId
  ```
</ParamField>

<ParamField path="projections.nextjs.options.deletedAtProperty" type="string" default="deletedAt">
  Property name used for soft-delete filtering in generated queries.

  ```yaml
  options:
    deletedAtProperty: deletedAt
  ```
</ParamField>

<ParamField path="projections.nextjs.options.appDir" type="string" default="apps/api/app/api">
  App Router base directory. Every generated route's `pathHint` is relative to this. For a non-monorepo Next.js project, set this to `app/api`.

  ```yaml
  options:
    appDir: app/api
  ```
</ParamField>

<ParamField path="projections.nextjs.options.strictMode" type="boolean" default="true">
  Whether generated TypeScript is written to be strict-mode friendly. Leave on unless you have a specific compatibility reason to disable it.
</ParamField>

<ParamField path="projections.nextjs.options.includeComments" type="boolean" default="true">
  Whether to emit explanatory comments above generated handlers. Set `false` to slim generated files.
</ParamField>

<ParamField path="projections.nextjs.options.indentSize" type="integer" default="2">
  Spaces of indentation in generated code. Must be between 1 and 8.
</ParamField>

### `projections.nextjs.options.tenantProvider`

Replaces the default `userTenantMapping.findUnique` pattern with a project-supplied lookup helper. When set, generated handlers import `functionName` from `importPath` and call it with the auth field named by `lookupKey`.

<ParamField path="projections.nextjs.options.tenantProvider.importPath" type="string" required>
  Module path that exports the tenant lookup helper.
</ParamField>

<ParamField path="projections.nextjs.options.tenantProvider.functionName" type="string" required>
  Named export to call (e.g. `getTenantIdForOrg`).
</ParamField>

<ParamField path="projections.nextjs.options.tenantProvider.lookupKey" type="string" required>
  Which auth-context field is passed as the lookup argument. Accepted values: `orgId`, `userId`.
</ParamField>

```yaml
projections:
  nextjs:
    options:
      tenantProvider:
        importPath: "@my-app/data"
        functionName: "getTenantIdForOrg"
        lookupKey: "orgId"
```

### `projections.nextjs.options.dispatcher`

The dispatcher is the canonical Manifest write surface at `POST /api/manifest/[entity]/commands/[command]`. Two execution modes are supported, and either can be selected from config alone â€” no projection-source edits required.

<ParamField path="projections.nextjs.options.dispatcher.enabled" type="boolean" default="true">
  When `false`, the `nextjs.dispatcher` surface emits no artifact and the CLI surfaces an info-diagnostic.
</ParamField>

<ParamField path="projections.nextjs.options.dispatcher.executionMode" type="string" default="inline">
  Where command execution happens.

  - `inline` (default, back-compat): handler calls `createManifestRuntime(...)` then `runtime.runCommand(...)` per request.
  - `externalExecutor`: handler imports the configured executor and delegates â€” **no `createManifestRuntime` or `runtime.runCommand` appears in the emitted code.**
</ParamField>

<ParamField path="projections.nextjs.options.dispatcher.executorImportPath" type="string" default="@/lib/manifest-executor">
  Module path for the external executor. Only used when `executionMode` is `externalExecutor`.
</ParamField>

<ParamField path="projections.nextjs.options.dispatcher.executorImportName" type="string" default="executeManifestCommand">
  Named export to call on the external executor module.
</ParamField>

<ParamField path="projections.nextjs.options.dispatcher.deriveInstanceId" type="boolean" default="true">
  When `true` (default), the dispatcher extracts an `instanceId` from `body.instanceId` or `body.id` and forwards it to `runCommand` (inline mode) or the configured executor (`externalExecutor` mode). Non-create commands (release, archive, update, â€¦) need this to address the target instance; create commands ignore it harmlessly. Override to `false` only with strong reason.
</ParamField>

<ParamField path="projections.nextjs.options.dispatcher.path" type="string" default="/manifest/[entity]/commands/[command]/route.ts">
  Dispatcher route path relative to `appDir`. Override only for non-canonical prefixes (e.g. `/api/v1/manifest/...`).
</ParamField>

<ParamField path="projections.nextjs.options.unauthorizedStatus" type="integer" default="401">
  HTTP status returned when the auth helper rejects the request OR when it throws (invalid/expired token). Must be in the 400â€“499 range. Auth failures MUST NEVER surface as 500. Override only if you standardise on 403 to avoid user-existence leak.
</ParamField>

### `projections.nextjs.options.concreteCommandRoutes`

Policy for the deprecated per-command "concrete" routes (the `nextjs.command` surface). As of v0.7.x these are **opt-in** â€” the canonical dispatcher is the only write surface by default.

<ParamField path="projections.nextjs.options.concreteCommandRoutes.enabled" type="boolean" default="false">
  **Default false (opt-in).** When `false`, `nextjs.command` artifacts are suppressed entirely and `manifest generate --surface all` does not emit them. Opt in only if you still need per-command routes for legacy callers.
</ParamField>

<ParamField path="projections.nextjs.options.concreteCommandRoutes.legacyAliasesOnly" type="boolean" default="true">
  When `true` (default) and `enabled` is `true`, generated concrete routes carry the `DEPRECATED ALIAS` banner pointing callers at the dispatcher. Set `false` only if you intentionally treat per-command routes as a first-class surface.
</ParamField>

### `projections.nextjs.options.readRoutes`

Policy for direct database read routes (`nextjs.route` for list, `nextjs.detail` for single-entity GET). Direct reads bypass the runtime engine for read performance and assume a Prisma-compatible client at `databaseImportPath`.

<ParamField path="projections.nextjs.options.readRoutes.enabled" type="boolean" default="true">
  When `false`, both list and detail read routes are suppressed (info diagnostic only).
</ParamField>

<ParamField path="projections.nextjs.options.readRoutes.directDbReads" type="boolean" default="true">
  When `false`, read route handlers are emitted but contain no inline Prisma call â€” useful for projects that route reads through a separate query layer.
</ParamField>

<Note>
**Prisma 7 note.** When `includeTenantFilter` and/or `includeSoftDeleteFilter` are true, the detail route emits `findFirst` (not `findUnique`). The previous template emitted `findUnique({ where: { id, tenantId, deletedAt: null } })`, which fails type-check on Prisma 7+. Fixed in v0.7.x.
</Note>

## externalExecutor migration

Use `executionMode: externalExecutor` when your app owns Manifest runtime construction (shared executor, custom store wiring, extra context) and wants the dispatcher to be a thin transport adapter.

**Step 1.** Author an executor in your app. Place it where `executorImportPath` points; the call shape below is the contract.

```ts
// apps/api/lib/manifest-executor.ts
import { createManifestRuntime } from "./manifest-runtime";

export interface ManifestExecutorCall {
  entityName: string;
  commandName: string;
  input: unknown;
  instanceId?: string;
  context: {
    tenantId?: string;
    orgId?: string;
    actorId?: string;
    requestId?: string;
    source?: string;
    user?: Record<string, unknown>;
  };
}

export async function executeManifestCommand(call: ManifestExecutorCall) {
  const runtime = await createManifestRuntime(call.context);
  return runtime.runCommand(call.commandName, call.input, {
    entityName: call.entityName,
    ...(call.instanceId ? { instanceId: call.instanceId } : {}),
  });
}
```

**Step 2.** Flip config to point at your executor:

```yaml
projections:
  nextjs:
    options:
      dispatcher:
        executionMode: externalExecutor
        executorImportPath: "@my-app/manifest-executor"
        executorImportName: "executeManifestCommand"
```

**Step 3.** Regenerate. The dispatcher route now imports your executor and delegates to it â€” no `createManifestRuntime` or `runtime.runCommand` in the emitted code.

**Step 4.** Confirm with the CLI:

```bash
manifest config inspect --json | jq '.projections.nextjs.options.dispatcher'
```

## CI / drift workflow

Downstream repos should commit `manifest.config.yaml` and pin generated output under VCS. The recommended four-step CI gate catches both Manifest default changes and local config drift:

```bash
# 1. Fail on invalid config.
manifest config validate

# 2. Snapshot the effective config (commit the snapshot under VCS).
manifest config inspect --json > .manifest/effective-config.snapshot.json
git diff --exit-code .manifest/effective-config.snapshot.json

# 3. Regenerate code from IR.
manifest build

# 4. Fail on drift between checked-in and just-generated code.
git diff --exit-code
```

This flow guarantees that the effective config has not silently changed and that emitted code matches what the current Manifest + config combination produces.

## Workspace monorepo imports

In a pnpm/yarn workspace (e.g. Turborepo or a custom monorepo), your shared packages are available under a workspace prefix like `@repo/`. Set the import path options to point at those packages instead of local `@/lib/` aliases:

```yaml
projections:
  nextjs:
    output: apps/api/app/api/
    options:
      authImportPath: "@repo/auth/server"
      databaseImportPath: "@repo/database"
      runtimeImportPath: "@repo/manifest/runtime"
      responseImportPath: "@repo/manifest/response"
```

Generated routes will then import from those workspace packages:

```typescript
import { auth } from "@repo/auth/server";
import { database } from "@repo/database";
import { createRuntime } from "@repo/manifest/runtime";
import { manifestSuccessResponse, manifestErrorResponse } from "@repo/manifest/response";
```

<Tip>
After editing `manifest.config.yaml`, run `manifest build` with no flags to confirm the CLI picks up your changes. The build summary printed at the end shows the resolved `ir-output` and `code-output` paths.
</Tip>

## Environment variable mapping

Store connection strings and sensitive configuration can be injected via environment variables instead of hardcoding them in the config file. The CLI supports `env()` expressions in string-valued fields:

```yaml
projections:
  nextjs:
    options:
      databaseImportPath: "@/lib/database"
      runtimeImportPath: "@/lib/manifest-runtime"

stores:
  postgres:
    url: env(DATABASE_URL)
    schema: env(DATABASE_SCHEMA, "public")
  redis:
    url: env(REDIS_URL)
  dynamodb:
    region: env(AWS_REGION, "us-east-1")
    table: env(DYNAMO_TABLE)
```

### `env()` syntax

| Expression | Behavior |
|------------|----------|
| `env(VAR_NAME)` | Read `VAR_NAME` from the environment. Throws if unset. |
| `env(VAR_NAME, "default")` | Read `VAR_NAME` from the environment. Falls back to `"default"` if unset. |

Environment variable resolution happens at config load time, before any command executes. This means:

- Missing required variables cause the CLI to exit with a clear error message naming the missing variable.
- Defaults are string values â€” no type coercion is performed.
- `env()` can only appear in string-valued config fields.

## Multi-environment configuration

For projects that deploy to multiple environments (development, staging, production), use config overlays:

```yaml
# manifest.config.yaml
src: modules/**/*.manifest
output: ir/

stores:
  postgres:
    url: env(DATABASE_URL)

---

# manifest.config.staging.yaml (overlay)
stores:
  postgres:
    url: env(STAGING_DATABASE_URL)
```

The CLI selects the overlay based on the `MANIFEST_ENV` environment variable:

```bash
# Use base config
manifest build

# Use staging overlay
MANIFEST_ENV=staging manifest build
```

<Note>
Overlays are merged shallowly â€” an overlay's `stores.postgres` replaces the base `stores.postgres` entirely. For deep merging, provide the full structure in the overlay.
</Note>

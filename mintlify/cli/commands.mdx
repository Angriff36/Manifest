Every command reads `manifest.config.yaml` from your project root when it exists. CLI flags always take precedence over config values. Run any command with `--help` to see its options inline.

## `manifest init`

Runs an interactive prompt that collects your project settings and writes a `manifest.config.yaml` to the project root. If a config file already exists, the command exits without overwriting it unless you pass `--force`.

```bash
manifest init
```

### Questions asked

| Prompt                                  | Default         |
| --------------------------------------- | --------------- |
| Where are your `.manifest` files?       | `**/*.manifest` |
| Where should IR output go?              | `ir/`           |
| Will you be generating code from IR?    | No              |
| What projection target?                 | `nextjs`        |
| Where should generated code be written? | `app/api`       |

If the target is `nextjs` or `remix`, the generated code output defaults to `app/api`. For any other target it defaults to `generated`.

### Options

<ParamField path="-f, --force" type="boolean" default="false">
  Overwrite an existing `manifest.config.yaml` (or CI workflow file when `--ci` is used). The
  interactive setup requires a TTY; in non-interactive shells it writes a minimal config with
  defaults.
</ParamField>

<ParamField path="--ci" type="string">
  Generate a CI workflow for the given provider (currently `github`) instead of running the
  interactive config setup. See [CI/CD](/cli/ci-cd).
</ParamField>

<ParamField path="--node-versions" type="string">
  Comma-separated Node.js versions for the generated CI matrix (default: `18,20,22`). Only
  meaningful with `--ci`. Prefer `20,22` — package `engines.node` is `>=20`.
</ParamField>

### Examples

```bash
# Interactive setup
manifest init

# Overwrite an existing config
manifest init --force
```

<Warning>
  `manifest init --force` requires an interactive terminal. In headless or non-TTY environments the
  command writes a minimal default config instead of prompting.
</Warning>

## `manifest compile [source]`

Parses one or more `.manifest` source files and writes the compiled IR as `.ir.json` files. When no source is provided, the CLI uses the glob pattern from your config (or `**/*.manifest` by default).

```bash
manifest compile [source] [options]
```

`source` can be a single `.manifest` file path, a directory, or omitted to use the config glob.

### Options

<ParamField path="-o, --output" type="string">
  Output directory or file path. The flag itself has **no default** — when omitted, the CLI falls
  back to the config's `output` field if a config file is present, and otherwise writes alongside
  the source / to stdout depending on the input. (The `ir/` value is the documented config-level
  fallback, not a flag default.) When the output is a directory, each source file produces a
  matching `.ir.json` file inside it.
</ParamField>

<ParamField path="-g, --glob" type="string">
  Glob pattern to select multiple source files. Use this when you pass a directory as `source` and
  want a non-default pattern, or when you want to specify patterns without a config file.
</ParamField>

<ParamField path="-d, --diagnostics" type="boolean" default="false">
  Print compiler diagnostics (errors, warnings, and informational messages) to the console after
  each file is compiled.
</ParamField>

<ParamField path="--pretty" type="boolean" default="true">
  Pretty-print the IR JSON output with two-space indentation. Disable for compact output.
</ParamField>

<ParamField path="--merge" type="boolean" default="false">
  Merge multiple source files into a single IR, resolving `use` declarations across files. Produces
  one combined IR rather than one `.ir.json` per source.
</ParamField>

<ParamField path="--entry" type="string...">
  Entry file(s) for merge compilation (repeatable). The merge starts from these files and follows
  their `use` declarations. Auto-detected when omitted. Only meaningful together with `--merge`.
</ParamField>

<ParamField path="--all" type="boolean" default="false">
  Read all settings from `manifest.config.yaml` and use the config-driven flow instead of individual flags.
  When set, the command uses config values for `output`, `glob`, `diagnostics`, and other options.
  This is the preferred workflow for projects with a config file.
</ParamField>

<ParamField path="--fail-on <policy>" type="string" default="block">
  CI exit policy: `block` (errors only), `warn` (errors or warnings), or `never` (report-only).
</ParamField>

<Note>
  **Idempotent output.** When an output `.ir.json` already exists and was produced from
  byte-identical source (matching `provenance.contentHash`), the compiler reuses the prior
  `provenance.compiledAt` (recomputing `irHash` against it) so re-compiling unchanged source is
  byte-identical — no spurious git diff. A fresh `compiledAt` lands only when the source actually
  changes. Applies to single-file and `--merge` compilation.
</Note>

<Tip>
  **Duplicate event names** produce a compile warning (shown with `--diagnostics` and in the LSP):
  two declarations of the same event name collide in the event registry, so one shadows the other.
  Rename one.
</Tip>

### Examples

```bash
# Compile a single file; output goes to the config output dir if set
manifest compile modules/recipe.manifest

# Compile a single file to a specific directory
manifest compile modules/recipe.manifest --output ir/

# Compile all .manifest files matching a glob
manifest compile --glob "modules/**/*.manifest" --output ir/

# Compile and show diagnostics
manifest compile modules/recipe.manifest --output ir/ --diagnostics

# Use all config defaults (finds all .manifest files via config src pattern)
manifest compile
```

## `manifest generate <ir>`

Reads one or more compiled IR files and generates code using the specified projection and surface. `<ir>` can be a single `.ir.json` file or a directory; when it is a directory, the CLI processes every `.ir.json` file found inside it.

```bash
manifest generate <ir> [options]
```

### Options

<ParamField path="-p, --projection" type="string" default="nextjs">
  Projection to use for code generation. Accepts **any registered projection name** — dispatch is
  generic via the projection registry, so this is not a fixed list. Registered projections include
  `nextjs`, `routes`, `prisma`, `prisma-store`, `zod`, `drizzle`, `convex`, `openapi`, `kysely`,
  `graphql`, `react-query`, `express`, `hono`, `mermaid`, `jsonschema`, `storybook`, `sveltekit`,
  `pydantic`, `dynamodb`, `dart`, `terraform`, `elasticsearch`, `analytics`, `materialized-views`,
  and more. Adding a new projection means registering it, not editing the CLI.
</ParamField>

<ParamField path="-s, --surface" type="string" default="all">
  Which surface(s) to generate. Valid surface names are projection-specific; pass `all` to emit every surface the projection declares.

**nextjs** (default): `route` · `detail` · `command` · `dispatcher` · `subscribe` · `subscriptionHook` · `sharedRuntime` · `schedule` · `webhook` · `companions` · `ts.types` · `ts.client` · `all` (everything except concrete per-command routes, which are opt-in)

For other projections the CLI default `all` resolves to the projection's declared surface set (e.g. the `prisma` projection emits a single `schema.prisma` artifact). An unknown surface name is reported as a diagnostic, not silently ignored.

</ParamField>

<ParamField path="-o, --output" type="string">
  Output directory for generated files. Falls back to the config's `projections.<name>.output` value (e.g. `projections.nextjs.output`, `projections.prisma.output`) when not provided.
</ParamField>

<ParamField path="--auth" type="string">
  *(nextjs only)* Auth provider name or import path (e.g. `clerk` or `@repo/auth/server`). Falls
  back to `projections.nextjs.options.authImportPath`, then `authProvider`, then the `authProvider`
  default of `none` (no auth wrapper).
</ParamField>

<ParamField path="--database" type="string">
  *(nextjs only)* Database import path used in generated files (e.g. `@/lib/database` or
  `@repo/database`). Falls back to `projections.nextjs.options.databaseImportPath`, then defaults to
  `@/lib/database`.
</ParamField>

<ParamField path="--runtime" type="string">
  *(nextjs only)* Runtime import path (e.g. `@/lib/manifest-runtime` or `@repo/manifest/runtime`).
  Falls back to `projections.nextjs.options.runtimeImportPath`, then defaults to
  `@/lib/manifest-runtime`.
</ParamField>

<ParamField path="--response" type="string">
  *(nextjs only)* Response helpers import path (e.g. `@/lib/manifest-response` or
  `@repo/manifest/response`). Falls back to `projections.nextjs.options.responseImportPath`, then
  defaults to `@/lib/manifest-response`.
</ParamField>

<ParamField path="--check" type="boolean" default="false">
  Drift mode: regenerate code in memory and compare it to the committed files **without writing
  anything**. Exits non-zero and lists the drifted files (a missing file or any byte difference
  counts as drift). Use it in CI to assert that committed code equals freshly generated code — the
  codegen analogue of `prettier --check`. See [CI/CD](/cli/ci-cd).
</ParamField>

<ParamField path="--all" type="boolean" default="false">
  Read all settings from `manifest.config.yaml` and use the config-driven flow instead of individual flags.
  When set, the command uses config values for `output`, `projection`, `surface`, and other options.
  This is the preferred workflow for projects with a config file.
</ParamField>

#### Examples

```bash
# Default: nextjs, all surfaces
manifest generate ir/

# Just the canonical dispatcher route
manifest generate ir/ --surface dispatcher --output app/api

# Prisma schema from compiled IR
manifest generate ir/ --projection prisma --surface prisma.schema --output prisma

# CI drift gate: fail if committed code is stale vs. the IR
manifest generate ir/ --output generated/ --check
```

Projection-specific options (table names, column mappings, decimal precision, indexes, type overrides, foreign-key names for prisma; auth provider, tenant filters, dispatcher mode for nextjs) live in `manifest.config.yaml` under `projections.<name>.options`. See [Configuration](/cli/configuration) and the per-projection guides ([Next.js](/integration/nextjs), [Prisma](/integration/prisma)).

#### Exit codes

`manifest generate` exits **non-zero** when any projection returns a diagnostic with `severity: 'error'`. Example trigger conditions for the Prisma projection: bare `number`-typed properties (`PRISMA_AMBIGUOUS_NUMBER`), unknown `type.name` with no `typeMappings` override (`PRISMA_UNKNOWN_TYPE`), or an unknown surface name (`UNKNOWN_SURFACE`). The failing diagnostic codes are surfaced in stdout/stderr so CI logs are actionable. Info / warning severities log but do not fail the command.

### Examples

```bash
# Generate using all config defaults (config file must exist)
manifest generate ir/recipe.ir.json

# Override the output directory
manifest generate ir/recipe.ir.json --output apps/api/app/api/

# Generate only TypeScript types
manifest generate ir/recipe.ir.json --surface types --output generated/types.ts

# Generate only the client SDK
manifest generate ir/recipe.ir.json --surface client --output generated/client.ts

# Generate from a directory of IR files with workspace imports
manifest generate ir/ \
  --output apps/api/app/api/ \
  --auth @repo/auth/server \
  --database @repo/database \
  --runtime @repo/manifest/runtime \
  --response @repo/manifest/response
```

## `manifest build [source]`

Runs compile and generate as a single pipeline. The source `.manifest` files are compiled to IR in `--ir-output`, then code is generated from that IR into `--code-output`. All config defaults apply to both stages.

```bash
manifest build [source] [options]
```

`source` can be a single `.manifest` file, a glob pattern, or omitted to use the config `src` field.

### Options

<ParamField path="-p, --projection" type="string" default="nextjs">
  Projection name for the generate stage. Accepts any registered projection (e.g. `nextjs`,
  `prisma`, `zod`, `drizzle`, `convex`, `openapi`, `kysely`, …) — see [`manifest
  generate`](#manifest-generate-ir) for the full list.
</ParamField>

<ParamField path="-s, --surface" type="string" default="all">
  Surface(s) to generate. Same values as `manifest generate` — projection-specific, with `all`
  emitting every declared surface (nextjs: `route`, `detail`, `command`, `dispatcher`, `types`,
  `client`).
</ParamField>

<ParamField path="--ir-output" type="string">
  Directory to write compiled IR files. Falls back to the config `output` field, then defaults to
  `ir/`.
</ParamField>

<ParamField path="--code-output" type="string">
  Directory to write generated code files. Falls back to `projections.nextjs.output` in config, then
  defaults to `generated/`.
</ParamField>

<ParamField path="-g, --glob" type="string">
  Glob pattern for selecting multiple source files during the compile stage.
</ParamField>

<ParamField path="--auth" type="string">
  Auth provider or import path, passed through to the generate stage.
</ParamField>

<ParamField path="--database" type="string">
  Database import path, passed through to the generate stage.
</ParamField>

<ParamField path="--runtime" type="string">
  Runtime import path, passed through to the generate stage.
</ParamField>

<ParamField path="--response" type="string">
  Response helpers import path, passed through to the generate stage.
</ParamField>

<ParamField path="--all" type="boolean" default="false">
  Read all settings from `manifest.config.yaml` and use the config-driven flow instead of individual flags.
  When set, the command uses config values for compile and generate stages (output, projection, surface, etc.).
  This is the preferred workflow for projects with a config file.
</ParamField>

### Examples

```bash
# Use all config defaults (recommended after manifest init)
manifest build

# Build a single source file with explicit outputs
manifest build modules/recipe.manifest \
  --ir-output ir/ \
  --code-output app/api/

# Override the code output directory for a staging target
manifest build modules/recipe.manifest --code-output apps/staging/app/api/

# Build multiple files via glob with workspace imports
manifest build \
  --glob "modules/**/*.manifest" \
  --ir-output ir/ \
  --code-output apps/api/app/api/ \
  --auth @repo/auth/server \
  --database @repo/database \
  --runtime @repo/manifest/runtime \
  --response @repo/manifest/response
```

## `manifest docs [source]`

Generates a static API reference site from one or more Manifest sources. Inputs can be `.manifest` (compiled first) or `.ir.json` files (loaded directly).

```bash
manifest docs [source] [options]
```

`source` can be a single file, a directory, or a glob.

### Options

<ParamField path="-o, --output" type="string" default="docs-site">
  Output directory for generated docs.
</ParamField>

<ParamField path="-f, --format" type="string" default="html">
  Output format: `html` or `markdown`.
</ParamField>

<ParamField path="-t, --title" type="string" default="Manifest API Reference">
  Site title used in generated pages.
</ParamField>

### IR merge workflow (`mergeIrs` behavior)

When `manifest docs` receives multiple inputs, it merges all compiled/loaded IR objects into one aggregate IR for page generation.

Expected merge contract:

1. Initialize all required IR arrays before merging: `modules`, `values`, `entities`, `enums`, `stores`, `events`, `commands`, `policies`.
2. Append each input IR array into the merged array.
3. Propagate `values` exactly like the other top-level arrays.

```ts
const mergedIR: IR = {
  version: '1.0',
  provenance: firstProvenance,
  modules: [],
  values: [],
  entities: [],
  enums: [],
  stores: [],
  events: [],
  commands: [],
  policies: [],
};

for (const ir of inputIRs) {
  mergedIR.modules.push(...ir.modules);
  mergedIR.values.push(...ir.values);
  mergedIR.entities.push(...ir.entities);
  mergedIR.enums.push(...ir.enums);
  mergedIR.stores.push(...ir.stores);
  mergedIR.events.push(...ir.events);
  mergedIR.commands.push(...ir.commands);
  mergedIR.policies.push(...ir.policies);
}
```

<Note>
  `values` is required by the IR v1 schema even when empty. Any merge path that does not initialize
  and populate `values` can produce nonconformant merged IR and break downstream tooling.
</Note>

### Examples

```bash
# Build docs from all manifest/IR files under src
manifest docs src/ --format markdown --output docs-site

# Build docs from a single compiled IR
manifest docs ir/app.ir.json --output docs-site
```

## `manifest validate [ir]`

Validates one or more IR JSON files against the IR v1 JSON schema using [Ajv](https://ajv.js.org). When `ir` is omitted, the CLI searches the current directory for all `*.ir.json` files (excluding `node_modules`, `dist`, and `.next`).

```bash
manifest validate [ir] [options]
```

`ir` can be a single `.ir.json` file, a directory of IR files, or omitted to scan the current directory.

### Options

<ParamField path="--schema" type="string">
  Path to a custom JSON schema file. When omitted, the CLI uses the IR v1 schema bundled with the
  installed package automatically.
</ParamField>

<ParamField path="--strict" type="boolean" default="false">
  Alias for `--fail-on warn` (fail the process on warnings).
</ParamField>

<ParamField path="--fail-on <policy>" type="string" default="block">
  CI exit policy: `block` (errors only), `warn` (errors or warnings), or `never` (report-only).
</ParamField>

### Examples

```bash
# Validate all IR files in the ir/ directory
manifest validate ir/

# Validate a specific IR file
manifest validate ir/recipe.ir.json

# Fail on warnings as well as errors
manifest validate ir/ --strict

# Validate against a custom schema
manifest validate ir/ --schema custom-schema.json
```

<Note>
  The bundled schema path is resolved relative to the installed package, so `manifest validate`
  works correctly whether you run it from inside the Manifest repo or from a consumer project.
</Note>

## `manifest config`

Inspection surface for `manifest.config.{yaml,yml,ts,js}`. Three subcommands let you prove what Manifest will actually do before regenerating code — the cornerstone of the CI drift workflow described in [Configuration](/cli/configuration).

```bash
manifest config <subcommand> [options]
```

### `manifest config validate`

JSON-schema validates the build config against [`docs/spec/config/manifest.config.schema.json`](https://github.com/Angriff36/Manifest/blob/main/docs/spec/config/manifest.config.schema.json), the same schema referenced via `$schema:` in your config file.

```bash
manifest config validate
manifest config validate --json
```

<ParamField path="--json" type="boolean" default="false">
  Emit structured JSON (`{ configPath, ok, diagnostics }`) instead of human-readable output. Useful for CI pipelines that need machine-parseable failures.
</ParamField>

The command exits non-zero on the first schema violation and prints a dotted-path diagnostic for each error, e.g.:

```text
  - projections.nextjs.options.authProvider: must be equal to one of the allowed values (allowed: clerk, nextauth, custom, none)
  - projections.nextjs.options: unknown property "unknownOption" (additionalProperties: false)
```

### `manifest config print-defaults`

Prints the canonical defaults snapshot — every value Manifest applies when an option is unset. Sourced from `src/manifest/projections/nextjs/defaults.ts` so the projection, the schema, and this command can never disagree.

```bash
manifest config print-defaults
```

Output is key-sorted JSON across six blocks: `nextjs`, `tenantProvider`, `dispatcher`, `concreteCommandRoutes`, `readRoutes`, `routes`. See the [configuration reference](/cli/configuration) for what each key controls.

### `manifest config inspect`

Aliased as `print-effective`. Prints the **effective** configuration — defaults merged with user overrides from your config file. The output is deterministic and key-sorted, making it safe to snapshot in CI.

```bash
manifest config inspect
manifest config inspect --json
```

```json
{
  "configPath": "/work/app/manifest.config.yaml",
  "build": { "output": "ir/", "src": "**/*.manifest" },
  "projections": {
    "nextjs": {
      "output": "generated/",
      "options": {
        "appDir": "apps/api/app/api",
        "authProvider": "clerk",
        "concreteCommandRoutes": {
          "enabled": false,
          "legacyAliasesOnly": true
        },
        "dispatcher": {
          "deriveInstanceId": true,
          "enabled": true,
          "executionMode": "inline",
          "executorImportName": "executeManifestCommand",
          "executorImportPath": "@/lib/manifest-executor",
          "path": "/manifest/[entity]/commands/[command]/route.ts"
        },
        "readRoutes": {
          "directDbReads": true,
          "enabled": true
        },
        "unauthorizedStatus": 401,
        ...
      }
    }
  }
}
```

<Tip>
  Commit the inspect output as a snapshot under VCS (e.g.
  `.manifest/effective-config.snapshot.json`) and gate CI on `git diff --exit-code` against it.
  Drift in either Manifest defaults or your local config will then surface as a diff before it
  changes emitted code.
</Tip>

## `manifest watch [source]`

Watches `.manifest` files and rebuilds (compile + generate) on every change. Useful during local development so generated code stays in sync with your source.

```bash
manifest watch [source] [options]
```

### Options

`watch` accepts the same flags as `manifest build` — `-p, --projection`, `-s, --surface`, `--ir-output`, `--code-output`, `-g, --glob`, and the `--auth`/`--database`/`--runtime`/`--response` import-path flags — plus:

<ParamField path="--debounce" type="number" default="300">
  Debounce delay in milliseconds before a detected change triggers a rebuild.
</ParamField>

<ParamField path="--events" type="boolean" default="false">
  Emit structured JSON change events to stdout, useful for piping into other tooling.
</ParamField>

<ParamField path="--clear" type="boolean" default="false">
  Clear the terminal on each rebuild.
</ParamField>

<ParamField path="--all" type="boolean" default="false">
  Read all settings from `manifest.config.yaml` and use the config-driven flow instead of individual flags.
  When set, the command uses config values for compile and generate stages (output, projection, surface, etc.).
  This is the preferred workflow for projects with a config file.
</ParamField>

### Examples

```bash
# Watch a directory and rebuild on change
manifest watch src/

# Watch with a custom debounce and JSON change events
manifest watch src/ --debounce 300 --events
```

## `manifest validate-ai [source]`

Validates a `.manifest` or IR file and returns scored diagnostics tuned for AI agents, so an agent can gate on a numeric pass threshold rather than parsing free-form output.

```bash
manifest validate-ai [source] [options]
```

`source` can be a `.manifest` file, an `.ir.json` file, a directory, or a glob.

### Options

<ParamField path="-f, --format" type="string" default="text">
  Output format: `text` or `json`.
</ParamField>

<ParamField path="--schema" type="string">
  Path to a custom JSON schema file.
</ParamField>

<ParamField path="--min-score" type="number" default="100">
  Minimum score required to pass. The command exits non-zero when the score falls below this value.
</ParamField>

<ParamField path="--verbose" type="boolean" default="false">
  Include info-level diagnostics in the output.
</ParamField>

### Diagnostic categories

Every diagnostic carries a `category`: `schema`, `compile`, `semantic`, `structural`, or `domain`. The **`domain`** category surfaces compile-time domain- and reaction-wiring checks against the IR with coded diagnostics (each includes a fix suggestion):

| Code                   | Severity | Meaning                                                                                                                                                                  |
| ---------------------- | -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `DOMAIN_UNWIRED_FK`    | error    | A `{parent}Id`-style foreign key with no `belongsTo`/`ref` and no command that sets it from `self.id`.                                                                   |
| `DOMAIN_ORPHAN_CREATE` | error    | A child entity with no command or reaction that wires its parent FK.                                                                                                     |
| `REACTION_UNWIRED`     | error    | A reaction whose event is never emitted by a command, or whose payload references can't be satisfied by emitter params, `payload._subject`, or create-result properties. |

Non-error domain findings are reported as `DOMAIN_COMPLETENESS` / `REACTION_COMPLETENESS`. These checks mirror the compiler's domain-completeness semantics (see `docs/spec/semantics.md`) — a half-wired model fails validation.

### Examples

```bash
# Score an IR file as JSON, passing at 80 or above
manifest validate-ai generated.ir.json --format json --min-score 80
```

## `manifest fmt [source]`

Formats `.manifest` source files using deterministic whitespace normalization. Pairs well with a pre-commit hook to keep source style consistent.

```bash
manifest fmt [source] [options]
```

`source` can be a file, directory, or glob.

### Options

<ParamField path="--check" type="boolean" default="false">
  Fail if any file would change. When set, writing is disabled — ideal for CI.
</ParamField>

<ParamField path="--write" type="boolean">
  Write formatted output. This is the default when `--check` is not set.
</ParamField>

<ParamField path="-g, --glob" type="string">
  Glob pattern to use when `source` is a directory.
</ParamField>

### Examples

```bash
# Check formatting without writing (CI mode)
manifest fmt src/ --check

# Format and write changes in place
manifest fmt src/ --write
```

## `manifest diagram [source]`

Generates Mermaid diagrams from IR — entity-relationship, state, and sequence views — so you can visualize a program's shape.

```bash
manifest diagram [source] [options]
```

`source` can be a `.manifest`, `.ir.json`, directory, or glob.

### Options

<ParamField path="-o, --output" type="string" default="diagrams">
  Output directory for generated diagram files.
</ParamField>

<ParamField path="-t, --type" type="string" default="all">
  Diagram type: `er`, `state`, `sequence`, or `all`.
</ParamField>

<ParamField path="-e, --entity" type="string">
  Filter output to a single entity.
</ParamField>

<ParamField path="--markdown" type="boolean" default="false">
  Wrap output in fenced code blocks for embedding in Markdown.
</ParamField>

### Examples

```bash
# Generate an ER diagram wrapped in Markdown fences
manifest diagram src/app.manifest -t er --markdown
```

## `manifest coverage`

Reports command/guard/policy/constraint coverage by correlating compiled IR with conformance and unit test evidence found in the project.

```bash
manifest coverage [options]
```

### Options

<ParamField path="--ir" type="string">
  Path to a compiled IR JSON file.
</ParamField>

<ParamField path="-s, --source" type="string">
  Path to `.manifest` source, compiled on the fly.
</ParamField>

<ParamField path="-r, --root" type="string" default=".">
  Root directory to scan for test evidence.
</ParamField>

<ParamField path="-f, --format" type="string" default="text">
  Output format: `text` or `json`.
</ParamField>

<ParamField path="--min-coverage" type="number">
  Minimum coverage percentage required to pass.
</ParamField>

<ParamField path="--strict" type="boolean" default="false">
  Exit non-zero when coverage falls below `--min-coverage`.
</ParamField>

### Examples

```bash
# Fail CI if coverage drops below 80%
manifest coverage -s src/app.manifest --min-coverage 80 --strict
```

## `manifest diff`

`diff` is a parent command for comparing source against IR, comparing two IR files, and classifying breaking changes.

```bash
manifest diff <subcommand> [options]
```

### `manifest diff source-vs-ir <entityName>`

Compares source-manifest parse output against precompiled IR for a single entity.

<ParamField path="--json" type="boolean" />
<ParamField path="--src" type="string">
  Source glob pattern.
</ParamField>
<ParamField path="--ir-root" type="string">
  One or more IR root paths.
</ParamField>

### `manifest diff ir-vs-ir <oldIR> <newIR>`

Compares two IR JSON files and produces a diff report, optionally rendering migration SQL or Prisma steps.

<ParamField path="--json" type="boolean" />
<ParamField path="--sql" type="boolean">
  Render the diff as migration SQL.
</ParamField>
<ParamField path="--prisma" type="boolean">
  Render the diff as Prisma migration steps.
</ParamField>
<ParamField path="-o, --output" type="string">
  Output path.
</ParamField>

```bash
manifest diff ir-vs-ir old.ir.json new.ir.json --sql
```

## `manifest diff breaking`

Classifies the changes in an IR diff as compatible, deprecated, or breaking, with consumer impact. Run it as a `diff` subcommand. Use `--ci` to gate pipelines on unacknowledged breaking changes.

```bash
manifest diff breaking <oldIR> <newIR> [options]
```

### Options

<ParamField path="--json" type="boolean" />

<ParamField path="--ack" type="string">
  Path to an acknowledgement file listing breaking changes that have been reviewed and accepted.
</ParamField>

<ParamField path="--ci" type="boolean" default="false">
  Exit non-zero when there are unacknowledged breaking changes.
</ParamField>

<ParamField path="-o, --output" type="string">
  Output path.
</ParamField>

### Examples

```bash
# Fail CI on unacknowledged breaking changes
manifest diff breaking old.ir.json new.ir.json --ci
```

## `manifest migrate`

Analyzes an IR diff to produce a database migration plan, with optional SQL/Prisma preview and reversibility checks.

```bash
manifest migrate [options]
```

### Options

<ParamField path="--old-ir" type="string" required>
  Path to the previous IR file.
</ParamField>

<ParamField path="--new-ir" type="string" required>
  Path to the new IR file.
</ParamField>

<ParamField path="--dry-run" type="boolean" default="false">
  Show the plan without applying it.
</ParamField>

<ParamField path="--preview" type="boolean" default="false">
  Show the generated SQL and Prisma steps.
</ParamField>

<ParamField path="--force" type="boolean" default="false">
  Apply despite warnings or unacknowledged breaking changes.
</ParamField>

<ParamField path="--tool" type="string" default="prisma">
  Migration tool: `prisma` or `drizzle`.
</ParamField>

<ParamField path="--no-check-reversibility" type="boolean">
  Skip reversibility validation.
</ParamField>

<ParamField path="--json" type="boolean" />
<ParamField path="-o, --output" type="string">
  Output path.
</ParamField>

### Examples

```bash
# Preview a migration plan between two IR versions
manifest migrate --old-ir old.ir.json --new-ir new.ir.json --preview
```

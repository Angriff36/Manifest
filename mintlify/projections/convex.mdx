The Convex projection emits a complete, self-contained Convex backend from compiled Manifest IR. It started as a schema-only projection (`convex.schema`, shipped in 2.9.0) and now spans eight surfaces covering schema, reads, governed writes, schedules, webhooks, sagas, computed properties, and React clients. Everything it emits is standalone Convex code with no Manifest runtime dependency, and typechecks against `convex@1.41`. Like the [Prisma](/integration/prisma) and [Drizzle](/projections/drizzle) projections, all Convex-specific interpretation (table names, validators, references, indexes) lives in the projection and its options — never in Manifest core grammar or IR.

## What it generates

The projection registers under the name `convex` and declares eight surfaces:

| Surface            | Path hint                                      | Emits                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| ------------------ | ---------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `convex.schema`    | `convex/schema.ts` (configurable via `output`) | `defineSchema` / `defineTable` with `convex/values` validators, enum unions, `v.id` references, and indexes. Optionally a system events table.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| `convex.queries`   | `convex/queries.ts`                            | Reactive reads over `ctx.db`: `list<Entity>`, `get<Entity>` (by Convex `_id`), and `list<Entity>By<Field…>` over `.withIndex` for **every** schema index — indexed/tenant/reference fields _and_ composite `options.indexes` entries (`list<Entity>By<A>And<B>`). `list`/`get` are tenant-scoped and soft-delete-filtered by default (see [Read filtering](#read-filtering)). When the events table is emitted, adds `listRecentEvents` plus `listEventsByType` / `listEventsByEntity` / `listEventsByEntityId`.                                                                                                                                                                                                                                                                                                             |
| `convex.mutations` | `convex/mutations.ts`                          | One governed `mutation` per IR command. Runs policies → guards → constraints (runtime order, fail-closed), applies `mutate` actions, appends an event row, and fires matched reactions. Honors G7 `emit Event { field: expr }`: declared fields populate the event-row and reaction payloads from the post-action instance.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| `convex.crons`     | `convex/crons.ts`                              | IR schedules → `cronJobs()` (`crons.cron` / `crons.interval`) referencing the command mutations.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| `convex.http`      | `convex/http.ts`                               | IR webhooks → `httpRouter` / `httpAction` routes that read the request body, map `transform` params, and `ctx.runMutation` the command.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| `convex.sagas`     | `convex/sagas.ts`                              | IR sagas → orchestrator `action`s that run each step via `ctx.runMutation`, tracking completed steps and compensating in reverse on failure (`onFailure: compensate`) or rethrowing (`abort`).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| `convex.computed`  | `convex/computed.ts`                           | Computed property definitions: `isFieldComputed()` helpers and `get<Field>()` selectors for each entity's computed properties. (Shipped 3.6.17+)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| `convex.react`     | `convex/react.ts`                              | React hook wrappers: `useQuery<Entity>`, `useQuery<Entity>By<Field>`, `useMutation<Entity,Command>`. Emits only when `convex.react` is enabled via options. (Shipped 3.6.19+)                                                                                                                                                                                                                                                                                                                   |

Only persistent entities are projected. Entities marked `external`, entities with no `store` declaration, and entities whose store target is `memory` / `localStorage` are skipped; `durable` / `postgres` / `supabase` targets are emission targets. `computed` properties are never emitted as fields, and the IR `id` is dropped because Convex's built-in document `_id` is identity.

## Usage

```ts
import { compileToIR } from '@angriff36/manifest/ir-compiler';
import { getProjection } from '@angriff36/manifest/projections';

const { ir } = await compileToIR(source);

const projection = getProjection('convex');
const result = projection?.generate(ir, {
  surface: 'convex.schema',
  options: {
    referenceMode: 'convexId',
    tableMappings: { CateringEvent: 'events' },
  },
});

for (const artifact of result?.artifacts ?? []) {
  // write artifact.code at artifact.pathHint
}
```

`ConvexProjection` can also be imported directly from `@angriff36/manifest/projections/convex`. Through the CLI it is dispatched by registry name. The CLI accepts either the full surface id (`convex.schema`) or its short suffix (`schema`), and `--surface all` walks every surface in one run:

```bash
# one surface
manifest generate ir/app.ir.json --projection convex --surface convex.schema --output convex

# every surface (schema, queries, mutations, crons, http, sagas, computed, react)
manifest generate ir/app.ir.json --projection convex --surface all --output convex
```

Each artifact is written at its `pathHint` (e.g. `convex/schema.ts`, `convex/queries.ts`) relative to `--output`.

## Type mapping & behavior

The IR `type.name` string maps to a `convex/values` validator through `DEFAULT_TYPE_MAPPING` in `type-mapping.ts`:

| Manifest type                                | Convex validator | Notes                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| -------------------------------------------- | ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `string`, `text`, `uuid`                     | `v.string()`     |                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| `boolean`, `bool`                            | `v.boolean()`    |                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| `int`, `bigint`, `float`, `decimal`, `money` | `v.number()`     | Convex Float64. Matches Manifest runtime semantics, which treat every numeric type as an ordinary JS number — decimal/money precision and integer width are projection metadata, not runtime-enforced. Storing them as `v.int64()` (bigint) or `v.string()` (text) would diverge from the runtime and break generated guard/mutation arithmetic (mixed bigint/number throws; string operands concatenate or compare lexically). Opt into `v.int64()` / `v.string()` per property via `typeMappings` where lossless transport is genuinely required. |
| `date`, `datetime`, `time`, `duration`       | `v.number()`     | Epoch milliseconds, the Convex-idiomatic timestamp. Override to `v.string()` for ISO-8601.                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| `json`                                       | `v.any()`        |                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| `bytes`                                      | `v.bytes()`      |                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |

As with Prisma and Drizzle, bare `number` is intentionally absent — it is ambiguous between integers, reals, and money. The projection emits a hard `CONVEX_AMBIGUOUS_NUMBER` diagnostic and skips the field; an unknown type with no override emits a hard `CONVEX_UNKNOWN_TYPE` diagnostic. There is no silent fallback.

Validators compose: enums become `v.union(v.literal("a"), v.literal("b"))`, `array<T>` becomes `v.array(<element>)`, nullable properties union with `v.null()`, and non-required properties wrap in `v.optional(...)` (enum + nullable flatten into a single union). For `belongsTo` / `ref` relationships in the default `convexId` mode, a property that backs the relationship is retyped from its declared scalar to `v.id("<targetTable>")`, preserving its own optional/nullable. Indexes are emitted for `indexed` properties (`by_<field>`), the tenant column, and every reference, plus any consumer-supplied composite/named indexes. Referential actions (`onDelete` / `onUpdate`) emit a deferred-info diagnostic — Convex has no schema-level cascade.

## Options

The options object is `ConvexProjectionOptions` in `convex/options.ts`, normalized through `normalizeOptions`. Per-entity and per-property options use the nested `Record<EntityName, Record<PropertyName, X>>` shape — there are no dotted-string `"Entity.property"` keys anywhere on this surface.

| Option                    | Type                                               | Default                                         | Purpose                                                                                                                                                                   |
| ------------------------- | -------------------------------------------------- | ----------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `output`                  | `string`                                           | `"convex/schema.ts"`                            | Path hint for the emitted schema artifact (flows to `ProjectionArtifact.pathHint`).                                                                                       |
| `tableMappings`           | `Record<EntityName, string>`                       | `{}`                                            | Per-entity table-name override; always wins over the naming convention.                                                                                                   |
| `typeMappings`            | `Record<EntityName, Record<PropertyName, string>>` | `{}`                                            | Per-property validator override; the value is the literal Convex validator expression, e.g. `"v.number()"` or `"v.int64()"`.                                              |
| `indexes`                 | `Record<EntityName, IndexEntry[]>`                 | `{}`                                            | Per-entity composite/named indexes. Each entry is a `string[]` (composite) or `{ fields, name? }`, becoming a `.index("name", [...])` call.                               |
| `references`              | `Record<EntityName, Record<string, string>>`       | `{}`                                            | Per-relationship foreign-key field-name override (default is the non-tenant `foreignKey.fields` column, or `${relationshipName}Id`).                                      |
| `referenceMode`           | `'convexId' \| 'stringId'`                         | `'convexId'`                                    | How `belongsTo` / `ref` references are represented. `'convexId'` → `v.id("<targetTable>")`; `'stringId'` → `v.string()` for app-level string ids (UUIDs).                 |
| `naming`                  | `NamingConventionInput`                            | `{ table: 'camelCase', pluralizeTables: true }` | Table-name casing/pluralization. The Convex-idiomatic default is applied when omitted; `tableMappings` is the escape hatch for irregular plurals.                         |
| `emitEventsTable`         | `boolean`                                          | `true`                                          | Emit a system events table (the reactive event log governed mutations append to).                                                                                         |
| `eventsTable`             | `string`                                           | `"manifestEvents"`                              | Name of the system events table. If the resolved name collides with an entity table it is deterministically suffixed (with a `CONVEX_EVENTS_TABLE_COLLISION` diagnostic). |
| `policyMode`              | `'enforce' \| 'skip'`                              | `'enforce'`                                     | Authorization-policy enforcement in generated mutations — see below.                                                                                                      |
| `includeTenantFilter`     | `boolean`                                          | `true`                                          | Scope `list<Entity>` / `get<Entity>` reads to the current tenant. Field-aware — see [Read filtering](#read-filtering).                                                    |
| `includeSoftDeleteFilter` | `boolean`                                          | `true`                                          | Exclude soft-deleted rows from generated reads. Field-aware.                                                                                                              |
| `tenantIdProperty`        | `string`                                           | IR `tenant.property`                            | Override the tenant property name used for read scoping.                                                                                                                  |
| `deletedAtProperty`       | `string`                                           | `"deletedAt"`                                   | Soft-delete property name; a read drops rows whose value is non-null.                                                                                                     |
| `authContextImport`       | `string`                                           | unset                                           | Module specifier of an author-owned auth-context module — see [Auth context seam](#auth-context-seam).                                                                    |

### Auth context seam (`authContextImport`)

By default (unset), generated functions read identity from an inline `(ctx as any).auth` bag — a shape the real Convex runtime **never populates** (`ctx.auth` only exposes `getUserIdentity()`), so role/tenant reads evaluate against `undefined` and fail closed until you wire auth by hand. Setting `authContextImport` (e.g. `"./lib/authContext"`) routes all identity reads through your module's exported `getAuthContext(ctx)` — which should derive role and tenant from the real authenticated identity (e.g. a Clerk/Auth0 JWT via `ctx.auth.getUserIdentity()`).

With the seam active, two write-side protections also turn on for tenant-scoped entities:

- **`create` mutations derive the tenant server-side.** The tenant column is filled from the auth context and removed from the mutation's client arguments (a create action targeting it is overridden, with a `CONVEX_TENANT_SERVER_DERIVED` diagnostic).
- **Instance mutations verify document tenancy.** A command against a document from another tenant throws the same "not found" error as a missing document (no cross-tenant existence oracle).

Contract: `getAuthContext(ctx)` returns (or resolves to) an object carrying at least `role` and the tenant property. Crons, sagas, and webhooks invoke the same governed mutations, so the module must also handle identity-less system calls (e.g. return a system context).

### Read filtering

Convex defaults `includeTenantFilter` and
`includeSoftDeleteFilter` to **`true`**. Next.js defaults both to **`false`** — they do **not**
mirror each other. Both Convex filters are _field-aware_: a clause is emitted only for entities
that actually declare the column, so entities that don't follow the tenancy / soft-delete
conventions get plain unfiltered reads.

<Note>
The tenant id for `list<Entity>` and `get<Entity>` is derived from the authenticated identity (via `getAuthContext(ctx)` when `authContextImport` is set, otherwise the legacy inline `ctx.auth.<tenantProp>` read), **never from a client argument** — so an un-scoped `list<Entity>()` cannot return rows across tenants (it fails closed: with no auth wired, it matches no tenant and returns nothing rather than everything). `get<Entity>` returns `null` on a tenant mismatch or a soft-deleted row. `list<Entity>By<Field…>` reads additionally drop soft-deleted rows and apply the auth-derived tenant filter unless the index already constrains the tenant column. Turn either filter off with `includeTenantFilter` / `includeSoftDeleteFilter`.
</Note>

### policyMode

`policyMode` (added in 2.10.x) controls whether generated mutations enforce authorization policies. Its accepted values are exactly `'enforce'` and `'skip'`, defaulting to `'enforce'`.

<Note>
  With `policyMode: 'skip'`, the role/policy (authorization) checks are omitted from generated
  mutations, while guards and constraints (state validation) are still enforced. When skipped, the
  `ROLE_PERMISSIONS` map and `checkRole` helper are not emitted, so there is no dead code. This is
  intended for dev/demo backends that have no auth context configured, so the generated files don't
  need hand-editing. Production builds should keep the default `'enforce'`.
</Note>

## Notes & limitations

<Note>
Mutation governance is rendered **fail-closed**: each policy, guard, and constraint is rendered from IR via a pure expression resolver, and any expression the resolver cannot fully map emits a hard `CONVEX_UNRESOLVED_{POLICY,GUARD,CONSTRAINT}` diagnostic together with a denying `throw` — never a silent pass. Roles become a `ROLE_PERMISSIONS` map plus a `checkRole()` helper (with an `all` wildcard). Reactions that create tenant-scoped targets automatically thread the source `tenantId` (2.10.3) unless an explicit reaction param sets it.
</Note>

<Note>
  The `convex.http` surface generates `httpRouter` / `httpAction` webhook routes that read the
  request body and invoke the command mutation, but does **not** verify `IRWebhook.signature` (HMAC
  sha256/sha512) — generated Convex `httpAction`s have no signature verification. It also does not
  implement `idempotencyHeader` deduplication: duplicate webhook deliveries re-run the command
  mutation. For verified, deduplicated webhook handling, use the reference webhook handler
  (`@angriff36/manifest/webhooks`) with a Next.js, Express, or Hono projection instead.
</Note>

### G7 emit payloads (`emit Event { field: expr }`)

A command may declare explicit event-payload fields (G7). The projection evaluates each declared field against the **post-action instance** and populates both the emitted event row's `payload` and the shared reaction `payload` with them — mirroring the reference runtime (`runtime-engine.ts` emit-time payload construction). This matters for reactions on **mutate** commands: the runtime's base event payload is `{ ...input, result }`, where `input` is only the command's parameters, so an entity-owned field the reaction needs (`payload.invoiceId`, `payload.amount`) or a computed field (`payload.total = self.qty * self.price`) is `undefined` unless declared. Declaring it via G7 makes the reaction resolve instead of silently no-op'ing — which is what lets a cross-entity cascade (e.g. "when a payment is processed, apply it to its invoice") be authored as a reaction rather than hand-written as after-emit middleware. Create commands evaluate against the built `doc`; non-create commands introduce a post-patch `{ ...doc, ...updates }` instance. It is strictly opt-in: commands with no `emitPayloads` generate byte-identical output. Unresolved field expressions emit a `CONVEX_UNRESOLVED_EMIT_PAYLOAD` warning and are omitted (non-governance, so no denying throw).

This projection's behavior is taken directly from `src/manifest/projections/convex/generator.ts`, `options.ts`, `type-mapping.ts`, `functions.ts`, and `orchestration.ts`. The schema surface shipped in 2.9.0; the five functions/orchestration surfaces, `listRecentEvents`, the create-mutation completeness fix, and `policyMode` landed across the 2.10.x releases. Schema↔query index parity (every schema index — single-field, reference FK in both reference modes, and composite — has a matching read), the events-table indexed reads, and default tenant + soft-delete read filtering landed in a later 2.10.x release.

## Rate limiting

Command- and policy-level `rateLimit` is implemented via a sliding-window bucket table. When any command or write/execute/delete policy declares `rateLimit`, the projection emits:

**Schema table** (`commandRateLimitBuckets` by default):

```ts
commandRateLimitBuckets: defineTable({
  scopeKey: v.string(), // "user:user-123", "tenant:org-abc", or "global"
  timestamps: v.array(v.number()), // Window timestamps
  windowStart: v.number(), // When the current window began
})
  .index("by_scopeKey", ["scopeKey"])
```

**Helper function** (`__consumeCommandRateLimit`):

- Checks existing bucket for the scope key
- Filters timestamps outside the rolling window
- Throws if `timestamps.length >= maxRequests + burstAllowance`
- Appends current timestamp and updates/inserts bucket
- Returns `retryAfterMs` on denial

The check runs **before policies/guards** inside each mutation (commands) or **before each policy expression** (policies), so idempotent replays that skip the runner do not consume additional tokens.

**Scope key prefixes**:
- Command rateLimit: `cmd:<entity>:<command>:<scope>`
- Policy rateLimit: `policy:<policyName>:<scope>`

User/tenant-scoped rate limits require `authContextImport` — the scope key derives from the authenticated identity (`getAuthContext(ctx)`). Without auth, these scopes fail closed.

<Note>
  Read/`all` policy `rateLimit` is **not supported** — Convex queries cannot mutate durable buckets. Declaring rateLimit on a read policy emits `CONVEX_UNSUPPORTED_READ_POLICY_RATE_LIMIT` (error diagnostic).
</Note>

## Unsupported features (REJECTED_LOUD)

The following Manifest language features are **not supported** in the Convex projection and emit compile-time error diagnostics (`CONVEX_UNSUPPORTED_*`):

| Feature | Diagnostic | Why it's unsupported |
|---------|-----------|---------------------|
| **Command retry** | `CONVEX_UNSUPPORTED_RETRY` | Convex mutations cannot honor per-attempt rollback + backoff sleep. Use platform optimistic concurrency (`versionProperty`) or caller-side retry instead. |
| **Entity approvals** | `CONVEX_UNSUPPORTED_APPROVAL` | No stage-state tracking or pre-command gate in Convex mutations. Approval workflows must be implemented at the application level. |
| **Read/`all` policy rateLimit** | `CONVEX_UNSUPPORTED_READ_POLICY_RATE_LIMIT` | Convex queries (`ctx.db.query`) cannot mutate durable buckets. Rate limiting is only supported on commands and write/execute/delete policies. |
| **Action kinds: `effect` / `publish` / `persist`** | `CONVEX_UNSUPPORTED_ACTION_KIND` | No mutation lowering for adapter actions. These require runtime adapters (`outboxStore`, `effectHandler`) not available in Convex. |
| **`async` commands** | `CONVEX_UNSUPPORTED_ASYNC_COMMAND` | No job queue or worker drain mechanism in Convex projection. |

These are **loud rejections** — the projection emits a hard diagnostic and skips the feature rather than silently ignoring it. This prevents "parsed but ignored" surprises and forces you to either remove the unsupported feature or restructure your model.

<Note>
  Partial capabilities exist for some features:
  - **Realtime**: Convex queries are already reactive via `useQuery` — no SSE artifact needed (unlike Next.js). This is `CONVEX_PARTIAL_REALTIME` (info diagnostic).
  - **Computed cache**: Computed helpers stay pure; Manifest's `request`/`session`/`ttl` cache strategies are not lowered. This is `CONVEX_PARTIAL_COMPUTED_CACHE` (info diagnostic).
</Note>

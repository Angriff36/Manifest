The **wiring** projection emits a deterministic product-wiring contract and TypeScript binding helpers from compiled IR. Use it so applications and agents connect real UI to Manifest commands without inventing types, lifecycle states, or client-side identity fields.

> **Verified (2026-07-15) @RYANSIGNED:** Registry name `wiring`; CLI
> `manifest generate -p wiring --surface all`, plus `manifest wiring-inspect` /
> `manifest wiring-remediate`. Package pin **3.6.4**.

This is **not** a UI generator. It does not create pages, forms, buttons, layouts, or copy. Your app still owns presentation.

## Generate

```bash
manifest generate -p wiring --surface all
```

Default artifacts:

- `src/generated/manifest-wiring-contract.json`
- `src/generated/manifest-wiring-bindings.ts`

## What the contract answers

For every entity command:

- Exact parameter types (including `string[]` vs `string`)
- Required vs optional
- Enum / finite values when statically known
- Numeric and non-empty constraints when statically known
- Which fields are **server-owned** (`from context.*`) vs client-owned
- Dispatcher route identity
- Proven lifecycle transitions (from `transition` + literal `mutate`)
- List/detail invalidation hints after success
- Structured error states to handle

Unprovable metadata is omitted — never guessed.

## Trusted server-owned inputs

```manifest
command markCompleted(completedBy: string from context.actorId) {
  mutate completedBy = completedBy
}
```

- Client input types omit `completedBy`
- Runtime strips spoofed client values and injects `context.actorId`
- Missing trusted context fails with `MISSING_TRUSTED_CONTEXT`

Ownership is declared explicitly. Naming heuristics are not used.

## Automatic consumer inspection

Manifest does **not** design the interface. It proves whether application code correctly consumes declared capabilities.

```bash
manifest wiring-inspect \
  --contract src/generated/manifest-wiring-contract.json \
  --root apps/app \
  --root apps/api \
  --strict
```

Primary coverage comes from source inspection (generated client calls, `executeCommand`, server actions → `runManifestCommand`, API route chains, imported helpers). Explicit registries are overrides for `backend-only` / `deferred` / accepted ambiguous cases.

Default CI fails on stale consumers and proven contract mismatches. Unwired capabilities fail only with `--strict-coverage`. Ambiguous findings never fail by default.

Repo-wide scans are supported: Capsule-Pro-scale roots complete in a few seconds with indexed import resolution and parallel loading. Include filters are optional for debugging, not required for the gate.

See the full reference in the repo: `docs/projections/wiring.md`.

## Automatic remediation

Manifest does **not** design the UI. When the correct integration is derivable from Manifest truth and an existing product surface, Manifest plans and applies a deterministic repair:

```bash
manifest wiring-remediate --contract src/generated/manifest-wiring-contract.json \
  --root apps/app --root apps/api --mode one-defect
```

Modes: `plan`, `dry-run`, `apply`, `one-defect`. Auto-fixable findings are patched via AST-aware edits and re-inspected. Ambiguous product placement is never auto-edited. `add-required-input` applies only when a unique proven in-scope value source exists (never invented defaults). Failed verification discards the patch.

## Boundary

| Manifest                                     | Application                                    |
| -------------------------------------------- | ---------------------------------------------- |
| Command truth, types, ownership, constraints | Layout, visuals, labels                        |
| Safe bindings + invalidation metadata        | Button vs menu vs gesture                      |
| Coverage truth + deterministic wiring repair | Explicit consumer registry / product placement |

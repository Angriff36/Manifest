The Remix projection generates route modules from compiled Manifest IR that follow Remix conventions: `loader` functions for entity reads, `action` functions for command dispatch, and TypeScript type definitions for your data. Use it when you are building a Remix or React Router v7 application and want Manifest entities wired directly into the data layer of your routes. The projection mirrors the [Express projection's](/projections/express) handler structure but targets Remix's `loader`/`action` exports instead of Express middleware.

The projection is registered under the name `remix` and lives at `src/manifest/projections/remix/generator.ts`.

## What it generates

The projection exposes five surfaces.

`remix.list` emits a route module with a `loader` function that queries entities with optional auth checks, tenant filtering, and soft-delete filtering. The generated code uses Prisma-style queries for list reads and returns data via Remix's `json` helper.

`remix.detail` emits a route module with a `loader` function that loads a single entity instance by ID. The route parameter is extracted from Remix's `params` object.

`remix.command` emits a route module with an `action` function that dispatches a command through the Manifest runtime via `runtime.runCommand`. The handler extracts `instanceId` from the request body, parses command parameters, and returns structured responses using `manifestSuccessResponse` and `manifestErrorResponse` helpers.

`remix.types` emits TypeScript interfaces for all entities and command parameter types. Computed properties are emitted as `readonly` fields. Each entity gets one `interface`, and each parameterized command gets a `<Entity><Command>Params` interface.

`remix.client` emits response helper utilities: `manifestSuccessResponse`, `manifestErrorResponse`, and `normalizeCommandResult`. These map Manifest `CommandResult` outcomes to Remix `Response` objects with proper HTTP status codes.

Command handlers map runtime error codes to HTTP status: `GUARD_FAILED` to 403, `CONSTRAINT_VIOLATION` to 422, `CONCURRENCY_CONFLICT` to 409, `NOT_FOUND` to 404, and anything else to 500. Optional error boundary exports are available when the `errorBoundaries` option is enabled.

## Usage

```ts
import { getProjection } from '@angriff36/manifest/projections';

const projection = getProjection('remix');

// Generate all list routes
const listRoutes = projection.generate(ir, { surface: 'remix.list' });

// Generate a command route for a specific entity
const commandRoute = projection.generate(ir, {
  surface: 'remix.command',
  entity: 'Order',
});

// Generate TypeScript types
const types = projection.generate(ir, { surface: 'remix.types' });
```

Through the CLI:

```bash
manifest generate ir/widget.ir.json --projection remix --surface remix.types --output src/types/manifest.ts
```

## Auth integration

The projection supports four auth providers, configured via the `authProvider` option:

- **`clerk`** (default) — Imports from `@clerk/remix`, uses `getAuth()` to retrieve the current user session.
- **`remix-auth`** — Imports from the configured `authImportPath`, uses ` authenticator.isAuthenticated()` for session checks.
- **`custom`** — Imports a `requireUser(request)` function from the configured `authImportPath`.
- **`none`** — No auth checks in generated code; suitable for public APIs or when auth is handled at a higher layer.

Unauthorized users receive a redirect to the configured `unauthorizedRedirect` path (default `/login`) with the appropriate HTTP status code from the `unauthorizedStatus` option.

## Generated file structure

For an entity named `Order` with a `placeOrder` command, the projection emits:

```
app/routes/
  orders/
    index.tsx          # remix.list — GET /orders
    $id.tsx            # remix.detail — GET /orders/:id
    place-order.tsx    # remix.command — POST /orders/place-order
app/types/
  manifest.ts          # remix.types — entity & command interfaces
app/utils/
  manifest-client.ts   # remix.client — response helpers
```

Route segment names are derived from the lower-cased entity name and kebab-cased command names, following Remix file-based routing conventions.

## Type mapping

The projection maps Manifest IR types to TypeScript: `string` to `string`; `number`/`integer`/`int` to `number`; `boolean`/`bool` to `boolean`; `date`/`datetime`/`uuid`/`email`/`url`/`uri`/`decimal` to `string`; `any` to `unknown`; `object` to `Record<string, unknown>`. Generic `array<T>` becomes `T[]` and `map<V>` becomes `Record<string, V>`. Nullable types append ` | null`.

## Options

`RemixProjectionOptions` accepts `authProvider` (`'clerk'` | `'remix-auth'` | `'custom'` | `'none'`, default `'clerk'`), `authImportPath` (default varies by provider), `runtimeImportPath` (default `./lib/manifest-runtime`), `runtimeFactoryName` (default `createManifestRuntime`), `basePath` (default `/api`), `remixVersion` (`'v2'` | `'v7'`, default `'v2'`), `includeTenantContext` (default `true`), `tenantIdProperty` (default `tenantId`), `errorBoundaries` (default `false`), `unauthorizedStatus` (default `401`), `unauthorizedRedirect` (default `/login`), `includeSoftDeleteFilter` (default `false`), `publicReads` (default `false`), and `includeComments` (default `true`).

## Notes & limitations

<Note>
  The generated modules import auth and runtime factory from the configured paths; they do not
  define those modules. Read routes (`list`/`detail`) bypass the runtime engine and call read
  methods directly. All writes go through `runtime.runCommand`, preserving guard, policy, and
  constraint enforcement. The `basePath` option is normalized but the entity route paths in
  generated handlers are not prefixed with it — apply the prefix where you mount the routes.
</Note>

The projection supports both Remix v2 and React Router v7 file-based routing conventions, controlled by the `remixVersion` option. Tenant filtering is optional and activated via `includeTenantContext`. Soft-delete filtering (adding `deletedAt: null` to read queries) is available through `includeSoftDeleteFilter`.

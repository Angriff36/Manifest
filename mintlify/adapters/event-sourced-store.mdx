The `EventSourcedStore` is an in-memory `Store<T>` implementation that persists entity state as an append-only event log. Each `create`/`update`/`delete` operation appends an event to the log, and the current state is maintained by replaying events from the last snapshot.

This store **ships** in the reference runtime (`@angriff36/manifest/runtime-engine`) and is auto-wired for any entity declared with `store Entity in eventSourced { ... }`.

## How it works

1. **Events append to a log** — Each write operation creates an event (`create`/`update`/`delete`) with a sequence number and timestamp.
2. **State is projected** — The current entity state is maintained in memory by applying all events from the log.
3. **Snapshots** — Periodically (default: every 100 events), the current state is snapshotted. Replay starts from the last snapshot instead of the beginning.
4. **Optional event log access** — When `exposeEventLog: true`, `getEventLog(id)` returns the full append-only log for an entity.

## Declaration

```manifest
store Order in eventSourced {
  snapshotInterval: 100    // Take a snapshot every N events (default: 100)
  exposeEventLog: true     // Allow getEventLog() to return the event log (default: false)
}
```

## Usage in TypeScript

The `EventSourcedStore` class is exported from `@angriff36/manifest/stores/event-sourced`. You rarely instantiate it directly — the runtime auto-wires it based on the IR store declaration. For custom instantiation:

```typescript
import { EventSourcedStore } from '@angriff36/manifest/stores/event-sourced';

const store = new EventSourcedStore<Order>({
  snapshotInterval: 100,
  exposeEventLog: true,
  generateId: () => crypto.randomUUID(),
  now: () => Date.now(),
});

// Standard Store interface
await store.create({ id: 'order-1', total: 100 });
await store.update('order-1', { total: 150 });
await store.delete('order-1');
const order = await store.getById('order-1');
const all = await store.getAll();

// Event-sourced specific methods
const events = store.getEventLog('order-1');  // EventSourcedEvent[] | undefined
const snapshot = store.getSnapshot('order-1'); // EventSourcedSnapshot<Order> | undefined
```

## Event structure

Each event in the log has:

| Field    | Type                               | Description                                  |
| -------- | ---------------------------------- | -------------------------------------------- |
| sequence | number                             | Monotonically increasing event number      |
| kind     | `'create'` `'update'` `'delete'`     | Operation type                               |
| at       | number                             | Timestamp (milliseconds since epoch)        |
| payload  | `Record<string, unknown>` \| undefined | For `update` events, the changed fields |

## Snapshot structure

| Field    | Type   | Description                             |
| -------- | ------ | --------------------------------------- |
| sequence | number | Sequence number at snapshot time        |
| state    | T      | Complete entity state at that sequence |

## Limitations

- **In-memory only** — The event log and projected state live in process memory and are lost on restart. For durable event sourcing, use the outbox/audit sinks or build a custom adapter.
- **No distributed replay** — Each process maintains its own log; for multi-process scenarios, use an external event store.

The DynamoDB store adapter persists Manifest entity state in AWS DynamoDB using the `Store<T>` interface. Use it when your deployment target is AWS and you need a fully managed, low-latency key-value store. The adapter supports single-table design patterns, composite primary keys, optimistic locking, and a dedicated transactional outbox store for durable event delivery.

## Configuration

`DynamoDBStore` is exported from `@angriff36/manifest/stores` (Node.js only). It requires the `@aws-sdk/client-dynamodb` and `@aws-sdk/lib-dynamodb` packages as optional peer dependencies, loaded via dynamic import.

```typescript
import { DynamoDBStore } from '@angriff36/manifest/stores';

const store = new DynamoDBStore('Order', {
  tableName: 'manifest-orders',
  region: 'us-east-1',
  partitionKey: 'pk',
  sortKey: 'sk',
  entityPrefix: 'ORDER', // single-table design prefix
});
```

The store declares itself in Manifest source as:

```manifest
entity Order {
  property required id: string
  property total: decimal
  property status: string
}

store Order in dynamodb
```

## Single-table design

DynamoDBStore supports the single-table design pattern through the `entityPrefix` option. When configured, all entities share a single DynamoDB table and are distinguished by their partition key prefix. The `buildDynamoDBKey()` helper constructs composite keys:

```typescript
import { buildDynamoDBKey } from '@angriff36/manifest/stores';

// Composite key: "ORDER#123" / "ORDER#123#DETAIL"
const key = buildDynamoDBKey('order-123', { entityPrefix: 'ORDER' }, 'Order');
```

When `entityPrefix` is set, the store uses `ConditionExpression` for atomic create (`attribute_not_exists`) and delete (`attribute_exists`) operations. Single-table design is auto-detected when `config.singleTable === true` in any store declaration.

## Composite key handling

The store maps Manifest entity IDs to DynamoDB composite primary keys. Partition and sort key names are configurable via `partitionKey` (default `pk`) and `sortKey` (default `sk`) in the store config. Entity instances are serialized as JSON strings and stored as DynamoDB attribute values.

## Optimistic locking

DynamoDBStore does NOT implement optimistic locking ‚Äî the `update` method is a plain `PutCommand` with no version check. For optimistic locking with version field enforcement, use **MongoDBStore** (`@angriff36/manifest/stores`), which includes a version check in its update filter. DynamoDB atomic create uses `attribute_not_exists` and delete uses `attribute_exists` for existence checks, but there is no version-based concurrency control.

## DynamoDBOutboxStore

A `DynamoDBOutboxStore` exists in the source tree ‚Äî it implements the `OutboxStore` interface using DynamoDB's `TransactWriteItems` for atomic batch enqueue (up to 100 actions), conditional `UpdateItem` for concurrent claim, and `markDelivered`/`markFailed` for state transitions ‚Äî but it is **not currently a public export**. The `@angriff36/manifest/outbox/dynamodb` subpath does not resolve, and the store has no test coverage yet, so it is not published on the `exports` map.

For durable at-least-once event delivery on AWS today, use one of the **exported** outbox stores: `PostgresOutboxStore` (`@angriff36/manifest/outbox/postgres`) or `RedisOutboxStore` (`@angriff36/manifest/outbox/redis`).

<Note>
  Unlike the doc's older example, the source `DynamoDBOutboxStore` constructor takes a single
  options object whose `client` field is **required** ‚Äî it does not accept a `region`. The `client`
  is an initialized AWS SDK document client that also carries the command constructors
  (`TransactWriteCommand`, `ScanCommand`, `UpdateCommand`), matching the convention used by
  `DynamoDBStore`. This shape is documented here for reference only; the store is not importable
  from the package until it is exported.
</Note>

## Infrastructure projections

The DynamoDB adapter includes a dedicated projection that generates infrastructure definitions from Manifest IR. The `DynamoDBProjection` is registered under the name `dynamodb` and emits three surfaces:

- `dynamodb.cloudformation` ‚Äî CloudFormation YAML template with table definitions, key schema, attribute definitions, billing mode, and stream specification.
- `dynamodb.cdk` ‚Äî AWS CDK TypeScript code with `dynamodb.Table` constructs.
- `dynamodb.terraform` ‚Äî Terraform HCL with `aws_dynamodb_table` resources.

All three surfaces support per-entity and single-table designs, and always emit an outbox table.

```ts
import { getProjection } from '@angriff36/manifest/projections';

const projection = getProjection('dynamodb');
const cfn = projection.generate(ir, { surface: 'dynamodb.cloudformation' });
```

## Wiring in DynamoDB

```typescript
import { RuntimeEngine } from '@angriff36/manifest';
import { DynamoDBStore } from '@angriff36/manifest/stores';
import { PostgresOutboxStore } from '@angriff36/manifest/outbox/postgres';
import { Pool } from 'pg';

const runtime = new RuntimeEngine(
  ir,
  { actorId: session.userId, tenantId: session.tenantId },
  {
    storeProvider: (entityName) =>
      entityName === 'Order'
        ? new DynamoDBStore('Order', { tableName: 'manifest-data', region: 'us-east-1' })
        : undefined,
    // The DynamoDB outbox store is not yet an exported subpath; use an exported
    // outbox store (Postgres or Redis) for durable delivery today.
    outboxStore: new PostgresOutboxStore(new Pool({ connectionString: process.env.DATABASE_URL })),
  },
);
```

## Notes & limitations

<Note>
  The `@aws-sdk/client-dynamodb` and `@aws-sdk/lib-dynamodb` packages are optional peer dependencies
  loaded via dynamic import. If they are not installed, the constructor throws a clear error:
  "DynamoDBStore requires '@aws-sdk/client-dynamodb' and '@aws-sdk/lib-dynamodb' to be installed."
  The `clear()` method uses Scan + BatchWrite in groups of 25 (DynamoDB batch limit).
</Note>

The Redis store adapter persists Manifest entity state in Redis using the `Store<T>` interface. Use it when you need high-throughput, low-latency entity storage with built-in TTL management and native pub/sub for event propagation. The adapter loads the `ioredis` library lazily as an optional dependency.

## Configuration

`RedisStore` is exported from `@angriff36/manifest/stores` (Node.js only). The constructor takes the **entity name** as its first argument, followed by a `RedisConfig` object.

```typescript
import { RedisStore } from '@angriff36/manifest/stores';

const store = new RedisStore<User>('User', {
  url: 'redis://localhost:6379',
  keyPrefix: 'manifest:', // defaults to `<entityName>:` lowercased
  defaultTTL: 3600, // 1 hour in seconds (optional)
});
```

`RedisConfig` exposes exactly three optional fields: `url`, `keyPrefix`, and `defaultTTL`. When `keyPrefix` is omitted it defaults to the lowercased entity name followed by a colon (for example, `user:`).

The store declares itself in Manifest source as:

```manifest
entity Session {
  property required id: string
  property userId: string
  property data: string
}

store Session in redis
```

## Key naming conventions

Each entity instance is stored as its own Redis string key holding a JSON-serialized value. The key pattern is `<keyPrefix><id>`. With `keyPrefix: 'manifest:'`, an entity with ID `user-123` is stored at key `manifest:user-123`. Collection reads use `KEYS <keyPrefix>*` followed by `MGET`.

## Key-based JSON storage

`RedisStore` implements the `Store<T>` interface using Redis string commands:

- `getAll()` runs `KEYS <keyPrefix>*` then `MGET`, deserializing each JSON value.
- `getById(id)` runs `GET` on `<keyPrefix><id>`.
- `create(data)` assigns an ID (or uses `data.id`) and runs `SET` with the serialized entity, then `EXPIRE` when `defaultTTL` is set.
- `update(id, data)` reads the existing value, merges the partial data, and `SET`s it back.
- `delete(id)` runs `DEL` on the key.
- `clear()` runs `KEYS <keyPrefix>*` then `DEL` on the matched keys.

## TTL management

`RedisStore` supports automatic entity expiration through Redis TTL:

```typescript
// Set TTL at store construction (applies to newly created keys)
const store = new RedisStore('User', { url: 'redis://localhost:6379', defaultTTL: 3600 });

// Manage TTL at runtime (operates across the store's keyPrefix namespace)
await store.setTTL(7200); // Set a 2-hour TTL on this store's keys
await store.setTTL(undefined); // PERSIST â€” remove TTL
const ttl = await store.getTTL(); // Remaining TTL in seconds for the namespace
```

When `defaultTTL` is configured, newly created entity keys receive the TTL automatically via `EXPIRE`.

## Pub/Sub â€” use `RedisEventBus`, not `RedisStore`

`RedisStore.publishEvent` and `RedisStore.subscribe`
were silent no-ops and now **throw**, pointing callers at `RedisEventBus`. Cross-process event
fan-out is `RuntimeOptions.eventBus` + `RedisEventBus` from `@angriff36/manifest/events/redis`
(see below). There is no auto-construct from env â€” wire the bus yourself.

## `RedisEventBus` (cross-process fan-out)

```typescript
import { RuntimeEngine } from '@angriff36/manifest';
import { RedisEventBus } from '@angriff36/manifest/events/redis';

const eventBus = new RedisEventBus({
  url: process.env.REDIS_URL, // default redis://localhost:6379
  channel: 'manifest:events', // default
});

const runtime = new RuntimeEngine(ir, context, { eventBus });
await runtime.connectEventBus(); // subscribe this process to remote envelopes
```

`RedisEventBus` implements the same `EventBus` contract as `MemoryEventBus`
(`@angriff36/manifest/events`). Optional peer: `ioredis`. Inject `client` in tests to avoid loading
ioredis. Call `close()` to quit bus-owned connections.

## Outbox delivery

`RedisOutboxStore` is a first-party Redis Streams implementation of the `OutboxStore` contract, exported from `@angriff36/manifest/outbox/redis`. It uses Redis Streams (`XADD` / `XREADGROUP` / `XACK`) with a consumer group so multiple dispatcher workers can drain the outbox in parallel.

```typescript
import { RedisOutboxStore } from '@angriff36/manifest/outbox/redis';

const outboxStore = new RedisOutboxStore({
  url: process.env.REDIS_URL,
  keyPrefix: 'manifest:outbox:', // default
  consumerGroup: 'manifest-dispatcher', // default
});
```

Like `RedisStore`, it loads `ioredis` lazily as an optional peer dependency and throws a clear error if the package is not installed. The `enqueue` method accepts an optional Redis pipeline for batching event persistence alongside state mutations.

The package `exports` map exposes `./outbox` (the `OutboxStore` / `OutboxEntry` contract) plus the concrete stores:

| Outbox store          | Import path                           |
| --------------------- | ------------------------------------- |
| `MemoryOutboxStore`   | `@angriff36/manifest/outbox/memory`   |
| `PostgresOutboxStore` | `@angriff36/manifest/outbox/postgres` |
| `RedisOutboxStore`    | `@angriff36/manifest/outbox/redis`    |

## Wiring in Redis

```typescript
import { RuntimeEngine } from '@angriff36/manifest';
import { RedisStore } from '@angriff36/manifest/stores';
import { RedisOutboxStore } from '@angriff36/manifest/outbox/redis';

const runtime = new RuntimeEngine(
  ir,
  { actorId: session.userId, tenantId: session.tenantId },
  {
    storeProvider: (entityName) =>
      entityName === 'Session'
        ? new RedisStore(entityName, { url: process.env.REDIS_URL, defaultTTL: 3600 })
        : undefined,
    // Durable at-least-once event delivery via the Redis Streams outbox:
    outboxStore: new RedisOutboxStore({ url: process.env.REDIS_URL }),
  },
);
```

## Notes & limitations

<Note>
  `ioredis` is loaded lazily and is optional for `RedisStore`, `RedisOutboxStore`, and
  `RedisEventBus`. If it is not installed, those adapters throw a clear install hint. Prefer TLS
  (`rediss://`) in production. Package pin for the main SDK: `@angriff36/manifest@3.6.41`.
</Note>

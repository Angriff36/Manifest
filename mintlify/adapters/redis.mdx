The Redis store adapter persists Manifest entity state in Redis using the `Store<T>` interface. Use it when you need high-throughput, low-latency entity storage with built-in TTL management and native pub/sub for event propagation. The adapter loads the `ioredis` library lazily as an optional dependency.

## Configuration

`RedisStore` is exported from `@angriff36/manifest/stores` (Node.js only). The constructor takes the **entity name** as its first argument, followed by a `RedisConfig` object.

```typescript
import { RedisStore } from '@angriff36/manifest/stores';

const store = new RedisStore<User>('User', {
  url: 'redis://localhost:6379',
  keyPrefix: 'manifest:',  // defaults to `<entityName>:` lowercased
  defaultTTL: 3600,        // 1 hour in seconds (optional)
});
```

`RedisConfig` exposes exactly three optional fields: `url`, `keyPrefix`, and `defaultTTL`. When `keyPrefix` is omitted it defaults to the lowercased entity name followed by a colon (for example, `user:`).

The store declares itself in Manifest source as:

```manifest
entity Session {
  property required id: string
  property userId: string
  property data: string
}

store Session in redis
```

## Key naming conventions

Each entity instance is stored as its own Redis string key holding a JSON-serialized value. The key pattern is `<keyPrefix><id>`. With `keyPrefix: 'manifest:'`, an entity with ID `user-123` is stored at key `manifest:user-123`. Collection reads use `KEYS <keyPrefix>*` followed by `MGET`.

## Key-based JSON storage

`RedisStore` implements the `Store<T>` interface using Redis string commands:

- `getAll()` runs `KEYS <keyPrefix>*` then `MGET`, deserializing each JSON value.
- `getById(id)` runs `GET` on `<keyPrefix><id>`.
- `create(data)` assigns an ID (or uses `data.id`) and runs `SET` with the serialized entity, then `EXPIRE` when `defaultTTL` is set.
- `update(id, data)` reads the existing value, merges the partial data, and `SET`s it back.
- `delete(id)` runs `DEL` on the key.
- `clear()` runs `KEYS <keyPrefix>*` then `DEL` on the matched keys.

## TTL management

`RedisStore` supports automatic entity expiration through Redis TTL:

```typescript
// Set TTL at store construction (applies to newly created keys)
const store = new RedisStore('User', { url: 'redis://localhost:6379', defaultTTL: 3600 });

// Manage TTL at runtime (operates across the store's keyPrefix namespace)
await store.setTTL(7200);        // Set a 2-hour TTL on this store's keys
await store.setTTL(undefined);   // PERSIST — remove TTL
const ttl = await store.getTTL(); // Remaining TTL in seconds for the namespace
```

When `defaultTTL` is configured, newly created entity keys receive the TTL automatically via `EXPIRE`.

## Pub/Sub event propagation

`RedisStore` includes pub/sub helpers for event propagation:

```typescript
// Publish an event to a Redis channel
await store.publishEvent('OrderCreated', { orderId: 'order-123', total: 99.99 });

// Register a local subscriber callback for a channel
await store.subscribe('OrderCreated', (event) => {
  console.log('Received event:', event);
});
```

<Note>
`publishEvent` issues a real Redis `PUBLISH`. `subscribe` currently registers the callback locally; a production deployment that needs cross-process delivery should wire a dedicated subscriber connection. Treat the pub/sub helpers as a building block, not a turnkey event bus.
</Note>

## Outbox delivery

A first-party **exported** Redis outbox store is **not** currently available. The outbox stores published from the package are the in-memory and PostgreSQL implementations:

| Outbox store | Import path |
|--------------|-------------|
| `MemoryOutboxStore` | `@angriff36/manifest/outbox/memory` |
| `PostgresOutboxStore` | `@angriff36/manifest/outbox/postgres` |

The package `exports` map only exposes `./outbox`, `./outbox/memory`, and `./outbox/postgres`. A Redis Streams outbox exists in the source tree but is **not** a public export, so `@angriff36/manifest/outbox/redis` does not resolve. For durable at-least-once delivery today, use `PostgresOutboxStore`, or implement the `OutboxStore` interface (`@angriff36/manifest/outbox`) against Redis Streams yourself.

## Wiring in Redis

```typescript
import { RuntimeEngine } from '@angriff36/manifest';
import { RedisStore } from '@angriff36/manifest/stores';
import { PostgresOutboxStore } from '@angriff36/manifest/outbox/postgres';
import { Pool } from 'pg';

const runtime = new RuntimeEngine(ir, {
  userId: session.userId,
  tenantId: session.tenantId,
  storeProvider: (entityName) => {
    if (entityName === 'Session') {
      return new RedisStore(entityName, { url: process.env.REDIS_URL, defaultTTL: 3600 });
    }
    return undefined;
  },
  // Durable event delivery via the exported Postgres outbox:
  outboxStore: new PostgresOutboxStore(new Pool({ connectionString: process.env.DATABASE_URL })),
});
```

## Notes & limitations

<Note>
`ioredis` is loaded lazily and is optional. If it is not installed, the store cannot connect to Redis. For production use, consider TLS for secure connections (`rediss://` URL scheme) and a managed/clustered Redis for scaling. The pub/sub `subscribe` helper registers callbacks locally and does not yet open a dedicated subscriber connection.
</Note>

Manifest is an IR-first domain-specific language for encoding business rules, authorization policies, and event-driven workflows. The questions below address the most common points of confusion — from how Manifest compares to ORMs, to how the runtime evaluates guards and policies, to what's coming in vNext.

<AccordionGroup>

  <Accordion title="What is Manifest and who is it for?">
    Manifest is a domain-specific language for defining business rules and workflows with declarative specifications. It is **IR-first**: the Intermediate Representation (IR) is the single source of truth for program semantics, not the generated TypeScript or route code.

    The primary consumers are **AI agents** that emit, validate, and reason about Manifest programs. Secondary consumers are developers who want:
    - Declarative business rules that compile to full-stack implementations
    - Deterministic execution — identical IR plus identical context always produces identical results
    - Guard-based authorization that cannot be bypassed
    - Event-driven architectures with guaranteed ordering

  </Accordion>

  <Accordion title="What makes Manifest different from ORMs like Prisma?">
    Prisma and Manifest solve different problems. Prisma gives you a type-safe query API over a database schema. Manifest gives you an executable contract over your business rules.

    | Prisma | Manifest |
    |--------|----------|
    | Database schema + type-safe queries | Business rules + execution semantics |
    | Queries are imperative (`findMany`, `update`) | Commands are declarative (`create Recipe { ... }`) |
    | No built-in authorization | Guards and policies are first-class |
    | No event system | Events are first-class with ordering guarantees |
    | Generated code is source | IR is source; generated code is derivative |

    You can use Prisma as a storage adapter behind Manifest — it is not a replacement.

  </Accordion>

  <Accordion title="What are the two integration patterns?">
    Manifest supports two ways to integrate with your application:

    1. **Projections** — Generate platform-specific code (Next.js App Router routes, Express controllers, TypeScript types, client SDKs) directly from IR. Use this when your mutation flow is standard runtime execution and you want convention-over-configuration.

    2. **Embedded Runtime** — Write application handlers that call `RuntimeEngine.runCommand` directly. Use this when you need custom orchestration, custom event handling pipelines, or framework-specific behavior.

    | Use projections when | Use embedded runtime when |
    |---------------------|--------------------------|
    | You want generated route/controller code | You need custom orchestration around commands |
    | Your mutation flow is standard | You need custom event handling pipelines |
    | You want convention-over-configuration | You need framework-specific behavior |

  </Accordion>

  <Accordion title="Can I use Manifest with my existing database?">
    Yes. Manifest does not require a specific database. You can connect any storage backend by implementing the `Store` interface and passing it through `storeProvider` in `RuntimeOptions`:

    ```typescript
    interface Store<T extends EntityInstance = EntityInstance> {
      getAll(): Promise<T[]>;
      getById(id: string): Promise<T | undefined>;
      create(data: Partial<T>): Promise<T>;
      update(id: string, data: Partial<T>): Promise<T | undefined>;
      delete(id: string): Promise<boolean>;
      clear(): Promise<void>;
    }
    ```

    Pass your store implementation when constructing `RuntimeEngine`:

    ```typescript
    const runtime = new RuntimeEngine(
      ir,
      { actorId: 'user-123' },
      {
        storeProvider: (entityName) => {
          if (entityName === 'Recipe') return new MyRecipeStore();
          return undefined;
        },
      },
    );
    ```

    Returning `undefined` uses built-in behavior only for `memory` and browser
    `localStorage`. Durable targets fail until they are bound. Generated
    companions can load bindings from `manifest.config.ts`; see the
    [complete PostgreSQL setup](/language/stores#complete-generated-companion-postgresql-setup).

  </Accordion>

  <Accordion title="What storage targets are supported?">
    The runtime **must** support `memory`. The following targets **may** be supported depending on your environment:

    - `localStorage` (browser)
    - `postgres` (Node.js) — requires a `storeProvider` supplying the adapter; the runtime does not auto-instantiate it
    - `supabase` (Node.js / Edge) — requires a `storeProvider` supplying the adapter; the runtime does not auto-instantiate it
    - Custom stores via the `Store` interface and `storeProvider`

    Unsupported targets emit diagnostics — there is no silent fallback unless you explicitly configure one.

  </Accordion>

  <Accordion title="Does Manifest handle authentication?">
    No. Manifest handles **authorization** (policies and guards), not authentication.

    Authentication is the responsibility of your application layer — Clerk, Auth0, NextAuth, and similar providers all work. You derive identity from your auth context and inject authoritative values (for example `userId`, `tenantId`, `user.role`) into the runtime context before command execution.

    <Warning>
      Never read `id`, `userId`, `tenantId`, or `orgId` from `request.json()`. Always derive these values from your auth context and inject them into the runtime. Client-supplied identity values cannot be trusted.
    </Warning>

  </Accordion>

  <Accordion title="What is the execution order for commands?">
    Manifest commands execute in a fixed, deterministic order defined in the language specification (`docs/spec/semantics.md`):

    ~~1. Rate-limit → 2. Policy → 3. Constraint → 4. Guard → 5. Action → 6. Emit → 7. Return (as previously listed without the pre-gates)~~

    > **Correction (2026-07-15) @RYANSIGNED:** Full command path:
    > 1. **Build evaluation context** — apply parameter defaults; fail closed on missing required parameters (`MISSING_REQUIRED_PARAMETER`) before later gates
    > 2. **Rate-limit check** — per-user, per-tenant, or global limits declared on the command; stop with `rateLimitDenial` if exceeded
    > 3. **Policy check** — evaluated for `execute` or `all` scope (policy-level rate limits run before each policy expression)
    > 4. **Command-level constraint validation** — severity `block` / `warn` / `ok`; unauthorized `block` failures halt execution
    > 5. **Guard evaluation** — declaration order; halts on the first falsey result
    > 6. **Action execution** — actions run in order
    > 7. **Event emission** — events emitted in declaration order
    > 8. **Return** — `CommandResult` is returned to the caller
    >
    > This order is not configurable. Tenant / idempotency gates (when configured) run before evaluation. Entity-level constraints and transitions are enforced on create/update mutations, not as a substitute for step 4.

  </Accordion>

  <Accordion title="What happens when a guard fails?">
    Execution halts immediately. There is no auto-repair, no fallback, and no permissive default.

    The runtime returns a `CommandResult` with `success: false` and a `guardFailure` object that includes the failing guard's index, expression, and resolved values at the time of failure. Always inspect `result.guardFailure`, not a `result.diagnostics` field — `CommandResult` does not have a `diagnostics` field.

    ```typescript
    const result = await runtime.runCommand('create', input, { entityName: 'Todo' });

    if (!result.success && result.guardFailure) {
      console.error('Guard failed:', {
        index: result.guardFailure.index,
        expression: result.guardFailure.formatted,
        resolved: result.guardFailure.resolved
      });
    }
    ```

  </Accordion>

  <Accordion title="What is the difference between read, execute, and all policy scopes?">
    Policy scope controls when a policy is enforced during command processing:

    - **`read`** — Enforced at the runtime read gate: `getInstance` / `getAllInstances`. A denied read fails closed (`getInstance` returns `undefined`, `getAllInstances` omits the row). Note that generated projections may bypass the engine and query storage directly; those reads are not gated.
    - **`execute`** — Enforced during command execution. Use this for policies that must block mutations.
    - **`all`** — Enforced during command execution AND at the read gate.

    `read` policies are enforced only when reads go through the engine's `getInstance` / `getAllInstances`, not on the store and not inside `runCommand` (the internal execution read path is un-gated so command results are unchanged).

  </Accordion>

  <Accordion title="What are constraint severity levels?">
    Constraints have three severity levels that control whether a violation blocks execution:

    - **`block`** — The constraint must pass for the command to proceed. This is the default.
    - **`warn`** — Execution continues even if the constraint fails. The violation is recorded in `result.nonBlockingViolations`.
    - **`ok`** — Informational only. Never blocks execution.

    ```typescript
    // Check non-blocking violations after a successful command
    if (result.nonBlockingViolations) {
      console.warn('Non-blocking violations:', result.nonBlockingViolations);
    }
    ```

    If a constraint is not blocking execution when you expect it to, verify that its severity is set to `block`.

  </Accordion>

  <Accordion title="Are read operations (GET) validated by the runtime?">
    Reads that go through the engine are gated. `read` (and `all`) policies are enforced at a central read gate in `getInstance` / `getAllInstances`: a denied read fails closed (`getInstance` returns `undefined`, `getAllInstances` omits the denied rows).

    However, generated projections (for example, Next.js GET route handlers) may bypass the engine entirely and query storage directly; those reads are not gated by the runtime. Mutations — commands executed via `runtime.runCommand()` — go through the full guard, policy, constraint, and event pipeline, and their internal reads are deliberately un-gated so command results stay deterministic.

  </Accordion>

  <Accordion title="What are vNext features?">
    vNext features are implemented in the IR and runtime and are ready for adoption today. They are not behind a flag. The current vNext features are:

    - **Constraint severity and outcomes** — `block`, `warn`, `ok` severity levels with `nonBlockingViolations` in `CommandResult`
    - **Command-level constraints** — Constraints attached to individual commands, not just entities
    - **Entity concurrency controls** — Optimistic concurrency on entity mutations
    - **Policy-based authorization** — `execute` and `all` scoped policies enforced at command execution; `read` (and `all`) scoped policies enforced at the runtime read gate (`getInstance` / `getAllInstances`, fail-closed)
    - **IR caching** — Compiled IR can be cached to disk to avoid recompilation on every request

    See the vNext spec and migration guide for adoption instructions.

  </Accordion>

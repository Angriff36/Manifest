This guide walks you through adding Manifest to an existing Next.js App Router project. By the end, you will have a `.manifest` domain model, generated route handlers for reads and writes, and Clerk auth wired into the identity pipeline.

## Prerequisites

- Next.js 13+ with the App Router
- ~~Node.js 18+~~
- An existing Next.js project (or `npx create-next-app@latest`)

> **Correction (2026-07-15) @RYANSIGNED:** **Node.js 20 or later** (`package.json` `engines.node` is `>=20`). Package pin `@angriff36/manifest` **3.6.4**.

## Workflow

<Steps>
  <Step title="Install the package">
    Add `@angriff36/manifest` to your project:

    ```bash
    npm install @angriff36/manifest
    ```

  </Step>

  <Step title="Write your .manifest file">
    Create a file describing your domain model. This example models a `Recipe` entity with a `publishRecipe` command (`publish` itself is a reserved word in Manifest and cannot be used as a command name):

    ```manifest
    entity Recipe {
      property required id: string
      property required name: string
      property category: string?
      property status: string = "draft"
      property authorId: string

      command publishRecipe() {
        guard self.status == "draft"
        guard user.id == self.authorId
        mutate status = "published"
        emit RecipePublished
      }

      policy canPublish execute: user.role in ["author", "editor"]
    }

    store Recipe in memory

    event RecipePublished: "kitchen.recipe.published" {
      recipeId: string
      authorId: string
    }
    ```

    Save this as `recipe.manifest` in your project root.

  </Step>

  <Step title="Generate routes">
    Use the projection CLI to generate the route files. Run these commands from your project root:

    ```bash
    # Compile the .manifest file to IR
    manifest compile recipe.manifest --output ir/

    # Generate all surfaces from the compiled IR
    manifest generate ir/recipe.ir.json \
      --auth clerk \
      --output .

    # Or generate specific surfaces individually
    manifest generate ir/recipe.ir.json --surface types --output .
    manifest generate ir/recipe.ir.json --surface client --output .
    ```

    `--output` is a directory; each artifact carries its own path relative to it. With the default surface (`all`), you get:

    | File | What it is |
    |------|------------|
    | `app/api/recipe/list/route.ts` | GET list route for `Recipe` |
    | `app/api/recipe/[id]/route.ts` | GET detail route for `Recipe` |
    | `app/api/manifest/[entity]/commands/[command]/route.ts` | The **command dispatcher** — the single canonical write route for every command |
    | `src/types/manifest-generated.ts` | TypeScript types |
    | `src/lib/manifest-client.ts` | Client SDK |

    <Note>
      Per-command concrete routes (e.g. `app/api/recipe/publish-recipe/route.ts`) are **not** generated by default — all writes go through the dispatcher. Opt in with `concreteCommandRoutes.enabled: true` in `manifest.config.yaml` if you need them for legacy callers.
    </Note>

    <Note>
      For repeatable builds, declare your sources and projections in `manifest.config.yaml` and run `manifest compile --all` followed by `manifest generate --all` — see the monorepo section below.
    </Note>

  </Step>

  <Step title="Review the generated read route">
    The generated GET handler uses a direct database query — it does not call `RuntimeEngine`. This is intentional: read policies are not enforced at the query level by default, and direct queries are more efficient.

    ```typescript
    // app/api/recipe/list/route.ts — auto-generated, illustrative (trimmed)
    import type { NextRequest } from "next/server";
    import { database } from "@/lib/database";
    import { manifestErrorResponse, manifestSuccessResponse } from "@/lib/manifest-response";
    import { auth } from "@clerk/nextjs";

    export async function GET(request: NextRequest) {
      try {
        const { orgId, userId } = await auth();
        if (!(userId && orgId)) {
          return manifestErrorResponse({ error: "Unauthorized", diagnostics: [] }, 401);
        }

        // Direct Prisma query — bypasses runtime deliberately
        const recipes = await database.recipe.findMany({
          orderBy: {
            id: "desc",
          },
        });

        return manifestSuccessResponse({ recipes });
      } catch (error) {
        console.error("Error fetching recipes:", error);
        return manifestErrorResponse(
          { error: "Internal server error", diagnostics: [] },
          500,
        );
      }
    }
    ```

    Tenant and soft-delete `where` filters are **opt-in** (`includeTenantFilter` and `includeSoftDeleteFilter` both default to `false`) — see the multi-tenant section below for how to enable them.

  </Step>

  <Step title="Review the generated command dispatcher">
    All writes flow through **one** generated route — the dispatcher at `app/api/manifest/[entity]/commands/[command]/route.ts`. It resolves the entity and command from the URL at request time and **must** call `RuntimeEngine.runCommand`: guards, policies, constraints, and event emissions only run through the runtime.

    Call it with the entity and command names from your `.manifest` file (both segments are case-sensitive):

    ```bash
    curl -X POST http://localhost:3000/api/manifest/Recipe/commands/publishRecipe \
      -H "Content-Type: application/json" \
      -d '{ "id": "recipe-1" }'
    ```

    Identity comes from Clerk on the server — the dispatcher builds the runtime context from the authenticated `userId`, never from the request body. The body's `instanceId` (or `id`) selects which instance the command runs against.

    ```typescript
    // app/api/manifest/[entity]/commands/[command]/route.ts — auto-generated, illustrative (trimmed)
    import type { NextRequest } from "next/server";
    import { manifestErrorResponse, manifestSuccessResponse, normalizeCommandResult } from "@/lib/manifest-response";
    import { createManifestRuntime } from "@/lib/manifest-runtime";
    import { auth } from "@clerk/nextjs";

    interface DispatcherContext {
      params: Promise<{ entity: string; command: string }>;
    }

    export async function POST(request: NextRequest, ctx: DispatcherContext) {
      try {
        const { orgId, userId } = await auth();
        if (!(userId && orgId)) {
          return manifestErrorResponse({ error: "Unauthorized", diagnostics: [] }, 401);
        }

        const body = await request.json();
        const { entity, command } = await ctx.params;

        if (!entity || !command) {
          return manifestErrorResponse("Missing entity or command in route", 400);
        }

        const instanceId = typeof body?.instanceId === "string"
          ? body.instanceId
          : typeof body?.id === "string"
            ? body.id
            : undefined;

        const runtime = await createManifestRuntime({
          actorId: userId,
          requestId: request.headers.get("x-request-id") ?? undefined,
          source: "route",
          user: { id: userId, tenantId: "__no_tenant__" },
        });

        const result = await runtime.runCommand(command, body, {
          entityName: entity,
          instanceId,
        });

        const normalized = normalizeCommandResult(entity, command, result);

        if (!normalized.success) {
          const firstDiagnostic = normalized.diagnostics?.[0];
          const status = firstDiagnostic?.kind === "policy_denial" ? 403
            : firstDiagnostic?.kind === "guard_failure" ? 422
            : firstDiagnostic?.kind === "constraint_block" ? 422
            : 400;
          return manifestErrorResponse({ error: normalized.error, diagnostics: normalized.diagnostics }, status);
        }

        return manifestSuccessResponse({ data: normalized.data, events: normalized.events, diagnostics: normalized.diagnostics });
      } catch (error) {
        // ... auth errors map to 401; everything else to 500
      }
    }
    ```

    Runtime construction is delegated to `createManifestRuntime` from `@/lib/manifest-runtime` — a module the generator emits for you (the `companions` surface, on by default) that embeds the compiled IR and constructs the `RuntimeEngine` with the given context. Regenerate it after every compile, or set `emitCompanions: false` and maintain your own — for example to enrich the `user` object with fields your guards and policies read (e.g. `role`).

  </Step>

  <Step title="Wire up Clerk auth">
    If you chose `--auth clerk`, install the Clerk SDK and configure your middleware:

    ```bash
    npm install @clerk/nextjs
    ```

    Add `clerkMiddleware` to `middleware.ts`:

    ```typescript
    import { clerkMiddleware } from '@clerk/nextjs/server';

    export default clerkMiddleware();

    export const config = {
      matcher: ['/((?!_next|[^?]*\\.(?:html?|css|js(?!on)|jpe?g|webp|png|gif|svg|ttf|woff2?|ico|csv|docx?|xlsx?|zip|webmanifest)).*)', '/(api|trpc)(.*)'],
    };
    ```

    Set your environment variables in `.env.local`:

    ```bash
    NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY=pk_...
    CLERK_SECRET_KEY=sk_...
    ```

    The generated routes call `auth()` imported from `@clerk/nextjs` to resolve `userId` (and `orgId`). Use the `authImportPath` option to import it from somewhere else (e.g. `@clerk/nextjs/server`). No additional configuration is needed in the route files.

  </Step>
</Steps>

## Multi-tenant setup

Tenant filtering is **opt-in**: `includeTenantFilter` defaults to `false`, so generated routes contain no tenant logic until you enable it. Turn it on in `manifest.config.yaml` (use `tenantIdProperty` to filter on a different column, e.g. `orgId`):

```yaml
# manifest.config.yaml
projections:
  nextjs:
    output: .
    options:
      authProvider: clerk
      includeTenantFilter: true
      tenantIdProperty: tenantId
      tenantProvider:
        importPath: '@/lib/tenant'
        functionName: 'getTenantIdForOrg'
        lookupKey: 'orgId' # or "userId"
```

Then regenerate with `manifest generate --all`.

When enabled, every generated route resolves the tenant through your `tenantProvider` function before touching data:

```typescript
// Emitted into generated routes when includeTenantFilter is true
const tenantId = await getTenantIdForOrg(orgId);

if (!tenantId) {
  return manifestErrorResponse({ error: 'Tenant not found', diagnostics: [] }, 400);
}

const recipes = await database.recipe.findMany({
  where: { tenantId },
  orderBy: { id: 'desc' },
});
```

You implement the provider function (shape: `(id: string) => Promise<string | null>`) — for example, backed by a `userTenantMapping` table keyed by `userId` if you set `lookupKey: "userId"`.

## Workspace / monorepo setup

In a monorepo, it is common to compile the IR once and share it across packages. A typical layout:

```
apps/
  web/           # Next.js app — consumes generated routes and client SDK
packages/
  domain/        # recipe.manifest, build script, generated IR artifact
  api-types/     # generated ts.types output, shared across apps
```

In `packages/domain`, declare the sources and projection targets in `manifest.config.yaml` instead of writing a build script by hand:

```yaml
# packages/domain/manifest.config.yaml
src: '*.manifest'
output: 'ir/'

projections:
  nextjs:
    output: '../../apps/web'
    options:
      authProvider: clerk
      # Redirect shared artifacts into the types package
      # (path is relative to the projection output above).
      paths:
        typesFile: '../../packages/api-types/src/recipe.ts'
```

Then the whole pipeline is two config-driven commands:

```bash
cd packages/domain
manifest compile --all     # compiles every source into ir/merged.ir.json
manifest generate --all    # runs every projection declared in the config
```

Routes land in `apps/web/app/api/`, the dispatcher in `apps/web/app/api/manifest/[entity]/commands/[command]/route.ts`, and types in `packages/api-types/src/recipe.ts`.

In `apps/web`, import the compiled IR directly:

```typescript
// apps/web/lib/manifest-ir.ts
import ir from '@repo/domain/ir/merged.ir.json';
export { ir };
```

## Listening to events in route handlers

Register an `onEvent` listener before calling `runCommand` to tap into emitted events. This is useful for publishing to an outbox or triggering webhooks:

```typescript
const runtime = new RuntimeEngine(ir, {
  user: { id: userId, role: 'author' },
});

const unsubscribe = runtime.onEvent((event) => {
  console.log(event.name, event.channel, event.payload);
  // Enqueue to your outbox, publish to Pub/Sub, etc.
});

try {
  const result = await runtime.runCommand(
    'publishRecipe',
    {},
    { entityName: 'Recipe', instanceId: params.id },
  );

  if (!result.success) {
    return manifestErrorResponse(result.error ?? 'Command failed', 422);
  }

  return manifestSuccessResponse({ events: result.emittedEvents });
} finally {
  unsubscribe();
}
```

## Realtime subscriptions (SSE)

Entities flagged with the `realtime` modifier unlock three additional surfaces: an SSE subscription route (`nextjs.subscribe`), a typed auto-reconnecting React hook (`nextjs.subscriptionHook`), and a shared singleton runtime accessor (`nextjs.sharedRuntime`) that lets command routes and subscriptions observe the same event stream:

```manifest
entity Recipe {
  realtime
  property required id: string
  // ...
}
```

```typescript
const projection = new NextJsProjection();
const sseRoute = projection.generate(ir, { surface: 'nextjs.subscribe', entity: 'Recipe' });
const hook = projection.generate(ir, { surface: 'nextjs.subscriptionHook', entity: 'Recipe' });
const shared = projection.generate(ir, { surface: 'nextjs.sharedRuntime' });
```

When any entity is realtime, generated command routes (inline mode) automatically use the shared runtime via `getSharedRuntime()` instead of constructing one per request, so subscribers see events emitted by commands. This requires a long-lived, single-instance server. See [Realtime Subscriptions](/extensibility/realtime-subscriptions) for the full contract.

## Common issues

<AccordionGroup>
  <Accordion title='"Entity not found in IR"'>
    Entity names are case-sensitive. Verify that the name you pass to `runCommand` exactly matches the entity name declared in your `.manifest` file.
  </Accordion>

<Accordion title="Guard fails even with the correct user">
  Guards reference `user.*` from the runtime context, not from the request body. The generated
  `@/lib/manifest-runtime` forwards the context it receives; if your guards read fields the
  dispatcher's auth step doesn't supply (e.g. `role`), enrich the context — either in a custom
  runtime module (`emitCompanions: false`) or by adjusting your auth provider wiring. If the guard
  references `user.role` and you did not supply it, the failure is correct behavior.
</Accordion>

<Accordion title="Generated route imports are wrong">
  The generated code uses import paths like `@/lib/database`, `@/lib/manifest-response`, and
  `@/lib/manifest-runtime`. The generator emits these companion modules itself (the `companions`
  surface, included in `--surface all`), so a fresh generate compiles without hand-written files —
  this error usually means the companions were generated into a different directory than your `@/*`
  alias points at, or generation ran with `emitCompanions: false`. Custom `databaseImportPath`,
  `responseImportPath`, and `runtimeImportPath` options relocate both the imports and the emitted
  modules; paths pointing at npm packages are left to you.
</Accordion>

<Accordion title="Routes don't match my react-query hooks">
  Both projections derive each entity's default URL segment from its name, and the casing must
  agree. Pass the same `routeCasing` option to both generators: `'lowercase'` (default — `PrepTask`
  → `preptask`), `'kebab-case'` (`prep-task`), `'snake_case'` (`prep_task`), or `'preserve'`
  (`PrepTask`). Explicit `routeSegments` / `entityRoutes` overrides still take precedence. Use
  `dateSerialization: 'iso-string'` (default `'date'`) when `date`/`datetime` fields cross the wire
  as ISO-8601 strings rather than `Date` instances.
</Accordion>

  <Accordion title="Tenant filtering returns no results">
    Tenant filtering is off by default (`includeTenantFilter: false`). If you enabled it and get no results, confirm that your database rows have the tenant column populated and that your `tenantProvider` function returns the correct value. To rule the filter out during development, remove `includeTenantFilter: true` and regenerate.
  </Accordion>
</AccordionGroup>

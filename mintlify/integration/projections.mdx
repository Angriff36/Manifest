A **projection** reads your compiled Intermediate Representation (IR) and emits platform-specific artifacts. Projections are tooling — they do not redefine runtime semantics. Adding or changing a projection never changes what a Manifest program _means_; it only changes what code gets generated from it.

Projections share a single contract — `ProjectionTarget.generate(ir, request)` returns one or more `ProjectionArtifact`s plus structured diagnostics. The CLI's `--projection` flag selects which projection runs; dispatch is generic (the registry resolves the name), so adding a new projection means registering it, not editing a hardcoded dispatch branch.

## What projections produce

The Next.js projection ships four surfaces:

| Surface            | CLI identifier   | What it generates                                                               |
| ------------------ | ---------------- | ------------------------------------------------------------------------------- |
| Entity read routes | `nextjs.route`   | Next.js App Router GET handlers using direct database queries                   |
| Command routes     | `nextjs.command` | POST/PUT/DELETE handlers that call `RuntimeEngine.runCommand`                   |
| TypeScript types   | `ts.types`       | `interface` declarations derived from your IR entity definitions                |
| Client SDK         | `ts.client`      | Typed fetch wrappers for calling your routes from browser or server-client code |

The Prisma projection ships a single surface:

| Surface | CLI identifier  | What it generates                                                                                                                        |
| ------- | --------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| Schema  | `prisma.schema` | One `schema.prisma` file with models, `@relation` wiring, `@db.Decimal` precision, and projection-config-driven `@map`/`@@map`/`@@index` |

## Reads vs. writes

This is the most important rule for projections:

- **Read routes (`nextjs.route`) may bypass the runtime.** Generated GET handlers use direct database queries (Prisma by default). That is a **projection** choice for efficiency — those handlers do not call `getInstance` / `getAllInstances`.
- ~~read policies are not enforced by default in Manifest semantics~~
- **Write routes (`nextjs.command`) MUST use `RuntimeEngine.runCommand`.** Bypassing `runCommand` for mutations bypasses guard evaluation, policy checks, constraint validation, and event emission. The spec requires these to run in a fixed order on every mutation.

> **Correction (2026-07-15):** Manifest **semantics** do enforce `read` / `all` policies at the
> runtime read gate (`getInstance` / `getAllInstances` — fail closed). Generated Next.js read
> routes that query the DB directly skip that gate unless you wire reads through the runtime or
> re-implement equivalent checks. Do not confuse projection bypass with language semantics.

<Warning>
  Never replace `RuntimeEngine.runCommand` with direct database writes in a command route. Doing so
  silently strips all semantic enforcement from that operation.
</Warning>

`packages/cli` @ package **v3.6.41**: `compile --all`, `generate --all`, `watch --all`, and

```bash
# Whole-project (config-driven) — preferred
manifest compile --all
manifest generate --all
manifest watch --all

manifest build --all
```

## Using the projection API programmatically

~~Import `NextJsProjection` and call `generateRoute` / `generateTypes` / `generateClient`.~~

> **Correction (2026-07-15):** `NextJsProjection` exposes the generic
> `generate(ir, request: ProjectionRequest)` contract — there are **no** public
> `generateRoute` / `generateTypes` / `generateClient` methods. Prefer the CLI `--all` path;
> if you must call the API, use `generate` with a `surface` id.

Struck phantom API (do not use): `generateRoute` / `generateTypes` / `generateClient` —
those methods are not on `ProjectionTarget`.

```typescript
import { compileToIR } from '@angriff36/manifest/ir-compiler';
import { NextJsProjection } from '@angriff36/manifest/projections/nextjs';
import fs from 'fs/promises';

const source = await fs.readFile('recipe.manifest', 'utf-8');
const { ir, diagnostics } = await compileToIR(source);

if (diagnostics.some((d) => d.severity === 'error')) {
  for (const d of diagnostics) console.error(d.message);
  process.exit(1);
}

const projection = new NextJsProjection();
const routeResult = projection.generate(ir, {
  surface: 'nextjs.route',
  entity: 'Recipe',
  options: { authProvider: 'clerk', appDir: 'app/api' },
});
for (const artifact of routeResult.artifacts) {
  await fs.writeFile(artifact.pathHint, artifact.code);
}
```

## Per-module output file splitting

When your Manifest source uses the `module` keyword to organize entities into modules (e.g., `module Recipe { … }`), several projections support **per-module path hints** that organize generated files by module instead of flattening all output into a single directory.

**Supported projections:**

| Projection | Module path pattern | Notes |
|-----------|-------------------|-------|
| Zod | `schemas/<module>/schema.ts` | Entity/command file organization when `IREntity.module` is set |
| JSON Schema | `schemas/<module>/<entity>.schema.json` | Shared module-path sanitizer with Zod |
| Pydantic | `models/<module>/models.py` + `models/<module>/commands/` | Entity and command file splitting by module |
| Dart | `lib/models/<module>/` + `lib/commands/<module>/` | Type definitions organized by module |
| Storybook | `stories/<module>/` | Command stories inherit entity module |
| Mermaid | `diagrams/<module>/` | State/sequence diagrams per module; ER diagram stays flat at `diagrams/er-diagram.mmd` |
| Analytics | `analytics/handlers/<module>/` | Per-entity tracking handlers; monolith tracking-plan/events stay flat |
| Express | `routes/<module>/` + `types/<module>/` | Router and type definitions; monolith `manifest-router.ts` / `manifest-types.ts` stay flat |
| Hono | `routes/<module>/` + `types/<module>/` | Same pattern as Express; `src/routes.ts` / `manifest-types.ts` stay flat |

**How it works:**

When an entity has `module: "<ModuleName>"` in the IR, the projection generates files into module-prefixed directories. Entities without a module emit to flat paths (preserving existing behavior).

```manifest
module Inventory {
  entity Product { … }
  entity Category { … }
}

module Orders {
  entity Order { … }
  entity LineItem { … }
}
```

Generated output (Zod example):
```
schemas/
  inventory/
    schema.ts       # Product, Category
  orders/
    schema.ts       # Order, LineItem
```

<Note>
  Per-module file trees are **optional** — projections emit flat paths when no module is set. Convex and Next.js per-module splitting remains unimplemented (see COMPLIANCE_MATRIX §6).
</Note>

## The projection boundary

Projections handle transport concerns: authentication, HTTP conventions, response shaping, and read-path strategy. They do not own runtime responsibilities.

| Projection owns                       | Runtime owns                    |
| ------------------------------------- | ------------------------------- |
| Reading IR and emitting code          | Building the evaluation context |
| Configuring auth and database imports | Enforcing policies              |
| Choosing the read-path strategy       | Evaluating guards in order      |
| Adapting to framework conventions     | Executing actions               |
|                                       | Emitting declared events        |

Projections are tooling. They cannot introduce new language semantics — only the runtime and its specification can define how Manifest programs behave. If you find a mismatch between what a generated route does and what the Manifest spec requires, the generated code is wrong, not the spec.

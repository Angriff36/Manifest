The **Prisma projection** reads compiled Manifest IR and emits a `schema.prisma` artifact. It is compile-time only — it does not connect to a database, run migrations, or know anything about any specific application. All relational concepts (table names, column names, decimal precision, composite indexes, foreign-key field names) live in projection config; Manifest core stays backend-neutral.

The projection ships **inside the main `@angriff36/manifest` package** — no separate install needed. Running `manifest generate -p prisma` works out of the box.

<Note>
An earlier pre-release explored shipping the projection as `@manifest/projection-prisma`. That package was never published. Since v1.0.0 the projection is bundled in `@angriff36/manifest` with the same internal separation (`src/manifest/projections/prisma/`) as the Next.js and routes projections.
</Note>

For programmatic use, import from the main package:

```ts
import { PrismaProjection } from '@angriff36/manifest/projections/prisma';
```

## Quickstart

A minimal `.manifest` source:

```manifest widget.manifest
entity Widget {
  property required id: string
  property required name: string
  property required qty: int
  property required unitCost: money
}

store Widget in durable
```

A minimal `manifest.config.yaml`:

```yaml manifest.config.yaml
projections:
  prisma:
    output: generated/
    options:
      provider: postgresql
```

Compile and generate:

```bash
manifest compile widget.manifest
manifest generate ir/ -p prisma -s prisma.schema -o generated/
```

Emitted `generated/schema.prisma`:

```prisma
// Auto-generated by @angriff36/manifest
// DO NOT EDIT — regenerate with the projection.

datasource db {
  provider = "postgresql"
}

generator client {
  provider = "prisma-client-js"
}

model Widget {
  id String @id
  name String
  qty Int
  unitCost Decimal @db.Decimal(12, 2)
}
```

Emitted `generated/prisma.config.ts` (Prisma 7+ — URL lives here, not in `schema.prisma`):

```typescript
// Auto-generated by @angriff36/manifest
// Prisma 7+: connection URL lives here, not in schema.prisma.
import { defineConfig } from 'prisma/config';

export default defineConfig({
  datasources: { db: { url: process.env.DATABASE_URL } },
});
```

## What gets emitted, what gets skipped

| IR shape | Projection behavior |
| --- | --- |
| `entity X { ... }` + `store X in durable` (or `postgres` / `supabase`) | Emits a Prisma `model` |
| `entity X { ... }` + `store X in memory` (or `localStorage`) | **Skipped** with `PRISMA_SKIPPED_NON_DURABLE` info |
| `entity X { ... }` with no `store` declaration | **Skipped** with `PRISMA_SKIPPED_NO_STORE` info |
| Entity marked `external` in IR (`external: true`) | **Skipped** with `PRISMA_SKIPPED_EXTERNAL` info |
| `property` entries | Emitted as Prisma fields |
| `computed` entries | **Never emitted** — derived values must not become columns |

Entities are iterated in IR source order; the projection does not re-sort.

<Warning>
A property literally named `id` becomes the `@id` field. Entities without an `id` property get an info diagnostic (`PRISMA_NO_ID_PROPERTY`) because Prisma's schema validator will reject a model with no primary key.
</Warning>

## Type mapping

The IR carries `type.name` as an arbitrary string (Manifest's type vocabulary is open). The projection holds the mapping table:

| IR `type.name` | Prisma scalar |
| --- | --- |
| `string`, `text`, `uuid` | `String` |
| `boolean`, `bool` | `Boolean` |
| `int` | `Int` |
| `bigint` | `BigInt` |
| `float` | `Float` |
| `decimal`, `money` | `Decimal` (with default `@db.Decimal(12, 2)`) |
| `date`, `datetime` | `DateTime` |
| `json` | `Json` |
| `bytes` | `Bytes` |
| anything else | **diagnostic** (`PRISMA_UNKNOWN_TYPE`) |

The default mapping does **not** include `number`. That is deliberate.

## The `number` ambiguity rule

Manifest's `number` type does not distinguish integers from real numbers from money. Silently mapping it to `Float` is the class of silent rounding bug this project exists to prevent — especially for currency. So the projection refuses to guess:

```manifest
entity Order {
  property required id: string
  property required total: number  // ← ambiguous
}
```

The projection emits an error diagnostic:

```
PRISMA_AMBIGUOUS_NUMBER (Order):
  Property 'Order.total' is typed 'number', which is ambiguous
  (Manifest does not distinguish integers from real numbers from money).
  Pick a precise type in the .manifest source: 'int' or 'bigint' for
  counts and ids, 'float' for measurements where rounding is acceptable,
  'money' or 'decimal' for currency and other exact-decimal values.
  Or supply a 'typeMappings.Order.total' override.
```

The ambiguous column is **skipped** in the output. Fix the source:

```manifest
entity Order {
  property required id: string
  property required total: money  // ← exact-decimal, picks up @db.Decimal(12, 2)
}
```

<Note>
The escape hatch is the projection config's `typeMappings` — supply an explicit Prisma scalar there and the diagnostic disappears. Useful when migrating a legacy `.manifest` file that already used `number` for a field whose precision intent is recorded elsewhere.
</Note>

## Money and decimal precision

`money` and `decimal` both resolve to Prisma `Decimal`. Either gets a default `@db.Decimal(12, 2)` — the conservative money default (up to 9,999,999,999.99 with cent precision).

```manifest
entity Order {
  property required total: money       // → Decimal @db.Decimal(12, 2)
  property required fxRate: decimal    // → Decimal @db.Decimal(12, 2)
}
```

Override per-property via `precision`:

```yaml
projections:
  prisma:
    options:
      precision:
        Order:
          fxRate: { precision: 18, scale: 8 }   # → @db.Decimal(18, 8)
```

The rule is keyed on the **resolved Prisma scalar**, not the IR type name. A property routed to `Decimal` via `typeMappings` picks up the same default precision.

## Relationships

All four IR relationship kinds emit real Prisma `@relation`:

| IR | Emitted Prisma | Notes |
| --- | --- | --- |
| `hasMany name: T` | `name T[]` | Parent side of 1:N |
| `hasOne name: T` | `name T?` | Parent side of 1:1 |
| `belongsTo name: T` | `nameId String` + `name T @relation(fields: [nameId], references: [id])` | Child side; FK type follows target's `id` |
| `ref name: T` | same shape as `belongsTo` | "Loose" — no missing-backside warning |

### 1:1 vs 1:N

The projection detects the cardinality from the opposite side:

```manifest
entity User {
  property required id: string
  hasOne profile: Profile        // 1:1 — declares opposite as hasOne
}

entity Profile {
  property required id: string
  belongsTo user: User           // child side → FK gets @unique
}
```

Emits:

```prisma
model User {
  id String @id
  profile Profile?
}

model Profile {
  id String @id
  userId String @unique          // ← @unique because opposite is hasOne
  user User @relation(fields: [userId], references: [id])
}
```

For 1:N (parent side is `hasMany`), the FK is **not** `@unique`.

### Foreign key field name

Default is `${relationshipName}Id` (camelCase). Override at IR level via the `foreignKey` annotation, or at projection-config level via `foreignKeys`:

```yaml
projections:
  prisma:
    options:
      foreignKeys:
        Book:
          author: writerId       # → Book.writerId instead of Book.authorId
```

The FK type follows the target entity's `id` property type. A `string` PK gets a `String` FK; an `int` PK gets an `Int` FK. No type mismatches.

### Foreign key columns honor `columnMappings`

FK fields are virtual properties from the consumer's perspective; the existing `columnMappings` knob applies:

```yaml
projections:
  prisma:
    options:
      columnMappings:
        Book:
          authorId: author_id    # → authorId String @map("author_id")
```

### `belongsTo` alongside an explicit FK property

If your `.manifest` source declares the FK property explicitly **and** declares the relationship, the projection deduplicates — it does NOT emit a second column with the same name:

```manifest
entity Book {
  property required id: string
  property required authorId: string    // ← explicit FK column
  belongsTo author: Author              // ← relationship using that column
}
```

Emits:

```prisma
model Book {
  id String @id
  authorId String                        // ← from the property declaration
  author Author @relation(fields: [authorId], references: [id])
}
```

The relation step sees that `authorId` already exists as a property and reuses it. The FK column line is **not** re-emitted; the `@relation(fields: [authorId], ...)` clause still wires it.

If you supply both an explicit FK property **and** a `foreignKeys` config override that points the relationship at a different column name, the override wins for the relation field — but the explicit property still emits as its own column, so you'll end up with both columns in the model. Pick one approach per relationship.

### Self-relations

A one-sided self-relation triggers `PRISMA_RELATION_MISSING_BACKSIDE`:

```manifest
entity Tree {
  property required id: string
  belongsTo parent: Tree   // no `children` declared → warning
}
```

A two-sided self-relation (`parent` + `children` on the same entity) is the textbook case where Prisma requires `@relation("name")` to disambiguate which `belongsTo` matches which `hasMany`. The projection refuses to invent names and emits `PRISMA_RELATION_AMBIGUOUS` for both sides — hand-add the relation names after generation, or refactor to a single relation:

```prisma
model Tree {
  id String @id
  parentId String?
  parent Tree? @relation("TreeHierarchy", fields: [parentId], references: [id])
  children Tree[] @relation("TreeHierarchy")
}
```

### MongoDB provider

When `provider: mongodb`, Prisma's Mongo connector requires the `@id` column to map to `_id` with ObjectId semantics. The projection applies the canonical pattern automatically:

```yaml
projections:
  prisma:
    options:
      provider: mongodb
```

```manifest
entity Order {
  property required id: string
  property required total: money
}
store Order in durable
```

Emits:

```prisma
model Order {
  id String @id @default(auto()) @map("_id") @db.ObjectId
  total Decimal @db.Decimal(12, 2)
}
```

The Mongo-specific attributes are auto-applied only when you have NOT supplied your own `columnMappings.<Entity>.id` or a literal `defaultValue` on the `id` property. If you want a non-ObjectId pattern (e.g. plain string ULIDs), override via `typeMappings.<Entity>.id` to something other than `String` — the projection will skip the ObjectId attribute but still emit `@map("_id")`.

## Configuration options

All per-property options use the nested `Record<EntityName, Record<PropertyName, ...>>` shape. There are no flattened `"Entity.property"` dotted-string keys anywhere in the config.

```yaml
projections:
  prisma:
    # Path hint for the emitted artifact (default: schema.prisma).
    output: generated/schema.prisma

    options:
      # Optional Prisma datasource provider. When set, the projection
      # emits a `datasource` block at the top of the artifact.
      provider: postgresql       # one of: postgresql | mysql | sqlite | sqlserver | mongodb | cockroachdb

      # Per-entity table-name override → @@map("...").
      tableMappings:
        Order: orders
        Customer: customers

      # Per-entity, per-property column-name override → @map("...").
      columnMappings:
        Order:
          createdAt: created_at
          updatedAt: updated_at

      # Automatic identifier casing (opt-in). Emits @map/@@map for any name
      # not already in the target case, so you don't hand-write the mappings
      # above for every field. Explicit tableMappings/columnMappings still win.
      naming: snake_case          # shorthand: snake_case columns + pluralized snake_case tables
      # Object form:
      # naming:
      #   table: snake_case        # snake_case | camelCase | PascalCase | preserve
      #   column: snake_case       # snake_case | camelCase | preserve
      #   pluralizeTables: true    # Widget → widgets (default true)

      # Per-entity, per-property decimal precision/scale.
      # Applies to any property whose resolved scalar is Decimal.
      precision:
        Order:
          total: { precision: 14, scale: 2 }
          fxRate: { precision: 18, scale: 8 }

      # Per-entity composite/named index definitions → @@index([...]).
      indexes:
        Order:
          - ["customerId", "createdAt"]
          - { fields: ["status"], name: "order_status_idx" }

      # Per-entity, per-property type override. Bypasses the default
      # IR-type-name → Prisma-scalar mapping. Value is the literal
      # Prisma scalar (e.g. "Int", "BigInt", "Decimal", "DateTime").
      typeMappings:
        Order:
          legacyAmount: Decimal   # opt out of PRISMA_AMBIGUOUS_NUMBER for a `number` field

      # Per-entity, per-relationship FK field name override.
      # Overrides default `${relationshipName}Id` and IR `foreignKey` annotation.
      # Value is a plain string (field name) or a structured object with referential actions.
      foreignKeys:
        Order:
          customer: customerRef   # → Order.customerRef instead of Order.customerId
          # Structured form — includes referential actions:
          # customer:
          #   fields: [customerId]
          #   references: [id]
          #   onDelete: Cascade
          #   onUpdate: NoAction

      # Per-entity, per-property native @db.* attribute (without the @db. prefix).
      # Applied after the Prisma scalar but before @default.
      # Use for types the projection can't infer, or to override the default @db.Decimal(12,2).
      dbAttributes:
        Order:
          amount: Decimal(14, 2)     # → @db.Decimal(14, 2)
          externalId: VarChar(36)    # → @db.VarChar(36)

      # Per-entity, per-property array of verbatim Prisma field attributes.
      # Appended after type and dbAttributes. Use for @unique, @updatedAt, @default(...), etc.
      fieldAttributes:
        Order:
          code: ["@unique"]
          updatedAt: ["@updatedAt", "@default(now())"]

      # Environment variable name for the database URL in the emitted prisma.config.ts.
      # Default: DATABASE_URL. Only relevant when `provider` is set.
      urlEnvVar: POSTGRES_CONNECTION_STRING
```

## Automatic casing (the `naming` convention)

By default the projection emits IR names verbatim — to render camelCase IR identifiers as `snake_case` database names you'd hand-write a `columnMappings` entry per field. The `naming` option automates that:

```yaml
projections:
  prisma:
    options:
      naming: snake_case
```

`createdAt` emits `@map("created_at")`, `Widget` emits `@@map("widgets")`. The shorthand `snake_case` expands to `{ table: snake_case, column: snake_case, pluralizeTables: true }`; the object form tunes each axis independently.

**The convention only ever adds `@map`/`@@map`.** The Prisma *model name* and *field identifiers* stay as the IR name, so relation `fields`/`references`, `@@id`, `@@unique`, and `@@index` are unaffected — only the physical database name changes, and a `@map`/`@@map` is emitted only when the physical name actually differs (so `id` stays bare):

```prisma
model UserAccount {
  id          String   @id
  createdAt   DateTime @map("created_at")
  displayName String   @map("display_name")

  @@map("user_accounts")
}
```

Resolution order per name:

1. explicit `tableMappings` / `columnMappings` (always win — also the **escape hatch** for irregular plurals the built-in pluralizer gets wrong)
2. the `naming` convention
3. the IR name verbatim

A **global** `naming` can also be set once at the top level of `manifest.config.yaml` and is inherited by the Prisma projection; a per-projection `projections.prisma.options.naming` overrides the global default.

## `prisma.config.ts` companion

When `provider` is set, the projection emits **two artifacts** from a single `generate()` call:

| Index | Path hint | Content |
| --- | --- | --- |
| `artifacts[0]` | `schema.prisma` | The full Prisma schema |
| `artifacts[1]` | `prisma.config.ts` | Prisma 7+ config file with `url: process.env.<urlEnvVar>` |

Prisma 7 moved the connection URL out of `schema.prisma` into a `prisma.config.ts` file. The projection handles this automatically so `prisma migrate` works without edits:

```typescript
// prisma.config.ts — emitted automatically when provider is set
import { defineConfig } from 'prisma/config';

export default defineConfig({
  datasources: { db: { url: process.env.DATABASE_URL } },
});
```

Customize the env var name via `urlEnvVar`:

```yaml
projections:
  prisma:
    options:
      provider: postgresql
      urlEnvVar: POSTGRES_CONNECTION_STRING
```

Without `provider`, only `schema.prisma` is emitted (no datasource block, no `prisma.config.ts`). Use this when you manage datasource configuration externally.

## Diagnostics reference

The projection emits structured diagnostics rather than silently doing the wrong thing. Each carries an entity name (where applicable) and a resolution hint. The CLI's `manifest generate` command **exits non-zero** when any error-severity diagnostic is returned, so CI pipelines detect generation failures.

| Code | Severity | Trigger | Resolution |
| --- | --- | --- | --- |
| `PRISMA_AMBIGUOUS_NUMBER` | error | Property typed `number` with no `typeMappings` override | Pick `int`/`bigint`/`float`/`money`/`decimal` in source, or supply `typeMappings.<Entity>.<Property>` |
| `PRISMA_UNKNOWN_TYPE` | error | `type.name` not in default mapping and not in `typeMappings` | Add a `typeMappings` entry or change the source type |
| `PRISMA_NO_ID_PROPERTY` | info | Entity has no property named `id` | Add `property required id: string` |
| `PRISMA_SKIPPED_EXTERNAL` | info | Entity marked `external entity` | Expected — projection intentionally skips external entities |
| `PRISMA_SKIPPED_NO_STORE` | info | Entity has no `store` declaration | Add `store EntityName in durable` to opt in |
| `PRISMA_SKIPPED_NON_DURABLE` | info | Store target is `memory` or `localStorage` | Flip to `durable` (or `postgres`/`supabase`) to opt in |
| `PRISMA_RELATION_VIA_THROUGH_UNIMPLEMENTED` | info | `hasMany ... through X` (explicit M2M) | Declare the join entity (`X`) with its own belongsTo relations |
| `PRISMA_RELATION_AMBIGUOUS` | info | Multiple relationships between same entity pair (including two-sided self-relations like `parent` + `children` on the same type) | Refactor to a single relation, or hand-edit `@relation("name")` after generation |
| `PRISMA_RELATION_MISSING_BACKSIDE` | warning | One-sided `hasMany` / `hasOne` / `belongsTo` / `ref` — all four emit a full `@relation` that Prisma rejects without the opposite side | Declare the matching relation on the other entity |
| `PRISMA_RELATION_TARGET_NOT_EMITTED` | warning | Relationship targets an entity that is not emitted as a Prisma model (external entity, memory/localStorage store, no store) | Declare the target as a durable entity, or remove the relationship |

## What the projection deliberately does NOT do

These are not bugs — they're scope boundaries:

- **No database connection.** The projection is pure IR → text. Running migrations is your job (or Prisma's).
- **No `@relation("name")` auto-generation.** Multi-relation pairs require named `@relation` clauses; the projection refuses to invent names. Diagnose, don't guess.
- **No join-entity inference for `through` relationships.** Manifest's `through` annotation declares the join model, but Prisma needs the join entity to be a fully-modeled entity with its own belongsTo declarations. The user provides those.
- **No `id` modifier.** Primary-key designation is by literal property name (`id`). A dedicated `@id` modifier in core grammar is a possible future addition; the projection will pick it up when it lands.
- **No app coupling.** No defaults reference tenant columns, soft-delete columns, auth fields, or any other application-specific concept.

## CLI usage

The projection registers automatically; `manifest generate -p prisma` works out of the box:

```bash
# Single source file
manifest compile orders.manifest
manifest generate ir/ -p prisma -s prisma.schema -o generated/

# One-step build
manifest build orders.manifest -p prisma -s prisma.schema --code-output generated/
```

The CLI's `--projection` flag accepts `prisma` alongside `nextjs`. The surface name `prisma.schema` is the projection's single emission surface.

## Programmatic usage

For build scripts that need direct control:

```typescript
import { PrismaProjection } from '@angriff36/manifest/projections/prisma';
import { compileToIR } from '@angriff36/manifest/ir-compiler';
import fs from 'fs/promises';

const source = await fs.readFile('orders.manifest', 'utf8');
const { ir, diagnostics: compileDiags } = await compileToIR(source);

if (!ir) {
  console.error('Compilation failed:', compileDiags);
  process.exit(1);
}

const projection = new PrismaProjection();
const result = projection.generate(ir, {
  surface: 'prisma.schema',
  options: {
    provider: 'postgresql',
    tableMappings: { Order: 'orders' },
    precision: { Order: { total: { precision: 14, scale: 2 } } },
  },
});

for (const d of result.diagnostics) {
  console[d.severity === 'error' ? 'error' : 'log'](`[${d.code}] ${d.message}`);
}

const errors = result.diagnostics.filter((d) => d.severity === 'error');
if (errors.length > 0) process.exit(1);

for (const artifact of result.artifacts) {
  await fs.writeFile(artifact.pathHint!, artifact.code, 'utf8');
}
```

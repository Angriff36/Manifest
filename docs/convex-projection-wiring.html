01

Manifest philosophy

Manifest is an IR-first DSL for business rules. You author .manifest → compile to IR (the contract) → projections emit Convex (or Prisma/Next/…). Generated files are views. Editing them is drift.

Golden rule If a behavior can be expressed in the .manifest (guard, constraint, policy, computed, event, reaction, schedule, webhook, saga, …), express it there — not in TypeScript. Before writing app logic, check the capability matrix. Hand-rolled duplicates of Manifest constructs are bugs, not shortcuts.
Entity ≠ table An entity is a governed domain object: data plus commands, guards, policies, events, and reactions. A Convex table is only the persistence projection of its properties. If you only emit CRUD tables, you missed the point of Manifest.
02

What is authoritative

One source of truth. Everything else is derivative or glue.

Layer Role Agent may edit?
*.manifest Domain meaning: entities, commands, reactions, schedules, webhooks, sagas YES — primary edit target
manifest.config.ts Src glob, IR out, projection options (authContextImport, …) YES — wiring only
IR JSON Executable contract every projection consumes NO — compile output
convex/*.ts (generated) Schema, queries, mutations, crons, http, sagas NO — regenerate
convex/lib/authContext.ts Maps Convex identity → role + tenant for generated code YES — required glue
UI pages / components Presentation; calls generated api.* YES — no business rules
Never manually create Convex mutations for business logic · HTTP routes that re-implement commands · Zod/OpenAPI schemas that duplicate Manifest constraints · DB models that duplicate entities · event handlers that duplicate reactions · crons that duplicate schedule · webhook endpoints that duplicate webhook. If you think you need those, first check whether Manifest already has the concept. One real gap to cover yourself: the generated webhook httpAction does not verify HMAC signatures — put a verifying edge in front of it (do not reimplement the route).

Language model (semantic map)

What a .manifest author is declaring — not “tables,” but governed behavior.

entity Order ├── property … persisted fields (+ modifiers) ├── computed … derived; never a Convex column ├── relationship … hasMany / hasOne / belongsTo / ref ├── constraint … validation (block/warn) ├── policy … authz: read / write / execute ├── transition … legal status moves ├── command claim(…) │ ├── guard … halt on first falsey │ ├── mutate … working-copy field writes │ ├── emit Event named IR event + optional payload │ └── retry / rateLimit / approval when declared └── store in supabase|postgres|durable|memory event TaskClaimed: "…" { … } on TaskClaimed run Station.sync… reaction (real runtime wiring) schedule … cron "…" run Entity.cmd webhook … "/path" run Entity.cmd saga Checkout { steps + compensate } // Parsed but NOT lowered to IR (no runtime effect) — do not rely on these: compose … { connect … } → use reactions instead expose … as rest … → use projections (Convex/Next/Express) flow / effect → stubs; warnings only
Command execution order (normative) Build context → policies → guards (ordered, halt on first falsey) → actions → emits → reactions cascade → result. Diagnostics explain; they must not change semantics. Identical IR + identical context ⇒ identical results.

Config that wires Convex

manifest.config.tsyou write this 
import type { ManifestConfig } from '@angriff36/manifest'

const config: ManifestConfig = {
  src: 'src/**/*.manifest',
  output: 'ir/',
  projections: {
    convex: {
      output: 'convex/',
      options: {
        referenceMode: 'convexId',
        authContextImport: './lib/authContext',
        policyMode: 'enforce',
        includeTenantFilter: true,
        includeSoftDeleteFilter: true,
        emitEventsTable: true,
      }
    }
  }
}
export default config
Stores Only durable / postgres / supabase entities become Convex tables. memory / localStorage are skipped — change the store target if the entity must persist in Convex.
07

Frontend consumption rules

Manifest owns the backend contract. The UI consumes generated Convex functions. Prefer Convex React hooks (useQuery / useMutation) against api from convex/_generated/api. The separate TanStack Query projection targets HTTP/Next stacks — not the primary Convex path. Authorization caveat: generated queries apply tenant + soft-delete filters but do not evaluate read/all policies (CONVEX_UNSUPPORTED_READ_POLICY). Entities gated by such policies emit internalQuery — fail-closed, not callable from clients. To surface their data in UI, write a policy-enforcing public query that wraps the internal one.

UI need Map to Do not
Entity list / detail useQuery(api.queries.listX) / getX Raw db access or ad-hoc REST
Filtered list by FK/index useQuery(api.queries.listXByY, { y }) Client-side full-table filter as the source of truth
Create / update / domain action useMutation(api.mutations.Entity_command) Patch documents from the client; invent parallel APIs
Form fields Command parameters (+ required props on create) Ask users for tenantId / auth ids / auto timestamps
Buttons enabled/disabled Reflect guards/policies in UX copy; still enforce on server Treat UI checks as security
Permissions / hide actions Role from auth context; policies already gate mutations Re-implement policy expressions in React
Derived totals / labels Manifest computed (display via query fields you store, or client calc) Persist computeds as columns (projection forbids it)
Follow-up writes after a command reaction in .manifest useEffect chaining mutations as business logic
app/orders/page.tsxUI pattern 
import { useQuery, useMutation } from 'convex/react'
import { api } from '../convex/_generated/api'

export function OrdersPage() {
  const orders = useQuery(api.queries.listOrder)
  const process = useMutation(api.mutations.Order_process)

  return (
    <ul>
      {orders?.map((o) => (
        <li key={o._id}>
          {o.status}
          <button onClick={() => process({ docId: o._id })}>Process</button>
        </li>
      ))}
    </ul>
  )
}
Custom UI logic belongs Layout, navigation, empty states, optimistic UX, third-party widgets (payments, maps), analytics beacons. It does not belong in: authorization, invariants, cascades, cron, webhook verification, or status machines — those are Manifest.
08

Auth & tenant rules

Real Convex only exposes ctx.auth.getUserIdentity(). Generated code must not read a fake (ctx as any).auth bag in production.

Required glue Set authContextImport: './lib/authContext'. Export getAuthContext(ctx) returning at least role and the tenant property. Handle identity-less system calls (crons, webhooks, sagas) explicitly — they invoke the same governed mutations.
  • Tenant id is server-derived on create — never a client argument.
  • Cross-tenant instance writes throw the same “not found” as missing docs (no existence oracle).
  • Queries fail closed: no auth → no rows (when tenant filter is on and the entity declares the column).
  • policyMode: 'skip' is for local demos only — never ship it.
09

Agent workflow (setup → loop → CI)

Bootstrap
once
create manifest.config.ts

Add src/**/*.manifest (or project convention), IR output, and projections.convex with authContextImport. Create convex/lib/authContext.ts.

Author
domain
write *.manifest

Entities, commands, events, reactions, stores (durable path), schedules/webhooks/sagas as needed. Prefer reactions over compose. Prefer projections over expose.

Compile
step 1
manifest compile --all

Merged IR to config.output. Fix error diagnostics before generating.

Generate
step 2
manifest generate --all

Emits all six Convex surfaces. Resolve any CONVEX_UNRESOLVED_* by simplifying expressions or fixing IR — never by editing generated files.

Convex
platform
npx convex dev

Types _generated/api and pushes functions. Then build UI against api.

Inner loop
on change
manifest watch --all

Or: edit .manifest → compile → generate → convex dev hot-reload. Commit regenerated artifacts with the source change when the host repo requires it.

CI
drift gate
manifest generate --all --check

Fail the build if committed Convex artifacts don't match a fresh generate.

Ship
deploy
npx convex deploy

Plus the frontend deploy. Ordering: compile → generate → convex deploy → app build.

10

Limitations matrix

Assume nothing in Manifest syntax “just works” on Convex until it appears as full or documented partial above.

Bucket Items
Supported Entities/properties/relationships · commands with guards/constraints/policies (enforced in mutations; read/all-gated entities emit internalQuery — expressions not evaluated in queries) · events/emits · reactions · schedules → crons · webhooks → http (routing/transform only — see Partial for signatures) · tenant-scoped reads/writes · roles/checkRole · events table · transitions (status checks in generated mutations) · fan-out / aggregate-count reaction params (when indexed)
Partial Computeds (no columns; usable in some emit/reaction exprs) · sagas (shared input only) · referential actions (diagnostic only) · memory stores (skipped) · webhook signature (declared but NOT enforced — the generated httpAction performs no HMAC verification; require a verifying edge/proxy in front of the endpoint until generated enforcement exists)
Ignored (parse-only) compose/connect · expose · flow · effect
Not in Convex projection approval / retry / rateLimit surfaces · live RuntimeEngine adapters (outbox/EventBus as Node packages) · Prisma-style migrations · TanStack hooks as the Convex default client
Unresolved expressions Anything the Convex expression renderer cannot map emits CONVEX_UNRESOLVED_{POLICY,GUARD,CONSTRAINT,…} and a denying throw — never a silent pass. Fix the .manifest; don't weaken the generated throw.
11

Forbidden manual patterns

  • Editing convex/schema.ts, mutations.ts, queries.ts, crons.ts, http.ts, or sagas.ts by hand.
  • Writing a Convex mutation that re-implements a Manifest command.
  • Adding a cron/webhook/event handler that duplicates schedule / webhook / reaction.
  • Client-supplied tenantId / auth user ids on create forms.
  • Bypassing generated mutations to ctx.db.patch from a custom action for domain writes.
  • Using compose/expose as if they executed — they don't.
  • Shipping with policyMode: 'skip'.
  • Saying “Manifest probably doesn't support X” without checking the matrix / skill catalog first.

Why IR-first matters Both projections read the same IR. The Prisma model and the Next.js handler that queries it can never drift, because neither is the source — they're two renderings of one fact. Editing generated files is overwritten on the next generate.

03

The config that wires it all

One manifest.config.ts declares the source glob, the IR output, and a block per projection. --all walks projections and runs each.

manifest.config.tsyou write this 
import type { ManifestConfig } from '@angriff36/manifest'

const config: ManifestConfig = {
  src: 'src/**/*.manifest',        // what compile --all reads
  output: 'ir/',                    // where merged IR lands
  naming: 'snake_case',             // global casing, projections inherit

  projections: {
    prisma: {
      output: 'prisma/',
      options: {
        provider: 'postgresql',
        urlEnvVar: 'DATABASE_URL',
        multiSchema: { enabled: true, schemas: ['auth', 'domain'] },
        autoBackRelations: true     // emit inverse relation fields
      }
    },
    nextjs: {
      output: '.',                    // app/ + src/ live at project root
      options: {
        appDir: 'app/api',
        authProvider: 'clerk',
        databaseImportPath: '@/lib/database',
        includeTenantFilter: true,
        includeSoftDeleteFilter: true,
        readRoutes: { enabled: true, directDbReads: true },
        dispatcher: { enabled: true, executionMode: 'inline' }
      }
    }
  }
}

export default config
Field-aware, not assumed Set includeTenantFilter/includeSoftDeleteFilter globally, but the nextjs projection still inspects each entity first — it only emits where: { tenantId } or where: { deletedAt: null } for entities that actually declare those properties. No phantom filters, no Prisma "unknown argument" type errors.
04

When each command runs

Two cadences. Build-time codegen regenerates artifacts from the IR; schema tooling turns the generated schema into a client and a database.

Develop
on save
manifest watch --all

Watches the src glob. On any change: compile → regenerate every projection in config. Your schema.prisma and route.ts files stay live as you edit .manifest. This is the inner loop.

Compile
build step 1
manifest compile --all

Reads every .manifest from config.src, resolves use/mixin, and writes one merged IR to config.output. Reports all files' errors in one pass.

Generate
build step 2
manifest generate --all

Reads the IR, iterates config.projections, runs each generator. Emits schema.prisma + prisma.config.ts and the full app/api route tree. Use generate --all --check in CI to fail on drift.

Prisma
after generate
npx prisma generate  npx prisma migrate dev

External Prisma CLI. generate reads the freshly-emitted schema.prisma → typed client into .prisma/client. migrate dev diffs the schema against your database. Run whenever the schema changes.

Commit
pre-commit hook
manifest fmt --check  manifest validate

Format the .manifest sources and validate the IR against the schema before the commit lands. Keeps the contract clean; doesn't touch generated output.

Ship
CI / deploy
next build

Bundles the generated route handlers + your client code. The handlers import the Prisma client and the runtime glue — both must exist before this step.

Ordering rule compile → generate → prisma generate → next build. Prisma client must exist before the Next.js handlers (which import it) can typecheck. watch --all collapses the first two for development.

// emitted ONLY because the User entity declares tenantId + deletedAt + createdAt
import { db } from '@/lib/database'

export async function GET(req: Request) {
  const users = await db.user.findMany({
    where: { tenantId, deletedAt: null },   // gated on real columns
    orderBy: { createdAt: 'desc' }            // falls back to id if absent
  })
  return Response.json(users)
}
06

How they meet at runtime

A write request lands on the dispatcher. The dispatcher builds the runtime, the runtime runs the command (policies → guards → actions → persist), and persistence flows through the Prisma client built from the same IR's schema.

Request hits the dispatcher route
POST /api/manifest/[entity]/commands/[command] — the one canonical write surface, generated by the nextjs projection.
Auth wrapper runs first
From authProvider (clerk / nextauth / custom). Failure returns 401 — never a 500.
Runtime is constructed
createManifestRuntime({ db }) from your glue, handed the Prisma client imported via databaseImportPath.
Command executes in fixed order
policies → guards → actions → emits → return. Guards halt on first falsey value; mutations buffer into a working copy.
Persistence flushes through Prisma
The working copy writes via db.* — the client Prisma generated from the same IR's schema.prisma. Schema and query can't disagree.
Reads bypass the runtime
GET list/detail routes query Prisma directly (directDbReads), field-aware. Writes go through the runtime; reads are lean.
app/api/manifest/[entity]/commands/[command]/route.ts  ·  dispatchergenerated 
import { db } from '@/lib/database'              // Prisma client (your glue)
import { createManifestRuntime } from '@/lib/manifest-runtime'

export async function POST(req: Request, { params }) {
  const { entity, command } = params
  const body = await req.json()

  const runtime = createManifestRuntime({ db })   // IR is baked in at build
  const result = await runtime.runCommand(entity, command, body)

  return Response.json(result)               // guard failure → diagnostic, not crash
}
Change the .manifest → recompile → regenerate. Never the other way.
Manifest is IR-first. schema.prisma and route.ts are renderings of one contract — generated artifacts, not edit targets. Order: compile → generate → prisma generate → next build.
